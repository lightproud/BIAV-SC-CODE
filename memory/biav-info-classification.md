## 1. 目的与适用范围

本文档定义脑缸组(Brain-in-a-Vat group)所有内部信息的分类框架。涵盖范围:聊天记录、会议记录、提交记录、社区原始信息、工程配置和代码、设计决策、流程、排期等。

**本文档自身就是一条法则**,用法则的方式被维护:全员可写、被使用即活、不被引用即过时、迭代由实际工作触发而非预设节奏。

-----

## 2. 核心原则

**原则一:主轴优先,标记其上**

信息分类的主轴是「**类型**」(它是关于什么领域的),不是「**性质**」(它怎么用、信几分)。性质、变体、阶段等是每条信息的标记字段,不是分类入口。

**原则二:多轴正交**

类型、性质、变体、阶段、事件类型,各自管一件独立的事,不互相替代。一条信息同时有所有轴的取值。

**原则三:开放写入,异常捕捉**

正典与法则全员可写,记载自动捕获不直接写。不在写入时设权限门,质量靠事后异常监测、同行评审、自然淘汰共同保障。

**原则四:演化容忍**

v1.0 是跑通用的最小集,不是终态。子类型、字段、规则随实际工作触发而迭代,不在规划阶段过度细化。

## 3. 主轴:7 个类型

类型回答的问题:**这条信息是关于什么领域的**?

类型决定信息存放的「分库」,决定 owner 团队,决定查询时的主入口。**顶层 7 类封闭**(改顶层即改框架,需走法则修订流程);**子主题(sub_topic)开放**,任何成员可提议新增,但要纳入法则迭代。

### 3.1 IP

关于世界观、设定、视觉风格、叙事基调、品牌资产的信息。

**典型内容**:角色档案、世界观大纲、视觉风格规范、叙事基调、关键术语表、品牌核心资产。

**最强性质**:正典(IP 的本质就是定义性真理)。

**特点**:寿命最长,变更最审慎,跨变体一致性要求最高,合规可触发分叉但默认全局。

### 3.2 游戏产品

关于游戏本体的设计、规则、数值、平衡、版本的信息。

**典型内容**:战斗系统、卡牌定义、数值表、平衡补丁、关卡设计、UI/系统规范、付费设计。

**最强性质**:正典(产品的现行规则)+ 记载(版本变更流水)。

**特点**:按运营变体分叉(全球/日服/国服),按子层独立继承(战斗/运营/叙事/视觉/音频/UI),阶段流转频繁。

### 3.3 周边产品

关于游戏衍生商品、授权、合作开发的信息。

**典型内容**:周边商品规划、授权协议、合作方资料、生产/物流记录、销售数据。

**最强性质**:正典(产品规格)+ 记载(销售/合作流水)。

**特点**:消费 IP 的品牌资产规范,但产品本身独立维护;按市场分叉。

### 3.4 品牌建设

关于对外形象、品牌叙事、跨渠道口径统一的信息。

**典型内容**:品牌故事、对外发声口径、视觉对外应用规范、跨平台 tone of voice、PR/媒体策略。

**最强性质**:正典(口径与规范)+ 法则(对外发声方法)。

**特点**:消费 IP 但服务于对外沟通;和社区运营有重叠,边界为「品牌建设负责长期形象,社区运营负责日常互动」。

### 3.5 社区运营

关于玩家社区、CM、UGC、活动 live ops、客服、用户反馈处理的信息。

**典型内容**:CM 公告、活动方案、客服记录、玩家反馈聚合、舆情监测、UGC 管理、付费/活动节奏。

**最强性质**:记载(尤其外部观测,即原「信号」)+ 法则(运营 SOP)。

**特点**:**外部观测的主入口**(玩家发言、社区反馈、舆情、外部 bug 报告),按变体强分叉(各服社区独立),数据量最大、噪音最高。

### 3.6 组织

关于人力、组织架构、协作流程、决策机制、内部治理的信息。

**典型内容**:JD、组织架构、汇报关系、协作流程、跨组协调机制、内部决策记录、复盘报告。

**最强性质**:正典(架构与流程)+ 法则(协作方法)+ 记载(决策与复盘流水)。

**特点**:**跨域决策的归属地**;通常不分变体;权限敏感(尤其薪酬/绩效相关条目,虽不在工具层卡入口,社会层有清晰边界)。

### 3.7 AI

关于 AI 在团队的应用、能力建设、协作模式、Agent 治理的信息。

**典型内容**:Agent 定义、prompt 库、工具目录、AI 使用边界政策、BIAV 心法、AI 协作模式、模型选型记录。

**最强性质**:法则(AI 协作方法)+ 正典(Agent 定义与规范)。

**特点**:**作为顶层独立而非组织子项**,反映 AI 在脑缸组的份量。注意:**游戏内的 AI 功能(智能 NPC、匹配)归游戏产品,不归本类**;本类指「团队使用的 AI」,不是「卖给玩家的 AI」。

## 4. 性质标记:3 个

性质回答的问题:**这条信息怎么被使用、信几分、怎么演化**?

每条信息必须有且仅有一个性质标记。

### 4.1 正典(Canon)

**定义**:定义性真理。被其他信息引用的根。

**特征**:

- 任何时刻对一个具体消费者必须有一个当前正确版本
- 修订需庄重,通常带变更链
- 可分阶段(设计中/已完成/已发布/已退役)
- 可分变体(按运营变体分叉)
- 错误版本不保留内容主体,只留占位 + 退役理由的指针

**消费方式**:作为 ground truth 被引用,引用时锁定版本。

**失守表现**:多版本并存且无主、宣发口径与游戏内不一致、错误版本仍在被消费。

### 4.2 记载(Record)

**定义**:时间戳事件流。Append-only,**修改即背叛**。

**特征**:

- 单条价值低,聚合价值高
- 来源分内部(internal)与外部(external);外部即原「信号」
- 不可改写,撤销作为新事件 append
- 自动捕获为主,人不直接「写记载」

**消费方式**:聚合(趋势)、扫描(时间线)、定点检索(那条具体的)。**永不作为「事实」消费,只作为「曾说/曾做/曾观察到」。**

**事件类型枚举**(`event_type`):

- `establish` — 首次确立
- `revise` — 修订前值
- `rescind` — 撤销前事(否认其合法性)
- `retire` — 退役(曾合法,现停止可见消费)
- `observe` — 外部观测
- `act` — 内部行动(commit、发布、上线)
- `discuss` — 讨论(聊天、会议片段)

**失守表现**:源系统断流、噪音淹没信号、聊天里有人改了历史、关键讨论丢失。

### 4.3 法则(Method)

**定义**:可复用的做事方式或判断框架。从经验中提炼,有保质期。

**特征**:

- 用例 > 规则(具体案例比抽象规则有效)
- 被使用才有效;不被引用即过时
- 包括从具体决断中抽象出的规则(如「角色叙事问题先区分节奏 vs 规划」)
- 全员可写,门槛在于「能否被泛化复用」

**消费方式**:做事前查、教新人时教、复盘时更新。

**失守表现**:写下后无人引用、与现实脱节、新人按文档做发现完全不一样、矛盾法则共存无人裁决。

## 5. 字段总表

每条信息记录的完整字段:

|字段               |取值范围                 |必填 |说明       |
|-----------------|---------------------|---|---------|
|`type`           |7 类之一                |是  |主轴,见 §3  |
|`sub_topic`      |类型内子分类               |是  |开放枚举,渐进添加|
|`nature`         |正典 / 记载 / 法则         |是  |性质标记,见 §4|
|`service_variant`|全球 / 日服 / 国服 / 全局    |是  |运营变体     |
|`stage`          |设计中 / 已完成 / 已发布 / 已退役|仅正典|见 §6.1   |
|`event_type`     |见 §4.2 枚举            |仅记载|事件类型     |

## 6. 关键机制

### 6.1 正典阶段流转

```
设计中  →  已完成(已签字、已进 build,未发布) →  已发布  →  已退役
```

- **状态转移是一等事件**,每次跳转写一条记载(`event_type=act`)
- **错误版本**与「已退役」不同:错误版本走撤销(`rescind`),内容主体不入正典;已退役曾合法存在,正典中保留为占位状态
- **可见性按阶段控制**:对外消费(玩家可见、对外 agent)只取「已发布」;团队内部可看全阶段

### 6.2 运营变体继承(线性链 + 按子层继承)

```
全球服  →  日服  →  国服
```

继承策略按子层独立:

|子层   |全球→日|日→国 |
|-----|----|----|
|战斗   |可分叉 |强制继承|
|运营   |可分叉 |可分叉 |
|叙事   |待定  |待定  |
|视觉   |待定  |待定  |
|音频   |待定  |待定  |
|UI/系统|待定  |待定  |

存储原则:**指针 + 覆盖**,不物化重复。下游变体未覆盖时透明继承上游;主动覆盖时落一行。

「钉版」是显式声明,本身要落决策记载,不可悄悄改字段。

### 6.3 派生视图(替代「状态」)

「当前国服线上版本」「本周 P0 bug 列表」「目前在做什么」这类信息**不是独立性质**,而是从正典 + 记载派生的查询视图。

每个派生视图必须有:

- `name`(命名)
- 查询定义
- `owner`
- 刷新承诺(实时 / 周期)
- 消费者列表

派生视图本身的定义是法则。

## 7. 写入与权限纪律

### 7.1 写入模式

|性质|谁能写   |写入路径   |质量保障          |
|--|------|-------|--------------|
|正典|内部团队全员|直接编辑/提议|异常监测 + 同行评审   |
|记载|无「写者」 |源系统自动捕获|源系统自身纪律       |
|法则|内部团队全员|直接撰写/提炼|异常监测 + 不被使用即淘汰|

### 7.2 异常监测(代替事前权限门)

**原则**:异常 ≠ 拒绝。异常被捕捉后是「通知 + 复核」,修改照常生效,但被高亮等待 review。

候选异常监测规则(具体阈值待接入信息源后定):

**正典异常**:

- 单次大幅修改
- 跨域修改(非 owner 域)
- 阶段跳跃(跳过中间阶段)
- 短时间内频繁反复
- 高敏感修改(已发布、合规相关、跨变体根)
- **对上游变体的修改触发强通知**(国服改全球/日服时,上游 owner 必须被通知)

**法则异常**:

- 与现有法则矛盾
- 修改高频引用法则
- 删除法则(法则应标记过时,不应删除)

**记载异常**(因人不直接写,异常意味着捕获本身出问题):

- 源系统断流
- 捕获频率异常(激增或骤降)
- 内容反模式(commit message 全是 "fix"、聊天突然全表情包)

### 7.3 「记载」两种入场

虽然记载性质上不区分,但写入路径分两种,写入者承担的判断责任不同:

- **自动捕获**:无门槛,机器决定。聊天、commit、社区帖默认走这条
- **手动记入**:有门槛,人决定。会议关键发言、关键决断、外部重要信号需人工触发

## 8. 反模式与失守信号

### 8.1 类型混淆

- **正典 ↔ 记载**:正典被噪音稀释,记载的不可改性被破坏
- **正典 ↔ 法则**:产品事实(IP)与做事方式(AI)混在一处,owner 模糊
- **记载里塞决断**:决断被讨论流水淹没,半年后翻不到「为什么这样定」

### 8.2 性质混用

- **错误版本进正典**:本应被否认的内容仍在被消费
- **修订当撤销**:「这次平衡调整」被对外说成「之前的内容是错的」,引发玩家误解
- **流水当事实**:把「曾说过」当成「现在仍如此」

### 8.3 法则失守

- 写下后无人引用 → 该法则名存实亡,定期审查时应标记过时
- 与实际做法脱节 → 法则需修订或回收
- 矛盾法则共存 → 必须裁决,不能并存

### 8.4 跨变体漂移

- 全球服改了战斗,日服未跟,国服按规则强制跟日服 → 出现「国服战斗 = 日服战斗(旧) ≠ 全球战斗(新)」
- 这种漂移是**常态**,不是 bug,但需要明确视图让团队随时看到漂移状态

### 8.5 派生视图失维护

- 视图定义存在但 owner 不更新 → 团队各看各的,典型缓存一致性病变
- 视图被删除但仍被引用 → 引用链断裂

## 9. 演化记录

本框架的关键转折,留作未来追溯:

**v0.1(初探)**:按性质粗分 6 类(正典 / 流水 / 状态 / 决策 / 信号 / 方法)。问题:状态可从其他类型派生,信号是流水的子集,决策大部分可抽象为法则。

**v0.2(简化性质轴)**:状态→派生视图,信号→记载子集,决断→法则(可抽象部分)+ 记载(具体撤销动作)。性质收敛为 3 类:正典 / 记载 / 法则。

**v0.3(轴心校正)**:意识到「性质」与「类型」是两条独立轴,且**类型才是分类主轴**,性质是其上的标记。引入 7 类类型轴。

**v1.0(本版)**:类型为主轴(7 类封闭),性质为标记(3 类),变体/阶段/事件类型/来源为字段。开放写入 + 异常捕捉代替事前权限门。

关键设计决定:

- **AI 顶层独立**:反映其在脑缸组的份量超过「组织的一个子项」
- **社区改名运营**:范围扩展,涵盖 live ops/CM/活动/客服全链路
- **业务壳取消**:游戏产品/周边产品/品牌建设/社区运营平级展开,不再嵌套在「业务」下
- **错误版本不入正典**:撤销作为记载事件保留,正典保持干净

## 10. 待办与开放问题

以下事项 punt 到接入实际信息源时再定,避免规划阶段过度推演:

**短期(接入第一个信息源时)**:

- [ ] 异常监测的具体阈值(改多少算大、间隔多短算频繁)
- [ ] 通知机制(实时 vs 摘要、push vs pull)
- [ ] Review 流程(谁有义务 review、多久内 response)
- [ ] 派生视图(原「状态」)的命名规范和 owner 机制
- [ ] 子主题(sub_topic)的提议-纳入治理流程

**中期(模型成熟后)**:

- [ ] 其他子层(叙事/视觉/音频/UI)的继承策略补全
- [ ] 国服→日服→全球服 的反向流动机制(信号回流?PR?补偿?)
- [ ] 跨类型查询入口(「关于 William 的所有信息」)的实现路径
- [ ] 法则的定期 review 节奏(否则会自然腐化)
- [ ] 记载到法则的提炼节奏(否则法则枯死)

**待补的字段**:

- [ ] 子层(layer)字段是否提升为一等字段(战斗/运营/叙事/视觉/音频/UI/系统)
- [ ] `confidence` 字段是否需要(标记记载或正典的可信度)
- [ ] `extraction_version` 字段(若引入 LLM 抽取,用于批量重抽)

## 11. 落地建议

本文档定下后建议的下一步:

**第一步**:选一个具体信息源做端到端样板,推荐**会议记录** —— 语义密度高、量小、撞到的所有问题在其他源都会再遇到一次。

**第二步**:样板跑通后,逐源接入。优先级建议(易接入 × 高价值):

1. 会议记录
1. GitHub commit 与 issue
1. 内部聊天(飞书/Slack)
1. 社区原始信息(已有部分能力)
1. 工程配置与文档
1. 排期与 OKR(派生视图密集区)

**第三步**:每接入一个源,回到本文档:

- 验证 7 类是否覆盖
- 记录新出现的子主题
- 更新待办与开放问题
- 必要时小版本迭代(v1.1、v1.2)

**框架重构(v2.0)** 的触发条件:

- 同一类信息反复落不进任何类型
- 异常监测无法识别真问题
- 派生视图无法满足主流查询

未触发以上条件前,只做小版本迭代,不做框架重构。
