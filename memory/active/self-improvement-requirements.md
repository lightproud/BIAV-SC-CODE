## 1. 问题陈述

BPT 目前的运行质量改进完全依赖唯一维护者的人工排查:断流原因靠事后翻监控,token 消耗靠账单倒推,记忆系统效果无法量化。运行数据(transportHealth、工具调用日志、失败会话)已经存在,但散落在监控面板和日志文件中,既不可被 agent 消费,也没有基准来判断任何一次改动是"变好"还是"变坏"。

不解决的代价:维护者的排查带宽成为整个平台质量的上限;每次 SDK 升级或提示词调整都是盲改;已定的结构化工具调用日志决策(2026-07-11)产出的数据无人消费,价值闲置。

**核心命题:自我改进 = 信号 → 评估 → 改进 → 把关的闭环。缺任何一环,"自我改进"退化为"自我幻觉"。**

## 2. 目标

1. **信号可消费**:所有运行信号聚合为 agent 可读的每日运行报告,替代人工翻面板。
2. **改动可度量**:任何代码或提示词改动,合并前必须有评估分数的前后对比。目标:100% 的改进 PR 附带评估结果。
3. **改进可持续**:agent 定期自主产出小粒度改进 PR,维护者只做审阅,不做排查。目标:维护者单次审阅耗时 ≤ 15 分钟。
4. **底线不破**:全量确定性测试(~1,675 个)在任何自动改动后保持 100% 通过;CI 版本号强制不变。

## 3. 非目标

1. **不做自动合并**。任何 agent 产出的改动必须经人工审阅。理由:唯一维护者场景下,自动合并等于无人负责。
2. **不重建已退役子系统**。TF-IDF、知识图谱、自建语义检索等 2026 年 6 月已退役的组件不因"自我改进"名义复活。
3. **不引入新运行时依赖**。闭环全部组件遵守零运行时依赖原则(仅 `zod`、`fast-glob`)。评估用的 LLM 调用走既有 Messages API 通道。
4. **不做在线自我修改**。agent 不在生产运行时修改自身代码或提示词,所有改动经沙箱 → 测试 → PR 流程。
5. **不在本期覆盖 BPT 业务层代码**。v1 范围仅限 Silver Core SDK 自身及其提示词资产;BPT 应用层的自我改进为 P2。

## 4. 架构总览:四环闭环

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│ 环一:信号   │───▶│ 环二:评估   │───▶│ 环三:改进   │───▶│ 环四:把关   │
│ 采集与聚合  │    │ 基准        │    │ 执行        │    │ (人工)      │
└─────────────┘    └─────────────┘    └─────────────┘    └──────┬──────┘
       ▲                                                        │
       └────────────────── 合并后回到运行,产生新信号 ◀──────────┘
```

| 环 | 产出物 | 责任方 |
|---|---|---|
| 环一 | `runtime-report.md`(每日) | SDK 内建聚合器 |
| 环二 | 评估集 + 评估运行器 + 分数报告 | SDK 内建 + 人工出题 |
| 环三 | 改进 PR(diff + 评估对比) | agent(夜间任务) |
| 环四 | 合并/驳回决定 | 唯一维护者 |

**依赖顺序:环二先于环三。** 没有评估基准之前,不启动任何 agent 自主改代码的任务。

## 5. 需求明细

### 5.1 环一:信号采集与聚合

**REQ-1.1(P0)每日运行报告生成器**
SDK 提供 `generateRuntimeReport()`,聚合过去 24 小时的运行信号,输出单一 Markdown 文件 `reports/runtime-report-{date}.md`。

必含四类信号:
- 传输健康:`metrics.transportHealth` 的断流分类统计(按 Layer 1–4 恢复层级归类),含未恢复会话清单
- token 消耗:输入/输出 token 总量、缓存命中率、按场景(编码/非编码)拆分的 top 消耗会话
- 工具调用:结构化工具调用记录的汇总(工具名 × 调用次数 × 失败率),直接消费 2026-07-11 决策产出的日志格式
- 失败会话:异常终止、重试耗尽、用户中断的会话摘要(仅事实,不含对个人的评价性内容——与夜间合成管线的剥离规则一致)

验收标准:
- [ ] 报告可被 agent 以单次文件读取消费(单文件 ≤ 8K token)
- [ ] 信号缺失时对应小节显式标注"无数据",而非静默省略
- [ ] 生成器本身零新增运行时依赖

**REQ-1.2(P1)报告归档与趋势**
报告保留 30 天滚动窗口;提供 `compareReports(dateA, dateB)` 输出关键指标差值,供环三的 agent 判断改动方向。

### 5.2 环二:评估基准

**REQ-2.1(P0)双层评估结构**

| 层 | 内容 | 判定方式 | 作用 |
|---|---|---|---|
| 底线层 | 现有 ~1,675 个确定性测试 | pass/fail | 守住"不坏" |
| 行为层 | 20 题行为评估集(SCS-REQ-001 已建议) | LLM 评分(1–5) | 衡量"变好" |

行为评估集覆盖至少三个维度:记忆召回准确性、断流恢复表现、token 效率(同任务的消耗对比)。题目由维护者人工编写并入库 `evals/`,agent 不得修改评估集本身(防止"改考题"式作弊)。

验收标准:
- [ ] `runEvals()` 一条命令跑完双层,输出结构化分数报告(JSON + Markdown 摘要)
- [ ] 评估集文件在 CI 中标记为保护路径,agent 产出的 PR 若触碰 `evals/` 直接拒绝
- [ ] LLM 评分使用固定模型版本与固定评分提示词,保证跨次可比

**REQ-2.2(P1)评估回归门禁**
CI 集成:行为层任一维度分数相对基线下降 > 0.5 分,PR 标红警示(不硬性阻断,由维护者裁量)。

### 5.3 环三:改进执行

**REQ-3.1(P0)夜间改进任务**
复用"深度睡眠"合成管线的调度框架,新增独立任务类型 `self-improve`。流程:

1. 读取最近 7 天运行报告 + 失败会话日志
2. 定位**单一**最高优先级问题(优先级顺序:未恢复断流 > 测试覆盖盲区 > token 异常消耗 > 提示词冗余)
3. 在沙箱工作区改代码或提示词
4. 跑 `runEvals()` 全量双层评估
5. 产出 PR:diff + 问题定位说明 + 评估分数前后对比表

硬性约束:
- **一次一个问题,一 PR 一改动**。diff 超过 200 行(不含测试)自动放弃并记录原因
- 沙箱与生产隔离,任务失败不影响生产运行
- 底线层测试任一失败,不产出 PR,仅产出诊断报告

验收标准:
- [ ] 连续 5 个夜间任务,产出的 PR 全部满足"单问题 + 附评估对比"格式
- [ ] 沙箱任务崩溃不影响次日 BPT 正常服务

**REQ-3.2(P0)踩坑记录先行路径**
在环三代码级改进上线**之前**,先启用 memory tool 踩坑记录:工具声明 + 协议注入 + 存储三条腿配齐(SCS-REQ-001 的 Layer 2),让 agent 在日常运行中把踩坑写入 `memory/pitfalls/`。这是零代码改动的质量爬坡通道,同时其记录质量作为环三是否值得上线的先导验证。

验收标准:
- [ ] 运行两周后 `memory/pitfalls/` 有 ≥ 10 条可复核的有效记录
- [ ] 记录遵守既有剥离规则:仅技术事实,无对个人的评价性内容

**REQ-3.3(P1)提示词资产纳入改进范围**
系统提示词、场景模板视同代码:版本化、可 diff、评估覆盖。agent 可对非编码场景的提示词裁剪提出 PR(呼应 token 体积削减方向)。

### 5.4 环四:人工把关

**REQ-4.1(P0)审阅界面最小化**
维护者审阅一个改进 PR 只需看两样:diff 本身、评估分数前后对比表。PR 模板强制这两项置顶,其余(问题定位说明、日志引用)折叠。

验收标准:
- [ ] PR 描述首屏内可见完整评估对比表
- [ ] 单个 PR 审阅耗时实测 ≤ 15 分钟(以 5 个样本 PR 计)

**REQ-4.2(P0)禁止自动合并**
仓库分支保护:agent 身份的提交不可直接进主干;CI 版本号强制、全量测试通过为合并前置条件,与现行规则一致。

## 6. Black Pool 消费方指南

BPT 作为消费方需要做的事(SDK 不越俎代庖):

1. **接通日志格式**:确保 BPT 侧结构化工具调用日志(工具名、参数、返回摘要)按 SDK 约定的 schema 落盘,否则环一的工具调用小节为空。
2. **调度授权**:`self-improve` 夜间任务与"深度睡眠"合成任务错峰,建议合成任务完成后串行触发。
3. **报告消费**:BPT 可选择把每日运行报告摘要推送到维护者的晨间视图,但这属于 BPT 功能,不在 SDK 范围。
4. **无痕模式边界**:无痕会话不进入运行报告的失败会话小节(与记忆写入拦截同一边界),但传输层健康统计(不含内容)可计入。此边界需与 REQ(SCS-REQ-001)的无痕验证一并测试。

## 7. 分阶段落地

| 阶段 | 内容 | 前置 |
|---|---|---|
| Phase 0 | REQ-3.2 踩坑记录先行(零代码改动) | Layer 2 三条腿配齐 |
| Phase 1 | 环二:评估集编写 + `runEvals()` | 维护者出 20 题 |
| Phase 2 | 环一:运行报告生成器 | 结构化日志格式落地 |
| Phase 3 | 环三:夜间改进任务(先只允许提示词类 PR,再放开代码类) | Phase 1、2 完成 |
| Phase 4 | 环四门禁收紧 + 复盘,决定是否扩展到 BPT 业务层(P2) | Phase 3 跑满一个月 |

Phase 3 内部再分两步(先提示词后代码)的理由:提示词改动风险面小、回滚成本低,适合先验证整条管线的可信度。

## 9. 成功度量

- 先导指标(2–4 周):踩坑记录条数与复核有效率;评估集跑通率;运行报告按日生成率
- 滞后指标(1–3 月):未恢复断流数趋势;同场景 token 消耗趋势;维护者用于排查(非审阅)的时间占比下降
- 评估节点:Phase 3 满一个月时复盘,决定 Phase 4 走向
