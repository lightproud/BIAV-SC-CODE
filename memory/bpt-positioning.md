## 一句话定位

**BPT = 团队级在产的项目专用 agent 终端：元工具层 + 元知识层双主词，两层皆自持，
自持是主权要求而非工程偏好。**

## 定位五条（2026-07-04 拷问裁定）

### 1. 服务对象：团队级，且已兑现

冻结版原命题「把 Claude Code 范式工作流产品化，替代 Qoder，交给全团队用」中的
「交给全团队用」**已是现实**——团队已在用 BPT（守密人 2026-07-04 亲述）。
BPT 不是规划中的产品，是**有真实用户的在产工具**；一切引擎变更（如 SDK 换装）
按「给行驶中的车换发动机」的分量对待。

### 2. 竞争格局：并存但已最强；护城河与软肋同源

- Qoder 对部分团队成员不可用，替代与并存同时存在（A 与 C 并存格局）。
- 官方 Claude Code **从未**在办公环境大规模可用——BPT 只是因旧内核基于
  claude agent sdk（claude.exe 子进程）而被 2026-07-03 禁令**波及**，
  不存在「被禁前的官方标杆体验」。
- 黑池现为**最强、最懂项目**的一方：聊天记录、社区记录、工程记录、任务管理、
  文档、知识全部打通。**护城河 = 数据打通的广度**（交互范式可被复制，打通且
  有结构的项目知识不可）。
- **头号软肋 = 记忆管理与知识结构本身相当混乱**——护城河的内伤：强与乱是同一
  件事的两面（什么书都有、但没有书架的图书馆）。

### 3. 双主词：元工具层 + 元知识层（术语，守密人 2026-07-04 命名）

- **元工具层**：BPT 自持的 agent 工具基座——引擎（净室 SDK）、工具环、权限、
  多 provider 接入。_避免_ 称「终端层」「交互层」（降格为单主词的旧框架）。
- **元知识层**：BPT 打通的项目知识整体——聊天/社区/工程/任务/文档/知识及其
  结构治理（记忆管理、知识本体、检索质量）。_避免_ 与「数据层」混用
  （数据是原料，元知识层含结构与治理）。

两层为**并列一等公民**，不分主次（守密人否决「知识操作系统为主」的收缩）。

### 4. 自持是主权要求（元工具层的存在理由）

原设想「工具层由他人做、我们只使用」已被守密人废止。理由为结构性外部条件：
大集团大公司跨国技术商业竞争与地缘政治交错、数据主权宣示、集团内部技术路线
摇摆与重复造轮子、乃至**公司本身可能被交易导致基建整体更换**。这些条件下
「依赖外部工具」本身就是定位级风险——净室自有引擎（bpt-agent-sdk）是主权的
必然，2026-07-03 禁令只是提前引爆点，不是原因。

### 5. 模型可替换性 = 定位硬承诺

依赖栈最大的外部依赖是模型 API 本身。裁定：**多 provider（含国产模型）可替换
写入定位硬承诺**，不是备胎配置。诚实代价：可替换 ≠ 等效，换模型必然换手感。
**以 Claude 体验为范式参考，但不作限定**——体验主权在 BPT 自己手里。
（承接冻结版 Day-0 差异化第一条「模型自由」，由用户体验特性升格为生存冗余。）
