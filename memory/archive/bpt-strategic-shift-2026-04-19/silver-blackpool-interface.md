## 一、核心原则

```
银芯（公开信息平台） ──→ 黑池（私有内部加工）
                       ↑↑↑
                  唯一方向，单向纯输出
                  黑池→银芯：完全关闭
```

**银芯→黑池**：开放（黑池消费银芯产出的公开信息）
**黑池→银芯**：**完全关闭**（任何形式都不允许）

## 四、传输机制

### 4.1 数据传输

- **机制**：黑池侧自行 `git pull` 银芯公开仓库，**银芯不主动 push 到黑池**（黑池在内网，银芯无法直接访问）
- **触发**：黑池侧自行决定何时拉取（无固定频率）
- **格式**：原文件直接消费，不要求银芯做额外整理

### 4.2 知识/概念传输（指导协议）

- **机制**：守密人从银芯会话中学概念，**手动搬到黑池会话**（人工对话搬运）
- **特点**：守密人是人形传输介质，无自动化层
- **依据**：`memory/bpt-guidance-protocol.md`（4-19 战略转向决策）
- **本质**：仍是单向（银芯→黑池），黑池经验不通过守密人反向搬入银芯

## 五、安全边界（黑池→银芯关闭）

### 5.1 黑池侧 AI 会话的硬约束

黑池 AI 会话读取银芯公开数据时，必须遵守（已在 `BIAV-SC.md` 末尾「黑池数据同步接口」章节明文）：

1. **绝不向银芯写入任何内容**：不创建 Issue / 不提交 PR / 不修改文件 / 不创建 commit
2. **不将黑池内部信息泄露到银芯**：输出中不能包含未发布内容、商业数据、内部决策

### 5.2 银芯本仓库的拒绝接收

- 银芯不允许接收任何来自黑池的 PR / Issue / commit
- 守密人作为唯一传输介质（指导协议）也只搬概念不搬数据
- GitHub Settings 层面：仓库 Issue/PR 模板可加注「不接受黑池来源信息」声明（如未来需要硬化）

### 5.3 守密人作为人形守门人

- 守密人在银芯-黑池之间充当唯一的双向认知桥梁
- 守密人**不允许**把黑池数据/经验文字化写入银芯文件
- 守密人**允许**把银芯方法论/概念学进自己脑里再带到黑池
