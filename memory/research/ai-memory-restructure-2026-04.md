### 关键结论 5 条

1. **碎片化非「写多了」** —— 是「索引器不分等级 + 入档无规范 + 顶层目录扁平」三因合力；52 MB memory/ 中 86.6% 是 session-digest，主导索引噪音（Q1 § 一）

## 一、Q1 — 当前 290+ 份 `memory/` md 的碎片化诊断

### 1.1 实测规模

| 指标 | 数值 | 工具 |
|------|------|------|
| `memory/` 总 md 数 | 291 | `find memory -name '*.md' \| wc -l` |
| 其中 session-digest md | 252 | `ls memory/session-digests/*.md` |
| `memory/` 总体积 | 51 MB | `du -sh memory` |
| 全仓库索引文件数 | 3029 | `rag-performance-baseline.md` § 一 |
| 索引切片块数 | 115822 chunks | 同上 |
| memory 类文件占比 | 9.4% | 同上 |
| **session-digest 占索引切片比** | **~80%（估算）** | 252 份 × 中位 1967 行 / 标准 chunk 大小 |

### 1.2 文件长度分布（session-digest 252 份）

| 分位 | 行数 | 解读 |
|------|------|------|
| min | 108 | 短轮次（少量工具调用） |
| p25 | 1223 | 中等会话 |
| **p50** | **1967** | 典型会话约 2K 行 |
| p75 | 3578 | 长会话 |
| **p95** | **19592** | 超长会话约 2 万行 |
| max | 25514 | 单次会话最大 |

每条 session-digest 行均≈ 100 字符 → p50 文档约 200 KB → p95 文档约 2 MB。**单份 session-digest 内含的「token 多样性」远大于一份 200 行的 decision/strategy 文档**，TF-IDF 自然偏向 session-digest。

### 1.3 内容类型分布（按文件数）

| 类别 | 文件 | 占比 | 备注 |
|------|------|------|------|
| 决策档 | `decisions.md` 1 份 175 行 | 0.3% | 含 60+ 决策条目，密度高 |
| 战略档 | `strategic-plan-2026.md` 365 / `strategic-assessment.md` 263 / `phase-d-plan.md` 269 等 | ~3% | 长文本，多次改写 |
| 协议档 | `contribution-protocol.md` / `bpt-guidance-protocol.md` 等 | ~2% | 流程规范 |
| 教训档 | `lessons-learned.md` 259 行 | 0.3% | 31 条 lesson |
| 设计档 | `advanced-memory-design.md` 476 / `dreaming-agent-design.md` 等 | ~3% | 历史方案 |
| 派发 brief | `dispatch-brief-*` 多份 | ~2% | 任务规范 |
| 研究档 | `research-best-wiki-features.md` / 本文件等 | ~2% | 调研产物 |
| **会话摘要** | `session-digests/*.md` 252 份 | **86.6%** | **数量碾压** |
| 其他 | dreams/ / archive/ 等 | ~1% | — |

### 1.4 碎片化的 5 个具体表现

1. **数量失衡**：291 份中 252 份是 session-digest（86.6%），高价值档案 39 份（13.4%）被淹没。RAG 检索头条命中 session-digest 是数学必然，不是算法 bug。
2. **同主题分散**：例如「直推 main 政策」相关信息分散在 `decisions.md`（2026-03-29 / 04-26 两条）+ `CLAUDE.md` § 1 + `methodology.md` 「分支管理」段 + `lessons-learned.md` #28 #29 + 若干 session-digest——**没有单一权威入口**。
3. **冷热不分**：`bpt-guidance-protocol.md`（已删除子项目的协议）与 `strategic-plan-2026.md` v2.0（当前最有效战略）权重相同，索引器无法区分「冷档案」与「热档案」。
4. **历史不剥离**：`strategic-plan-2026.md` 包含 v1.0 / v1.1 / v1.2 / **v2.0** 四代版本叠加（共 365 行），新会话读完不知道哪段还有效——靠「v2.0 当前最新有效版本」标题人工提示。
5. **格式异质**：决策档是表格 + 长 markdown 段、教训档是编号列表、战略档是层级章节、session-digest 是工具调用流水——不同结构混在同一索引里，无法按结构化程度分级处理。

### 1.5 三层根因

| 层次 | 根因 | 责任面 |
|------|------|------|
| **索引层** | session-digest 与高价值档案同权重入索引 | `memory_search.py` 不分 doc class |
| **写入层** | 没有「**进哪一层 / 跟谁去重 / 何时归档**」的入档规范 | 缺写入协议 |
| **架构层** | `memory/` 目录按时间堆积，没有按「持久度 / 热度 / 用途」分层 | 缺顶层结构 |

**Q1 一句话答**：碎片化不是「写多了」造成的——是「**没有结构化入档规范 + 索引器不分等级 + 顶层目录扁平**」三因合力的结果。

## 三、Q3 — 不同知识类型的结构化策略分类

银芯当前的「知识」其实是 6 个完全不同的物种，强行用统一 RAG 处理是性能打折的根本原因。

### 3.1 六类知识矩阵

| # | 知识类型 | 当前位置 | 体量 | 演化频率 | 检索模式 | 推荐结构化策略 |
|---|---------|---------|------|---------|---------|---------------|
| 1 | **决策档** | `memory/decisions.md` 1 份 | 175 行 60+ 条 | 每周新增 1-2 条 | by topic / by date / by scope | **结构化 JSON 条目**（id / date / scope / supersedes / status），保留 markdown 渲染层 |
| 2 | **战略档** | `strategic-plan-2026.md` / `strategic-assessment.md` 等 | 长篇 200-500 行 | 每月大改 1 次 | 时序版本 + 主题深度 | **版本化分文件**（v2.0 当前 / v1.x 归档），加 `is_active: bool` 字段；附「主题入口 hub」 |
| 3 | **游戏静态数据** | `assets/data/card-system.json` / `wiki/data/extracted/lua_tables/` | 64 MB lua + 27 KB 加工 JSON | 版本更新时批量改 | by entity（角色 / 卡 / 界域）| **事实卡片**（已有 card-system.json 雏形），按实体粒度切，每个实体 1 个 JSON / md |
| 4 | **角色配置** | `wiki/data/db/characters.json` 24/72 | 部分 | Phase 2 W1 自举中 | by character | **结构化数据库**（schema v1.0 已锁），与游戏静态数据共用粒度 |
| 5 | **会话日志** | `memory/session-digests/` 252 份 | 51 MB | 每会话 1-N 份 | rarely by content（多按 session_id 追溯） | **冷热分层**（热 30 天保留 / 温归档 / 冷压缩或剥离索引）+ session-level meta |
| 6 | **社区数据** | `projects/news/output/*.json` + Discord 归档 | 1647 文件 | 每日新增 | by date / by platform / by entity | **timeline 索引**（已部分实现），按日期分片 + 平台维度 |

### 3.2 当前的「错位」

| 知识类型 | 应有结构 | 当前状态 | 错位 |
|---------|---------|---------|------|
| 1 决策档 | 结构化条目 | 单 md 长表 | **可读但难自动消费**——AI 要查「v2.0 之后哪些 4-19 决策仍生效」需通读 |
| 2 战略档 | 版本分文件 | 单 md 多代叠加 | **历史不剥离**——v1.0 + v1.1 + v1.2 + v2.0 同处一文件 |
| 3 游戏静态数据 | 事实卡片 | lua 原文 + 部分 JSON | **未 Cognify**——Q5 AwakerConfig 召回失败的根因 |
| 4 角色配置 | 结构化数据库 | 24/72 完整 + schema 已锁 | **进度问题**——Phase 2 W1 推进中，方向已对 |
| 5 会话日志 | 冷热分层 | 全部入主索引 | **session-digest 噪音**——Q3 指令牌 / Q5 AwakerConfig 排序偏差的根因 |
| 6 社区数据 | timeline 索引 | 已对（按日期 + 平台分片） | **方向已对** |

### 3.3 核心洞察

**银芯的真实痛点**：
- 类型 1（决策）+ 类型 2（战略）+ 类型 3（游戏数据）**结构化程度最不足**——这是「碎片化」的主要源头
- 类型 5（会话日志）**结构化是过的，但被错误纳入主索引**——这是「噪音」的主要源头
- 类型 4（角色）+ 类型 6（社区数据）方向对，照原节奏推进即可

**Q3 一句话答**：6 类知识需要 4 套不同结构化策略——决策 / 战略 → 结构化条目 + 版本化；游戏数据 → 事实卡片化；会话日志 → 冷热分层 + 索引剥离；角色 + 社区数据 → 维持。**不是一招通用方案能覆盖的**。

## 四、Q4 — 「AI 阅读友好」的可量化衡量标准

「AI 阅读友好」如果只是直觉判断，下次会不知道有没有变好。提议 **8 个可量化指标 + 1 个主观锚点**，分 4 组。

### 4.1 检索质量组

|---|------|---------|------|---------|

### 4.2 性能组

| # | 指标 | 当前实测 | 目标 | 测量方法 |
|---|------|---------|------|---------|

### 4.3 上下文效率组

| # | 指标 | 当前实测 | 目标 | 测量方法 |
|---|------|---------|------|---------|

### 4.4 时效性组

| # | 指标 | 当前实测 | 目标 | 测量方法 |
|---|------|---------|------|---------|
| M8 | **失效信息率** | 未测（已知 strategic-plan-2026 有 v1.0 ~ v2.0 共存） | < 5% | 抽 20 个查询，标注头 5 条中「已被覆盖但未标记」的比例 |

### 4.5 主观锚点（不可量化但必须问）

| # | 锚点 | 标准 |
|---|------|------|
| S1 | **守密人主观感受**：「同样的问题，新会话答得跟老会话一样准吗？」 | 守密人觉得「准」即过 |

### 4.6 推荐基准建立流程

1. **标准查询集 v1**：扩展 `rag-performance-baseline.md` § 三 现有 5 查询到 20 查询，覆盖六类知识（决策 3 / 战略 3 / 游戏数据 5 / 角色 3 / 会话历史 3 / 社区数据 3）
2. **基线测量**：对当前系统跑一遍，记录 M1-M8 数值，作为「优化前」基线
3. **每次架构改动后重测**：5 方向（A/B/C/D/E）任一落地后跑同一查询集
4. **目标值由守密人 + 主控台审定**：本表给出的是艾瑞卡的初步目标值，不是绝对标准

**Q4 一句话答**：8 个量化指标分 4 组（检索质量 / 性能 / 上下文效率 / 时效性）+ 1 个主观锚点。基线建立约 1 个会话工作量，之后每次改动可对比。

## 五、Q5 — 5 方向（A/B/C/D/E）技术评估

Code-memory 在 baseline 报告 + 命题响应中提出 5 候选方向。本节按统一维度（工程难度 / 维护成本 / 可逆性 / 与 9 模块耦合度 / 业界对标）逐一评估。

### 5.1 方向 A — 事实卡片（Fact Cards）

**定义**：把游戏内实体（角色 / 卡牌系统 / 界域 / 命轮 / 关卡）按统一 schema 写成「事实卡片 JSON」，每个卡片自包含 `evidence` 字段引用原始 lua / 解包数据。

**雏形**：`assets/data/card-system.json` v1.0（Code-memory 2026-04-26 已写）。

**业界对标**：Mem0 的 fact extraction（轻量版）+ Cognee Cognify 步骤的「Cognify 输出」格式。

**评估**：

| 维度 | 评分 | 说明 |
|------|------|------|
| 工程难度 | 中 | schema 设计 + 单卡片人工/半自动写入 |
| 维护成本 | 低 | 数据源（lua）变更时按字段同步更新 |
| 可逆性 | **高** | 卡片是增量数据，删除不影响其他模块 |
| 与 9 模块耦合度 | 低 | 写入 `assets/data/`，索引器自动消费即可 |
| 验证速度 | 快 | 每写 1 张卡片可立即用 RAG 验证召回 |

**优点**：可立即验证；可逆；试点失败成本低。
**缺点**：覆盖游戏静态数据，对决策档 / 战略档 / 会话日志噪音问题**无直接帮助**。

---

### 5.2 方向 B — 主题入口（Subject Hubs）

**定义**：给关键主题（如「直推 main 政策」「v2.0 三新使命」「黑池-银芯接口」）写**单一入口页**，索引该主题在仓库内的所有相关位置 + 当前权威结论。

**雏形**：`memory/strategic-plan-2026.md` 的 v2.0 章节、`memory/contribution-protocol.md` 已有此性质。

**业界对标**：Letta 的 core memory 块 + Native CLAUDE.md 的「policy 层」。

**评估**：

| 维度 | 评分 | 说明 |
|------|------|------|
| 工程难度 | 低 | 写 markdown 入口页 |
| 维护成本 | 中 | 主题相关条目变化时需同步更新入口页 |
| 可逆性 | **高** | 入口页是辅助读物，删除不影响原始档案 |
| 与 9 模块耦合度 | 低-中 | `context_manager.py` 推荐时可加权入口页 |
| 验证速度 | 快 | 新会话查关键主题命中入口页 = 验证 |

**优点**：解决「同主题分散」问题；几乎零风险。
**缺点**：需要人工维护同步（或建立「写决策时同步入口页」的规范）；不解决会话日志噪音问题。

---

### 5.3 方向 C — Q&A 预计算缓存

**定义**：深睡阶段用 LLM 对高频问题预生成结构化回答，缓存在 `assets/data/precomputed-cache.json`，下次会话命中即用。

**雏形**：`advanced-memory-design.md` 模块 5 设计，`dream.py` 部分实现。

**业界对标**：Berkeley Sleep-Time Compute / Mem0 cloud 的 query-pattern caching。

**评估**：

| 维度 | 评分 | 说明 |
|------|------|------|
| 工程难度 | 中 | LLM 调用 + 缓存命中检测 + TTL 管理 |
| 维护成本 | 中 | 缓存过期需重生 / 数据源变更需失效 |
| 可逆性 | 高 | 缓存可全量删除回退 |
| 与 9 模块耦合度 | 中 | 与 `dream.py` 深睡层集成 |
| 验证速度 | 中 | 需观察实际命中率 |

**优点**：可显著降低高频查询的延迟和 token 消耗。
**缺点**：**需要 API 预算**（每次深睡 ~$0.03-0.10）；TTL 设错会返回失效信息；不解决底层结构化问题，**只是表层缓存**。

---

### 5.4 方向 D — 分层记忆（Tiered Memory）

**定义**：仿 Letta 三层（core / archival / recall）或 Cognee 双层（session / permanent）模型，把 `memory/` 按「持久度 / 热度 / 用途」分目录：
- `memory/core/` — 始终在 boot-snapshot 的核心规则（< 5 KB）
- `memory/active/` — 当前阶段活跃档案（决策 / 战略 / 协议）
- `memory/archive/` — 历史 + 冷档案（已部分存在）
- `memory/session-digests/` — 会话日志（保留独立）

**业界对标**：Letta core/archival/recall + Native 的 policy/playbook/role/enforcement 四原语。

**评估**：

| 维度 | 评分 | 说明 |
|------|------|------|
| 工程难度 | **大** | 需要重构 `memory/` 目录 + 改 `memory_search.py` 按层加权 + 改 boot-snapshot 引用路径 |
| 维护成本 | 高 | 写入时要决定「这条信息属于哪一层」，需写入规范 |
| 可逆性 | **中** | 重构后回退成本高（git 路径变化导致 history 追溯困难）|
| 与 9 模块耦合度 | **高** | 影响所有 9 模块的引用路径 + index 配置 |
| 验证速度 | 慢 | 全量迁移后才能整体测 |

**优点**：根本性解决碎片化；与业界主流模型对齐。
**缺点**：**rip-and-replace 风险高**；仓库级目录变动会让所有引用过 `memory/*.md` 的脚本 / brief / decisions 失效；需要长准备期。

---

### 5.5 方向 E — 叙事记忆（Narrative Memory）

**定义**：用 LLM 把零散事实编织成连贯叙事段落（「2026 年 3 月底到 4 月底，银芯经历了 v1.0 → v2.0 三次战略转向，核心动因是…」），存入 `memory/narratives/<topic>.md`。

**业界对标**：Cognee 的 graph-narrative 输出 / Anthropic Skills「按需加载长参考材料」。

**评估**：

| 维度 | 评分 | 说明 |
|------|------|------|
| 工程难度 | **大** | 需要 LLM 自动叙事化 + 一致性校验 |
| 维护成本 | **高** | 事实变更后叙事需重写；难自动化 |
| 可逆性 | 中 | 叙事档案可删但 LLM 调用成本不可逆 |
| 与 9 模块耦合度 | 中 | 主要在 dream 深睡层 |
| 验证速度 | 慢 | 需主观评定叙事是否准确连贯 |

**优点**：对长尺度战略档案 / 历史档案最友好；新会话读 1 篇叙事 ≈ 读 10 份零散 md。
**缺点**：**ROI 不确定**——叙事如果不准比没有更糟；维护成本高；与「事实精度」诉求有张力。

| 维度 | A 事实卡片 | B 主题入口 | C Q&A 缓存 | D 分层记忆 | E 叙事记忆 |
|------|-----------|-----------|-----------|-----------|-----------|
| 工程难度 | 中 | **低** | 中 | 大 | 大 |
| 维护成本 | 低 | 中 | 中 | 高 | **高** |
| 可逆性 | **高** | **高** | 高 | 中 | 中 |
| 9 模块耦合 | 低 | 低-中 | 中 | **高** | 中 |
| API 预算 | $0 | $0 | **$10+/月** | $0 | **$30+/月** |
| 解决类型 1-2 决策战略 | ✗ | ✓ | 部分 | ✓ | ✓ |
| 解决类型 3 游戏数据 | **✓** | 部分 | 部分 | ✓ | ✗ |
| 解决类型 5 会话噪音 | ✗ | ✗ | ✗ | **✓** | ✗ |
| 试点失败可回滚 | **✓** | **✓** | ✓ | ✗（部分）| ✓ |

**Q5 一句话答**：A / B 是「低成本高可逆」选项，C 是「需 API 预算的表层加速」，D 是「根治但风险大」，E 是「ROI 不确定」。**没有任何单一方向能覆盖全部 6 类知识的痛点**——必须组合。

## 六、Q6 — 渐进式 vs 一次性重构权衡 + 决策树

### 6.1 两种路径对比

| 维度 | 一次性重构 | 渐进式 |
|------|-----------|--------|
| 时间窗口 | 集中 4-8 周 | 分批 2-4 周/批 × N 批 |
| 单次工作量 | 极大（停摆其他工作）| 小（不阻塞日常）|
| 风险 | 高（路径变更 + 索引重建 + 引用更新同时发生）| 低（每批独立验证）|
| 可逆性 | 低（涉及 git 历史路径变更）| 高（每批可单独回退）|
| 守密人参与度 | 高（需连续审定）| 低（每批 30-60 分钟决策）|
| 与 Phase 2 三新使命冲突 | **冲突**（M1 / M2 期间停摆其他工作）| 不冲突（穿插推进）|

### 6.2 艾瑞卡的决策树

```
是否需要立即解决某个量化指标的痛点？
├─ 是 → 确定主要痛点是哪一个？
│       ├─ M1 Top-1 准确率低 → 走方向 A（事实卡片）+ B（主题入口）
│       ├─ M3 召回不均衡 → 走方向 D 子集（先做 .searchignore 排除 session-digest 主索引）
│       ├─ M4 冷启动慢 → 走 MCP 常驻模式（非本调研范围，主控台已知方案）
│       └─ M6 boot token 过大 → 走 Native Skill 化（拆 BIAV-SC.md / CLAUDE.md）
│
└─ 否（追求长期最优结构）→ 一次性重构方向 D
       └─ 但需先：
              1. 建立 M1-M8 基线
              2. 试点方向 A + B 验证组合效果
              3. 等 Phase 2 M3 之后（有自动化稳定窗口期）再做
```

### 6.3 推荐：B + A + D 子集 三批渐进

| 批次 | 方向 | 工作量 | 验收 | 守密人参与 |
|------|------|--------|------|----------|
| **批 1** | B 主题入口 + 索引排除 session-digest 老旧档案 | 1 个会话（~3 小时）| 抽 10 个查询，看 Top-1 是否提升 | 1 次审定（30 分钟） |
| **批 2** | A 事实卡片化（试点 1 主题，见 § 八 Q8）| 2-3 个会话 | 该主题 RAG 召回 > 90% | 2 次审定 |
| **批 3** | D 子集（顶层目录分层但不动 session-digest）| 3-5 个会话 | 量化指标 M1 / M2 / M3 全部达目标 | 多次审定 |

**优点**：每批独立验证 + 可中途叫停 + 不与 Phase 2 三新使命冲突。
**缺点**：长尾 + 中间阶段会有「半新半旧」的过渡期。

### 6.4 不建议一次性重构的三个理由

1. **Phase 2 三新使命**已占满 84 天窗口（4-27 → 7-19），M1 / M2 需 wiki 自举推进，停摆代价高
2. **路径变动风险**：`memory/*.md` 被 `CLAUDE.md` / `BIAV-SC.md` / `boot_snapshot.py` / 各种 dispatch brief / decisions 等多处引用，一次性改路径会触发多文件级联失效
3. **Q4 基线还没建立**：在没有 M1-M8 实测数据的情况下做大重构，无法证明改进有效

**Q6 一句话答**：**渐进式 B+A+D 子集**，三批分次推进；一次性重构在 Phase 2 之后视基线数据再评估。

## 七、Q7 — 与现有 9 模块关系：替代 / 增强 / 并列

### 7.1 三种姿态对比

| 姿态 | 含义 | 9 模块去向 | 维护成本 | 风险 |
|------|------|-----------|---------|------|
| **替代（rip-and-replace）** | 引入新框架（如 Mem0 / Letta / Cognee）取代当前 9 模块 | 删除或归档 | 引入新框架的 onboarding | **极高**——9 模块刚跑通，立即推翻浪费投入 |
| **增强（layer on top）** | 在 9 模块之上加层（事实卡片层 / 主题入口层 / 分层目录），让 9 模块继续工作 | 保留，逐步优化 | 双层维护 | **低**——可逐步演化 |
| **并列（parallel tracks）** | 9 模块继续，新方案独立跑（如 dream 深睡跑 LLM 萃取写新索引）| 保留，独立运行 | 双索引维护 | **中**——双索引一致性问题 |

### 7.2 推荐姿态：**增强（layer on top）**

理由：

1. **9 模块刚跑通，工程沉没成本不可忽视**——`memory_search.py` 780 行 / `knowledge_graph.py` 217 节点 / `dream.py` 三层全部上线。Stage 1 验证已通过，Phase 2 自动化稳定运行依赖这套基础设施。
2. **业界先进方案没有任何一家「刚好」适合银芯**（vendor matrix § 4.3 已论证）——Mem0/Zep/Cognee 的 LLM 流水线会让月预算翻倍；claude-mem AGPL；Letta 的 agent-active core memory 与「凭据 / 决策」铁律冲突。**全替代不划算**。
3. **5 方向的 A/B/D 子集都是「在 9 模块之上加层」的形态**——事实卡片入 `assets/data/`、主题入口入 `memory/strategy/` 或 `memory/active/`、目录分层包装现有 md。**不破坏 9 模块**，只是给它们更好的输入。
4. **可逐批验证**——每加一层用 Q4 指标测一次，不达目标可回退该层。

### 7.3 增强方案的层次架构

```
┌────────────────────────────────────────────────────┐
│  新会话 / 艾瑞卡查询入口                            │
└────────────┬───────────────────────────────────────┘
             │
   ┌─────────▼──────────┐  ← 新增「主题入口分诊」层（方向 B）
   │ Subject Hub Lookup │     基于 query 关键词路由到入口页
   └─────────┬──────────┘
             │
   ┌─────────▼──────────┐  ← 新增「文档类别加权」层（方向 D 子集）
   │  Doc-Class Boost   │     index 时给每个 chunk 标 doc_class，rerank 时加权
   └─────────┬──────────┘
             │
   ┌─────────▼──────────┐  ← 现有 9 模块（保留）
   │ memory_search.py   │     TF-IDF + 4 维 rerank
   │ knowledge_graph.py │     223 节点 565 边
   │ context_manager.py │     4 层推荐
   │ dream.py + ...     │     浅 / 深 / REM
   └─────────┬──────────┘
             │
   ┌─────────▼──────────┐  ← 新增「事实卡片优先」层（方向 A）
   │ Fact-Card Override │     游戏数据查询时优先返回事实卡片
   └─────────┬──────────┘
             │
   ┌─────────▼──────────┐
   │  Top-K results     │
   └────────────────────┘
```

**三层增强**：分诊（B）→ 加权（D 子集）→ 卡片覆写（A）。原 9 模块完全保留，承载中间层。

### 7.4 与替代姿态的对比成本

| 对比项 | 替代（引入 Mem0）| 增强（本提议）|
|--------|----------------|--------------|
| 重写代码量 | ~1500 行（删 9 模块 + 接入 Mem0 + 适配） | ~500 行（三层增强）|
| 学习曲线 | Mem0 文档 + 19 vector store 选型 | 沿用已熟悉的 9 模块 |
| 月预算变化 | +$10 ~ $30（Mem0 自动萃取需 LLM）| 0 ~ +$5（仅可选 LLM 卡片化）|
| 可逆 | 难（删 9 模块后回退成本高）| 易（每层可独立移除）|
| Phase 2 时间影响 | **冲突**（重写期间停摆）| 不冲突（穿插）|

**Q7 一句话答**：**增强**——保留 9 模块，加 3 层（主题入口分诊 / doc-class 加权 / 事实卡片覆写）。**不替代**（投资浪费 + 风险高）；**不并列**（双索引维护成本高）。

brief 建议从「卡牌系统」或「战略规划」二者之一选试点。本节扩展到 5 个候选主题对比，最后给推荐。

### 8.1 5 个候选主题

| T2 | **战略规划** | `strategic-plan-2026.md` v1.0~v2.0 多代叠加 365 行 | 部分召回（v2.0 摘要可命中）| 高——历史不剥离典型，验证「版本化分文件」效果 |
| T3 | **直推 main 政策** | 散在 6 处（decisions / CLAUDE.md / methodology / lessons #28 #29 / CONTEXT 等）| 中 | 中——验证「主题入口」效果 |

| T5 | **银芯-黑池接口** | 多份 silver-blackpool / blackpool-architecture | 未测 | 中——但黑池已上线 |

### 8.2 推荐：T1 卡牌系统作主试点

**理由**：

1. **雏形最近 + 工作量最低**——`card-system.json` v1.0 已存在，扩展到 12 张卡片（覆盖卡牌系统全部公开实体）约 1 个会话。
2. **召回基线最差，提升空间最大**——baseline Q3 指令牌 0/5 命中、Q5 AwakerConfig 0/5 命中，做完后「指令牌 / AwakerConfig 类查询召回 > 80%」是可验证目标。
3. **覆盖游戏静态数据这一类知识**——回答了 brief Q3 「不同知识类型是否需要不同结构化策略」的关键论证（**事实卡片化对游戏数据效果最好**）。
4. **试点失败成本极低**——卡片是增量 JSON，删除不影响 9 模块；Phase 2 W1 角色自举受益（Code-wiki 可复用 schema）。
5. **跟 Phase 2 三新使命对齐**——使命 #2「社区共建知识底座」要求 wiki 数据完整，事实卡片化是必经路径。

### 8.3 试点失败可逆吗？

**完全可逆**：
- 卡片是新增 JSON 文件（`assets/data/*-cards.json`），删除不影响任何现有模块
- 9 模块继续按原方式索引 lua 原文，不依赖卡片
- 试点失败 = 删除新 JSON + 移除 schema 文档，无遗留

**可逆性最高的方向**——这是选 T1 的关键加分项。

### 8.4 试点成功后如何扩展

如果 T1 试点验证（M1 召回 > 80% / S1 守密人主观评定 OK）：

1. **横向扩展到 T3 / T5**（同模式：单一权威入口 + 引用源数据）
2. **触发 Code-wiki 接力**：把 72 角色 / 4 界域 / 命轮 / 关卡 都按事实卡片格式落档
3. **触发 Code-memory 接力**：在 `memory_search.py` 的 rerank 加「事实卡片优先」加权（Q7 § 7.3 第三层）
4. **进入批 2 -> 批 3 节奏**（参见 Q6 § 6.3）

### 8.5 副试点候选

如果守密人想同时验证两个方向，副试点推荐 **T2 战略规划**（验证「版本化分文件」效果）：

- 将 `strategic-plan-2026.md` 拆成 `strategic-plan-v2.0-active.md`（当前版）+ `strategic-plan-archive/v1.0.md` ~ `v1.2.md`（历史版）
- 在 `boot-snapshot.md` 只引用 active 版
- 测试 M8「失效信息率」是否下降

但 T2 改动会触发多处引用更新（CLAUDE.md / boot_snapshot.py / 各 dispatch brief），**风险高于 T1**。**不建议同期并行**。

**Q8 一句话答**：**T1 卡牌系统**作主试点（雏形最近 / 基线最差 / 完全可逆 / 与 Phase 2 对齐），1-2 个会话可验证；成功后横向扩展 + 触发 Code-wiki / Code-memory 接力。**T2 战略规划**作备选，待 T1 验证后再启动。
