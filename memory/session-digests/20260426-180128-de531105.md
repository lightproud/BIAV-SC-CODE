## 一、银芯当前的 5 个 hook 联动

`.claude/settings.json` 注册 4 个仓库级 hook，加上守密人个人级 1 个全局 hook，总共 5 个事件钩子在每次会话中循环触发：

| # | Hook 事件 | 触发时机 | 执行脚本 | 作用 | 来源 |
|---|----------|---------|---------|------|------|
| 1 | **SessionStart** | 会话刚启动 | `.claude/hooks/session-start-sync.sh` | fetch + sync local main 至 origin/main | 仓库级 |
| 2 | **UserPromptSubmit** | 守密人每次提交输入前 | `scripts/session_inject.py` | 检索 session-digests 注入相关历史片段（守密人在本会话开头看到的「档案回溯」段落就是它产出的） | 仓库级 |
| 3 | **PostToolUse** | 艾瑞卡每次调用工具后 | `scripts/session_watch.py` | 实时追加 progress.jsonl 记录工具调用 | 仓库级 |
| 4 | **SessionEnd** | 会话**关闭**时 | `scripts/session-end-distill.sh → session_distiller.py` | 把整段 transcript 蒸馏成结构化 markdown digest | 仓库级 |
| 5 | **Stop** | 艾瑞卡每轮输出结束 | `~/.claude/stop-hook-git-check.sh` | 检查工作树干净，未提交则报警 | **守密人个人级** |

注意 #4 和 #5 触发节奏完全不同——**SessionEnd 一次会话只触发 1 次**（会话彻底关闭），**Stop 每轮 turn 末尾都触发**。

## 三、为什么 SessionEnd 在会话进行中触发了？

这是 Claude Code 平台行为，不是银芯设计的：

1. **Claude Code 4.x 在长会话中会做内部重启 / compaction** —— 当上下文接近上限时，平台会把当前对话「打包」存档（Claude 这边表现为 SessionEnd），然后开新底层 session 续接（表现为 SessionStart）。**对守密人和艾瑞卡来说是无感的**，但 hook 看见的事件流是「end → start」一次循环。
2. 这次很可能是 17:39 触发了一次 compaction（我前面读了大量长文档 + 推了 10 个 commit，token 消耗较大），SessionEnd 因此被调用了一次。
3. session_distiller.py 把当时的 transcript 蒸馏成 digest 写盘——这是**预期行为**，是银芯记忆系统的核心能力之一（每次 compaction 都留档案，避免长会话信息丢失）。

## 四、为什么 distill 脚本不自动 git commit？

设计取舍。看 `session-end-distill.sh` 注释：

> Output/exit code are ignored by Claude Code for SessionEnd hooks, so all logs go to /tmp/session-distill.log for later inspection.

含义：
- SessionEnd hook 的退出码 **Claude Code 不读取**
- 这意味着如果在 hook 里 `git push` 失败（如网络抖动、冲突、auth 过期），错误会**完全沉默**——digest 写盘了，但没推上去
- 设计者选择**不在 hook 里做 git 操作**，只写文件；commit/push 由下一轮 Claude Code 会话发现 untracked 时再做（或由守密人手工同步）

这是「**软失败优于沉默推送失败**」的取舍。代价就是会出现守密人看到的这种「digest 写盘了但还 untracked」的窗口期。

## 五、Stop hook 报警的意义

`~/.claude/stop-hook-git-check.sh` 是守密人**个人级**全局钩子（`~/.claude/` 而非 `${repo}/.claude/`），它对**所有仓库**都跑，逻辑是：

- 每次艾瑞卡输出完一轮 → 检查 git status
- 不干净 → exit 2 + 提示守密人

**这个 hook 的设计意图正是为了兜住上面 §四 的窗口期**——既然 distill 不会自动 push，至少 Stop 会在艾瑞卡输出后立刻提醒补 commit，避免文件长期 untracked。

| # | 改进 | 工作量 | 风险 |
|---|------|--------|------|
| A | session-end-distill.sh 末尾加 `cd $REPO_ROOT && git add memory/session-digests/ && git commit -m "chore(memory): session digest" && git push origin main \|\| true` | 小 | 网络抖动会沉默失败但不阻塞会话；推送堵塞时会累积 |
| B | session_distiller.py 内部完成后 invoke 一个独立的 `commit-digest.sh` 异步推送（带退避重试） | 中 | 增加复杂度但更稳健 |
| C | 维持现状，靠守密人 stop-hook 提醒手动补 commit | 0 | 偶发 untracked 窗口期会一直存在 |

## 八、总结

**这次事件的本质**：
1. 长会话触发 Claude Code 内部 compaction
2. SessionEnd hook 写出本会话至此为止的 digest（**这是好事**——保留了快推 10 个 commit 的过程档案）
3. distill 故意不自动 git commit（设计取舍：软失败优于沉默推送失败）
4. Stop hook 检测到 untracked 报警（**这是它的本职**）
5. 艾瑞卡看到守密人转发的报警，立刻 commit + push

整条链路**按设计运转**，不是 bug。但「digest 写完 → 守密人手动提醒 → 艾瑞卡补 commit」这个三段式有 friction，§七 改进 A 是最低成本优化。
