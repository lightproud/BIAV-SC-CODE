## 问题

`claude.ts` 原来的代码：

```typescript
// stream() 方法里
const tools = config.tools.map((t) => this.toAnthropicTool(t, config.cacheControl));

// toAnthropicTool — 每个工具都标记了 cache_control
private toAnthropicTool(tool, cacheControl: boolean) {
  const result = { name, description, input_schema };
  if (cacheControl) {
    result.cache_control = { type: 'ephemeral' };  // <-- 所有工具都加
  }
  return result;
}
```

注释写的是 "Mark the **LAST** tool"，但代码实际是给**每个工具**都打了 `cache_control` 标记。

## 为什么这是严重问题

Anthropic 的 prompt cache 是**前缀匹配**（prefix-matched）机制：

```
请求内容：[system prompt] [tool_1] [tool_2] [tool_3] [messages...]
                                              ↑
                                        cache breakpoint
```

API 从请求开头向后扫描，遇到 `cache_control: ephemeral` 标记的位置就设一个 cache breakpoint。**只有 breakpoint 之前的内容会被缓存**。

正确做法是只在**最后一个工具**上标记 — 这样整个 `[system + 全部 tools]` 前缀都会被缓存，后续轮次命中时省掉这段 token 的费用（cache hit 只收 0.1 倍价格）。

在每个工具上都标记的问题：
- 每个 `cache_control` 标记都会创建一个 breakpoint，产生额外的 cache_write 开销
- cache_write 是正常 input 的 **1.25 倍**价格
- 多个 breakpoint 不会比一个 breakpoint 缓存更多内容 — 前缀匹配的特性决定了只需要在**前缀末端**标一次
- 结果：白花 cache_write 费用，没有额外收益

## 修复后

```typescript
// stream() — 只给最后一个工具传 true
const tools = config.tools.map((t, i) =>
  this.toAnthropicTool(t, config.cacheControl && i === config.tools.length - 1),
);

// toAnthropicTool — 参数名改为 isLastAndCacheable，语义清晰
private toAnthropicTool(tool, isLastAndCacheable: boolean) {
  const result = { name, description, input_schema };
  if (isLastAndCacheable) {
    result.cache_control = { type: 'ephemeral' };
  }
  return result;
}
```

加上 system prompt 上也有一个 `cache_control`（`stream.ts` 里构建的），整个请求的缓存结构就是：

```
[system prompt (cache breakpoint 1)] [tool_1] [tool_2] ... [tool_N (cache breakpoint 2)] [messages...]
```

两个 breakpoint：system prompt 单独缓存（因为它跨对话不变），tools 数组整体缓存（切换 gear 时重发一次）。后续轮次只有 `[messages...]` 部分按正常价格计费，前面的全部走 cache hit（0.1 倍）。

## 对 token 经济的影响

假设 BPT 挂 6 个工具，每个 schema ~500 token = 3000 token tools 总量：

| 场景 | 每轮 cache_write 开销 | 后续轮 cache_hit |
|------|----------------------|-----------------|
| 修复前（标记全部 6 个） | 6 次 cache breakpoint write | 相同 |
| 修复后（只标记最后 1 个） | 1 次 cache breakpoint write | 相同 |

首轮 cache_write 成本降为 1/6，后续轮 cache_hit 效果一样。这正是计划书 T1 红线要求的 "schema 首轮发 + cache，后续轮必须命中"。

## 黑池终端到底要做什么

从计划书和这几轮实际建设来看，BPT 的核心命题不是"做一个聊天界面"，而是：

### 第一堵墙：模型自由

不是"支持多模型"这么简单。真正的自由是：Light 在讨论剧情时用便宜的 Haiku，转头让它改配置时切到 Sonnet，遇到复杂架构决策时拉 Opus — 切换不重启、不清对话、不丢上下文。MODEL_REGISTRY 已经铺好了基础，但最终体验应该是用户几乎不用想模型这件事，档位切换自动匹配合适的默认模型。

2. Token Meter 一行挤了 9 个数字（sys/tools/hist/gen/cache-hit/cache-write + total + cache% + cost）— 信息过载，非程序员看不懂 "sys: 2.1k" 是什么
3. 工具调用显示原始 JSON（`{"query": "暴击率"}`）— 非程序员看到 JSON 会困惑
4. 工具结果在对话气泡里内联显示 — 长结果撑爆对话流

**改善**：

**Token Meter 分层**：
- 默认只显示一行：`总计 12.3k tokens | cache 85% | ~0.02 USD (0.14 CNY)`
- 点击展开看 6 维明细
- cache < 80% 时明细自动展开 + 用中文提示"缓存命中率偏低，建议检查工具配置"

**工具调用人性化**：
- 不显示 JSON，显示自然语言：`正在搜索黑池代码: "暴击率"` / `正在查询银芯记忆: "狂战士"`
- 工具名 → 中文动作映射表：`bpe_semantic_search` → "搜索黑池代码"，`memory_search` → "查询银芯记忆"
- 工具结果折叠，默认显示摘要行（"找到 5 条结果"），点击展开看细节

**对话标题**：
- 第一条用户消息发送后，用消息前 20 个字作为标题
- 或者（更好）下一轮 AI 回复时让 AI 顺便起个标题（但这会增加 token 开销，不如直接截取用户消息）

## 场景 4：档位切换

**当前体验**：
- 输入框左边一个小按钮 "Chat" / "Work"，text-xs
- 切到 Work 时弹确认对话框
- 状态栏也显示档位

**问题**：
1. 按钮太小，text-xs 级别 — 容易忽略
2. 每次切 Work 都弹确认框 — 频繁切换时很烦
3. Chat 模式和 Work 模式在对话区没有视觉差异 — 用户可能忘了自己在哪个档

**改善**：
- 按钮放大，加明确图标或更重的视觉权重
- 确认框只在**首次**切换时出现，之后记住用户选择（config 存一个 `gearConfirmShown: boolean`）
- Work 模式时对话区添加细微视觉线索 — 比如输入框边框变成 warning 色（橙色），或者顶部出现一条细横条

## 场景 5：BPE 面板浏览 + @Cite

**当前体验**：
- 右侧面板 w-80（320px）打开 BPE 搜索
- 输入关键词 → 看到代码切片（每个最多 300 字符预览）
- 点 @Cite → 对话输入区上方出现 cite chip
- 发消息时 cite 跟着一起发

**问题**：
1. 320px 看代码极其拥挤 — C# 一行经常 80+ 字符，全部折行
2. 300 字符预览太短 — 一个函数可能 500+ 字符，看不到关键部分
3. @Cite chip 在输入框上方水平排列 — 多个 cite 时挤压输入空间
4. 没法预览 cite 的完整内容 — 点了就注入，不确定注入了什么

**改善**：
- BPE 面板支持**双击结果展开**看完整切片（仍在面板内，但该切片展开为全高度）
- @Cite chip 显示文件名 + 行号，hover 显示前几行预览 tooltip
- 面板宽度可拖拽调整（或提供 w-80 / w-120 两档）

## 场景 6：长对话中的压缩

**当前体验**：
- 超过 20 轮或 token 超预算 → 自动压缩
- 压缩后在对话顶部插入一条 "[Compressed context — N earlier turns summarized by AI]"
- 用户看不到被压缩的内容

**问题**：
1. 压缩无预警 — 用户突然发现早期消息消失了
2. 压缩不可撤回 — 如果压缩丢了关键信息，没法恢复
3. "summarized by AI"文本是英文 — 应该中文

**改善**：
- 压缩前提示用户："对话较长，将压缩早期内容以节省 token，保留最近 10 轮原文。确认？"
- 压缩摘要可展开查看（"AI 认为这些是关键点"）
- 计划书说"用户可手动触发/可撤回" — 当前代码没实现手动触发和撤回

---

## 场景 7：状态感知

**当前体验**：
- 状态栏在最底部：MCP 状态 + BPE 状态 + 档位 + 版本号
- 用绿/红点表示连接状态

**问题**：
1. 状态栏在最底部，用户几乎不看
2. MCP 断开时没有主动提醒 — 用户可能在 MCP 断了的情况下发了好几条消息，AI 无法调工具但不会报错（只是没工具可用）

**改善**：
- MCP/BPE 断开时，在对话区顶部显示一条 warning banner："银芯连接断开，记忆搜索不可用"
- 重连后自动消失

## 汇总：优先级排序

| 优先级 | 改善 | 理由 |
|--------|------|------|
| **P0** | 首次启动引导 + Settings 自动打开 | 没有这个用户连第一步都走不动 |
| **P0** | Token Meter 分层（默认精简、点击展开） | 当前 9 个数字一行，非程序员看不懂 |
| **P1** | 工具调用中文化显示 | 原始 JSON 对非程序员不友好 |
| **P1** | 首条消息自动生成对话标题 | "New Conversation" x 20 完全无法区分 |
| **P1** | MCP/BPE 断连 banner | 无声失败是最差的体验 |
| **P2** | 档位确认只弹一次 | 频繁切换的摩擦 |
| **P2** | BPE 面板宽度 + 预览改善 | 320px 看代码太挤 |
| **P2** | 压缩预警 + 手动触发 | 计划书要求但未实现 |
| **P3** | @Cite chip hover 预览 | 锦上添花 |
