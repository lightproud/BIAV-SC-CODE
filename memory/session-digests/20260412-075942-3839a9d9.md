### 1. cache_control 是核心防线

BPT 的 `cache_control: ephemeral` 设计意味着：
- **system prompt + tool schema** 每轮都发送，但走缓存命中 = **0.1x 价格**

### 2. 历史压缩是第二防线

BPT 的 compressor（T3 规则）在以下条件触发：
- 对话超过 **20 轮** 或超过 **120k token**
- 触发后，早期对话被 Haiku 压缩成摘要（几百 token），只保留最近 K 轮原文
- 这直接控制了"历史越来越长"的膨胀问题

### 3. 实际成本结构

以 Sonnet 4.6 为例（$3/MTok 输入），一个 20 轮对话：

| 部分 | 每轮 token | 10 轮累计 | 有缓存后 |
|------|-----------|-----------|----------|
| System prompt | ~300 | 3,000 | 300 + 2,700×0.1 = **570** |
| Tool schema | ~800 | 8,000 | 800 + 7,200×0.1 = **1,520** |
| 历史消息 | 递增（~500/轮） | ~27,500 | **27,500**（无法缓存）|
| **总输入** | | ~38,500 | **~29,590** |

成本：29,590 / 1M × $3 = **$0.089**（约 0.65 元人民币 / 10 轮对话）

历史消息是增长最快的部分，但它**不能被缓存**——这正是压缩器存在的原因。当历史超过阈值，压缩后回落到可控范围。

窗口大小只是容量上限。你实际用了 30k token 就只付 30k 的钱。1M 窗口的意义是：**你不会被截断**，长对话不用丢失上下文。但日常使用中，压缩器会确保历史维持在合理范围。
