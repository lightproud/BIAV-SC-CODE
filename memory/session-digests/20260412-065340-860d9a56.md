现在全貌清楚了。具体的追溯机制如下：

## 双轨制采集架构

### Track 1：增量采集（每小时，`update-news.yml`）
```
aggregator.py → collect_global.py → split_output.py → archive_platforms.py
```
- 每小时跑一次，只看 **最近 24h** 的数据
- `news.json` 每次**覆写**（不追加），只保留 24h 窗口内的内容
- 最后 `archive_platforms.py` 把 `*-latest.json` 里的条目**按日期归档**到 `data/platforms/{source}/YYYY-MM-DD.json`
- 归档是 **merge-append + dedup**：已存档数据永不丢失

### Track 2：历史回溯（每小时，`backfill-news.yml`）
```
backfill_platforms.py --platform {name} --pages 5
```
- 9 个平台并行，每平台一个独立 job
- **翻页式向历史深处爬取**，不受 24h 窗口限制
- 用 `state.json` 记录每个平台爬到了第几页，下次从断点继续
- 爬到的数据直接写入日期归档文件（跳过 `news.json`），不进实时管线

### 具体翻页逻辑（以 Bilibili 为例）

```
state.json: {"bilibili": {"page": 5, "done": false, "total": 230}}
```

1. 读 state → 上次爬到第 5 页
2. 从第 5 页开始，每次翻 `--pages 5` 页（即爬 5~9 页）
3. 每页调 Bilibili 搜索 API（`keyword=忘却前夜`, `order=pubdate`, `page=N`）
4. 拿到结果 → 按 `time` 字段分组到对应日期
5. 写入 `data/platforms/bilibili/2026-04-08.json`（与已有数据合并去重）
6. 更新 state → `{"page": 10, "total": 480}`
7. 如果某页返回 0 结果 → `done: true`，后续跳过

### 9 个回溯平台

|------|----------|----------|

| arca_live | HTML 列表分页 | 空结果 |

| naver_cafe | 搜索 API 分页 | 空结果 |
| pixiv | 搜索 API 分页 | 空结果 |

### 安全机制
- **30 分钟超时**：`MAX_RUNTIME_SECONDS = 1800`，防止单次跑太久
- **1.5s 请求间隔**：避免触发反爬
- **状态持久化**：每翻一页就写 state.json 并 git push，中断不丢进度
- **done 标记**：爬完的平台自动跳过

## Assistant [92] 06:48:57

意思是这样的：

**增量采集只看最近 24h 的内容。** 比如 Reddit，每小时跑一次 aggregator，调 API 拿「最近 24 小时内发布的帖子」。

假设 CI 在 4 月 10 日 00:00 停了，4 月 12 日 00:00 恢复：

- 4 月 10 日的帖子：停机时 < 24h，本该被采集，但 CI 没跑 → 没人采
- 4 月 12 日 CI 恢复：调 API 拿「4 月 11 日 00:00 ~ 4 月 12 日 00:00」的帖子
- **4 月 10 日的帖子已经超出 24h 窗口，API 不会返回了 → 永久丢失**

而有回溯脚本的 9 个平台（bilibili、appstore 等）不受这个限制，因为 `backfill_platforms.py` 是**按页码翻历史**，不按时间窗口过滤，所以即使停机也能在恢复后把旧数据补回来。

剩下的 Reddit、NGA、Discord、YouTube、Twitter 等 **没有回溯脚本**，只靠增量采集那条线，所以停机超过 24h 就真的丢了。

**实际影响有多大？** CI 每小时跑一次，GitHub Actions 很少停 24h 以上，所以正常情况不太会触发这个问题。但如果你想要更安全的保障，可以给更多平台加回溯——要做吗？
