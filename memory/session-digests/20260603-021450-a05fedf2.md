## 判断：不是 gitignore，是补全已有的自动提交缺口

排查发现 SessionEnd 脚本**早就有一段自动提交逻辑**，但它只覆盖了 `session-digests/`，漏掉了 `session-continuity.json`：

```bash
# session-end-distill.sh 现状（只管 digests）
DIGEST_DIR_REL="memory/session-digests"
if [[ -n "$(... git ls-files --others --exclude-standard -- "$DIGEST_DIR_REL")" ]]; then
    git add -- "$DIGEST_DIR_REL"

```

这就是病灶：distiller 在会话末改写 `session-continuity.json`（一个**已跟踪**文件 → 变 ` M`），但这段自动提交只 `git add` digests 目录、且只匹配**未跟踪**文件，于是 continuity 的改动被漏下，悬在工作区漏进下一个会话，被**每回合级**的 Stop hook 反复拦。艾瑞卡这两次噪音 commit 就是这么来的。

**为什么不 gitignore：** 这文件的全部价值是「跨会话连续性」。远端容器每次从零克隆，跨会话 = 必须 git 持久化。`session-digests/` 几百个文件都进了 git、`session_briefing.py` 开场就读它——这套记忆系统是**设计上要跨容器持久化**的。gitignore 会直接斩断远端的连续性链。

## 推荐改法：把 continuity 并入同一条自动提交通道

```bash
CONTINUITY_REL="memory/session-continuity.json"
DIGEST_DIR_REL="memory/session-digests"
cd "$REPO_ROOT"
git add -- "$DIGEST_DIR_REL" "$CONTINUITY_REL" 2>>"$LOG_FILE" || echo "  add failed (non-fatal)" >>"$LOG_FILE"
if ! git diff --cached --quiet -- "$DIGEST_DIR_REL" "$CONTINUITY_REL" 2>/dev/null; then
    git commit -m "chore(memory): session digest + continuity auto-commit (SessionEnd hook)" \
        2>>"$LOG_FILE" || echo "  commit failed (non-fatal)" >>"$LOG_FILE"
    git push origin main 2>>"$LOG_FILE" \
        || echo "  push failed (non-fatal, will retry next session)" >>"$LOG_FILE"
fi
```

改动效果：

1. continuity 在**会话末**随 digest 一次性提交推送，时机正确、机器自动
2. 下一个会话开场工作区是干净的 → **每回合 Stop hook 再也看不到它** → 噪音循环根除
3. 走和 digest 同一条 `push origin main` 通道，**永不再碰特性分支 / PR**——这才是这类记账该走的路（PR #168 本就是错的载体，是这个 hook 缺口逼出来的产物）

这是补全既有 pattern、不是新发明（§6.3 外科手术式）：复用了原脚本的 soft-fail 风格和 commit message 措辞。

唯一沿用的既有怪癖：原脚本 `git push origin main` 在特性分支上语义存疑（commit 落在当前分支、push 的却是 `main` ref）——但这是**既有行为**，艾瑞卡按 §6.3 原样匹配、不顺手重构。
