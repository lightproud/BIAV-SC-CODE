## 搬运包:并发槽占道治理(治「卡死 / 没反应」第一刀)

**指导主题**:让「等人」的轮次交还并发槽,让排队的人看得见队伍——消除堵死三车道的两类路霸。

**背景上下文**(BPT 会话可能缺的):

1. BPT 全局并发闸上限 3(stream-semaphore,6 月 5 日为防并发挤压误判掐线而设),acquire 在 chat-sdk 轮次入口一带,槽位在 finally 释放。
2. 6 月 10 日 50 轮遥测实测:一轮 t_api 仅 60 秒、进程却持槽存活约 8 小时后干净退出——推断该轮停在等用户审批,人没点、槽不还。同期 t_queue 极值 167 秒,cancel-events 日志 6 月 8-9 日成批出现「Aborted while waiting for a concurrency slot」。
3. 用户侧连锁:槽被占满 → 新消息静默排队或静默放弃 → 用户以为没发出去 → 重发 → 复印 bug 的破碎重发现场。

**具体建议**(按优先级,三条):

**第一条(命门):审批等待期释放槽位。**
轮次进入「等用户批准工具调用」状态时 release 槽,用户响应后重新 acquire 再继续。
比喻:在银行窗口排到号,却站在窗口接了两小时电话——后面全堵死。改成接电话先让位,打完凭回头票优先插队。
三个陷阱必须防:
- **先验证假设再动刀**(Think Before Coding):先在代码里确认审批等待确实发生在持槽区间内。若审批其实在 acquire 之前,本条不成立,8 小时轮要另查(比如 MCP 工具挂起)。
- **重 acquire 给插队优先权**:审批归来的轮次优先于新轮次,否则人批准完还要再排 40 分钟。
- **槽柄一次性化防双重释放**:release 过的柄作废,finally 见作废柄跳过。比喻:一次性门票,撕过角就作废——否则同一本书登记归还两次,账上凭空多出可借名额,限 3 名存实亡。

**第二条:排队可见、放弃不静默。**
acquire 等待时向 UI 上报「排队中,前面 N 轮」;等待超时不再静默 abort,给出明确可重试的提示;遥测补 queue_abort 事件,区分「用户主动取消」与「等待超时」。
比喻:取餐叫号器没电,排到了也没人喊,你以为没下单成功就又下一单——这就是复印的用户侧成因。

**第三条(保险丝,排后):占槽看门狗,只告警不掐。**
单轮持槽超阈值(建议先 30 分钟)只打遥测告警,**不强制释放**,收集一周分布再定是否强制。
比喻:先装摄像头看清谁老占车位、占多久,看一周再立拖车规矩——直接拖会误拖合法装卸货的车(巨型对话单轮 8-40 分钟是合法的),上次复印 bug 正是误判掐线烧出来的。

## 自建工具层的三种典型病,各有指纹

**病型一:spawn 底噪(嫌疑最大,因为和暗格量级吻合)。**
原生工具是进程内函数调用,微秒级;自建工具如果走「每次调用起一个子进程/脚本」的形态(BPT 对接 MCP 时正是 spawn 子进程的模式),每次都重付解释器启动费,几百毫秒一次。一轮 agent 任务调几十次工具很正常——**每次 300ms × 40 次调用 = 12 秒;若再叠加每次重连/重载,凑足暗格里几十秒毫无压力**。
比喻:每次叫人搬个凳子,都要先给他办一遍入职手续——活十秒,入职半分钟。
数据指纹:每次调用都有一截**固定底噪**,快的工具和慢的工具起步价一样。

**病型二:网络长尾(被 ban 替代实现的特产)。**
网络工具被 ban 后的替代路径(代理中转/镜像/换端点),天然带「试错链」:先试的端点要等到超时才认输、再 fallback。超时若是默认 30 秒,一次失败就白吃 30 秒。
比喻:封路后绕乡道,平时只是慢一点,但偶尔撞上断头路,司机非按导航死等半分钟才肯掉头——超时设太长,就是认输认得太慢。
数据指纹:**长尾分布**——多数次数正常,偶发一次 30-60 秒的尖刺。这和「有时快有时慢」的体感最吻合。
旁证:lessons 档案里 idealab 网关那条(`/code/` 端点 401、必须锁定特定路径)早就证明 BPT 的网络是「网关+受限」形态,连 API 主通道都要走特殊路径——环境税真实存在。
