## 诊断：当前系统的根本矛盾

九个模块的现状是**各自为政的工具箱**，不是**自我强化的循环系统**。

具体断裂点：

| 断裂 | 现象 |
|------|------|
| **会话间失忆** | 每次新会话读 boot-snapshot.md，但它是静态快照——不知道上次会话做了什么、留下什么未完成 |
| **摘要只写不读** | 11 份 session-digest 存在 git 里，但没有任何模块消费它们 |
| **事实库孤立** | facts.json 有 9 条记录，但搜索、图谱、做梦都不引用它 |
| **MemRL 信号稀疏** | 只靠做梦层的 access-log 计算效用，不知道文件在会话中是"读了就忘"还是"深度使用" |
| **REM 有形无实** | 周报只有 hot_topics 计数，ai_reflection 是空对象 `{}`，insights.json 是空数组 |
| **Reflexion 被动** | 只在做梦时扫描失败信号，不在会话中实时反馈 |
| **上下文管理无惯性** | 不知道最近三次会话都在做 wiki，仍从零推荐 |

**本质问题：数据产生了但没有回路。每个模块都在"写"，但没人在"读+用+反馈"。**

---

## 方案：记忆飞轮 (Memory Flywheel)

核心思想：每次会话既**消费**前序积累的知识，也**产出**结构化数据反哺系统。每一轮循环都让下一轮更强。

```
            [1] 会话智能启动
                 ↓ 读取 briefing：上次做了什么、发生了什么变化、今天该关注什么
            [2] 会话中实时增强
                 ↓ 搜索结果自动关联历史会话，事实实时写入，反馈信号持续采集
            [3] 会话结晶
                 ↓ 结构化元数据提取（主题、决策、未竟事项），写入连续性链
            [4] 睡眠时智能进化
                 ↓ 做梦层消费所有会话元数据，发现跨会话模式，校准权重，填充洞察
            [1'] 下次会话启动 ← 更精准的 briefing、更聪明的上下文推荐、更丰富的图谱
```

**飞轮效应**：会话越多 → 元数据越丰富 → REM 模式越清晰 → briefing 越精准 → 会话效率越高 → 产出更多高质量元数据。

### 工程五：MemRL 参与度跟踪 (Engagement-Weighted Utility)

**目标**：从"文件是否被扫描过"升级为"文件在会话中的真实价值"。

**当前信号**：access_log 只记录"做梦时扫描了哪些文件"。

**新信号来源**：`.meta.json` 的 `files_engagement`：

```python
# 参与度阶梯权重
ENGAGEMENT_WEIGHTS = {
    "read_only": 0.3,        # 读了但没用
    "read_and_edited": 0.7,  # 读了并修改 → 有价值
    "read_edit_commit": 1.0, # 读、改、提交 → 高价值
    "search_hit_used": 0.5,  # 搜索命中且后续被引用
    "search_hit_ignored": 0.1, # 搜索命中但没使用
}
```

**效用计算公式升级**：

```
旧: utility = 0.4 * access + 0.3 * insight + 0.3 * recency
新: utility = 0.3 * engagement + 0.25 * insight + 0.25 * recency + 0.2 * momentum
```

新增 `momentum` 维度：文件在最近 N 次会话中的连续出现次数。连续出现 = 持续重要。

**搜索 reranker 动态校准**：
- `calibrate_reranker_weights()` 用 engagement 数据替代简单的方差分析
- 如果"高参与度文件"总是在搜索结果前 3 → 当前权重合适
- 如果"高参与度文件"总是排在搜索结果后面 → 调整权重
