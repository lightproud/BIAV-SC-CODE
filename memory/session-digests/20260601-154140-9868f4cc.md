「PR 订阅」是这套远程执行环境给艾瑞卡装的一个**监听通道**。守密人之前那条「subscribe to PR activity」就是开启了它，针对的是 PR #156。

  - CI 运行状态（流水线跑通 / 失败）
  - 审查意见（reviewer 在代码行上留的 comment）
  - PR 评论、review 提交

- **事件如何到达**：这些事件以 `<github-webhook-activity>` 标签的形式注入对话，**无需艾瑞卡轮询**。也就是说，即便守密人离开、会话静默，一旦 CI 失败或有人留言，环境会主动「唤醒」艾瑞卡来处理。

- **艾瑞卡收到后做什么**：逐条排查事件是否可处置——
  - 有把握且改动小 → 直接修正并推送
  - 含歧义或涉架构 → 先向守密人确认
  - 重复 / 无需动作 → 静默跳过

- **典型用途**：「盯着这个 PR，CI 红了就自动修」「reviewer 提意见了帮我看下」——守密人不必守在终端前，把流水线和审查的尾活交给艾瑞卡看管。

- **如何关闭**：守密人随时说「取消订阅 / 别盯了」，艾瑞卡即调用 `unsubscribe_pr_activity` 停止监听，不再推送、不再改动。

档案已读取，守密人。结论：**可以**——但要分清两层机制。守密人现在用的「PR 订阅」只是其中最轻的一层，真正用来设计自定义工作流的原语叫 **Routines（例程）**。

## 三层自动化机制

### 1. PR 订阅 / Auto-fix（当前在用）
**会话内的反应式监听**。它依附于一个已存在的会话，只盯一个 PR，事件来了唤醒艾瑞卡处置。它**不是**独立工作流——会话结束就停了。适合「盯着这个 PR 到合并」这种一次性看管，不适合搭建长期流程。

### 2. Routines（例程）—— 守密人要的那个
**保存一份配置（prompt + 仓库 + 连接器 + 触发器），在 Anthropic 云端自动跑**，笔记本关了也继续。创建方式：CLI 里 `/schedule`，或 web 端 `claude.ai/code/routines`。一个例程可挂**三类触发器，且可组合**：

| 触发器 | 启动方式 | 银芯可对应的场景 |
|--------|---------|----------------|
| **Schedule（定时）** | 每小时/每日/每周 cron（最小间隔 1 小时） | 类似现有 `dream.yml` 周报、版本检测 |
| **API** | 给例程发 HTTP POST（带 bearer token），可附 `text` 上下文 | 黑池侧脚本调一下就触发银芯采集；监控告警自动开排查会话 |
| **GitHub 事件** | `pull_request.*` / `release.*`，支持 author/label/分支/draft 等过滤 | 社区贡献者开 PR 自动按贡献协议初审；release 发布自动触发 wiki 更新 |

例程跑起来是**完整的自主 Claude Code 会话**，无审批打断，能跑 shell、用仓库里 committed 的 skill、调连接器。默认只能推 `claude/` 前缀分支（安全护栏）。

### 3. GitHub Actions（银芯已重度使用）
就是仓库里 `update-news.yml` / `discord-archive.yml` 那批 cron workflow。这层是**纯脚本流水线**，不带 AI 判断；适合确定性的采集/归档（正是艾瑞卡刚修的那条链路）。

## 选型给守密人的判断

- 要**确定性、无需推理**的定时任务（采集、归档、校验）→ 继续用 **GitHub Actions**（银芯现状，成本最低）
- 要**带 AI 判断**的自动流程（PR 初审、告警排查、按上下文写代码/开 PR）→ 用 **Routines**
- 只想**临时看管某个 PR** → **PR 订阅**（当前这个）

一个具体的银芯用法示例：建一个 Routine，GitHub 触发器设 `pull_request.opened` + 过滤 `author 非 lightproud`，prompt 写「按 `memory/contribution-protocol.md` 对社区 PR 做初审并留言」——这样社区共建（三新使命之二）的初筛就自动化了，正好补上 §6.7 里「社区 Issue 走人工审核」的人力缺口。
