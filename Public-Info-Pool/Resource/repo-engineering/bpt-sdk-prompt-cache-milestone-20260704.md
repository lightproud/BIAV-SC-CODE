# BPT Agent SDK：提示词与缓存这条线的里程碑综述

> 类型：repo-engineering ｜ 日期：2026-07-04 ｜ 作者：艾瑞卡会话（守密人派发）
> 定位：把「官方公开文档 → OpenCode 开源实践 → 缓存 0% 诊断 → v2/v3 净室提示词 →
> 多层注入设计 → SDK 分段 seam」整条工作线收束成一份可追溯的账。
> **红线**：本线全程只学习**公开文档 + 开源实践**，**绝不**吸收任何流传的官方专有系统提示词原文（泄漏≠公开）。

## 0. 一句话

这条线回答的是守密人一个忧虑——「整体效率低下会累积成巨大差别」。做法不是拍脑袋优化，而是
**先量后改**：把效率变成可测数字，诊断出缓存为什么是 0，据此写出我们自己的净室提示词（v2/v3），
并把「组织级缓存红利」的设计留在 host、只在 SDK 长一个通用 seam。**每一步结论都带证据标签：
【已测】/【已设计待测】/【结构性上限】。**

小学生比喻：不是凭感觉给车做保养，而是先装仪表盘看油耗，再决定换哪个零件——换完还要再上路测一圈才认。

## 2. 诊断：缓存为什么是 0%（【已测】，不是 bug）

早前实测短任务缓存命中 0%，一度以为是 cwd 变动打碎前缀。**run #40 证伪了这个假设**——诚实纠错后，
用「写/读分离」诊断（`cache_creation` vs `cache_read`）定位到真因：

- **短任务**：`cache_creation=0`——前缀根本够不着 Haiku 的**最小可缓存前缀 2048 token 门槛**（Sonnet/Opus 是 1024），
  所以服务端压根不写缓存，谈何命中。
- **长任务**（id 8/9，逼出 20+ 轮）：写+读都出现，命中率 45–57%。**证明缓存机制本身没坏**，0% 是短任务前缀太瘦的症状。

**关键认知（且是纪律）**：0% 不是缺陷，是我们提示词**精简 = 成本优势**的副作用。
**灌水凑 2048 门槛是自毁**——为省 10% 缓存读价，先付 100% 的膨胀写价，净亏。所以**不为缓存而灌水**。

小学生比喻：复印店满 20 页才给存档折扣。我们的文件只有 5 页，享受不到折扣是正常的——
为了折扣硬塞 15 页废纸进去，反而多付了 15 页的钱，赔本。

## 3. 我们自己的提示词：v2 与 v3（净室原创）

在两口公开井的基础上，写了**完全原创**的两版默认 harness 提示词（`src/engine/prompts.ts`）：

- **v2**：v1（极简）之上补真实行为纪律——先规划、先取上下文、并行只读查询、改后回读确认、
  基于工具输出说话、干净停机。**因更能干而变大，非灌水。**
- **v3**：v2 之上再补公开最佳实践对比里标记为「缺失/部分」的四条：
  ① **改后按成功标准自检**（re-read + 跑测试确认）；② **解决一般问题、不硬编码通过某个检查**；
  ③ **何时委派子代理**（大或独立才派，快速查直接做）；④ 收尾摘要给**一个具体范例**。

### 测量纪律（关键，别被「看起来更好」骗）

- **v2 在简单基准上无可测收益** → **正确地扣住没提拔**。看起来更周全 ≠ 数字更好，不测不认。
- 为让提示词质量差异**能被测出来**，加了两个**会真失败的硬任务**（`tests/integration/ab-benchmark.mjs` id 10/11）：
  - **id 10**：修 `calc.mjs` 真 bug（sum 从 1 起），`verify(dir)` **动态 import 跑真代码**，
    测 `total([1,2,3,4])===10 && total([])===0 && total([5,5])===10 && total([10])===10`——
    边界用例专门**反硬编码**（只改成 `return 10` 会挂在 `total([])`）。
  - **id 11**：写 `isPrime(n)`，`verify` 跑 10 个用例含 0/1/边界——**硬编码 `return n===17` 必挂**。
  - 这两任务的判定从「宽松文本 check」升级为「**执行产物代码**」，且 verify 在清理临时目录**之前**跑。

**当前状态**：v3 代码 + 硬任务 + verify 接线**已落盘、702 单测全绿、typecheck/build 干净**【已测：本地单测层】；
v1-vs-v3 的**真 API 探针待 dispatch**【已设计待测】。提拔与否，**等探针数字说话**，不提前宣布胜负。

小学生比喻：光看两份作业「字更工整」不算数——得把两个学生的解法都拿去做**新的、更难的、防抄答案的测验卷**，
谁的解法在没见过的题上也对，谁才真更强。

## 4. 组织级缓存红利：设计留 host，SDK 只长 seam

诊断（§2）说明「一个人多轮」的缓存收益有限。真正的跃升在**组织级共享**——这正是 BPT 内网 SVN 部署的场景。

- **多层注入设计**（`Public-Info-Pool/Resource/proposal/bpt-prompt-layering-design-20260704.md`，host 侧）：
  栈按「**稳定 = 共享 = 信任**」同序叠——L0 硬编码核心（按 agent 选基座）→ L1 团队 team-root →
  L2 个人 user-root → L3 项目 AGENTS.md（**最低信任、关小黑屋**）→ 易变尾 cwd（永不缓存）。
  两条正交语义：**策略向下只收紧、偏好向下就近覆盖**——与 §1.1-HC 防火墙同构（安全单向、只加不减）。
- **缓存红利**：团队/个人层叠起来是**真实内容**，自然撑过 2048 门槛；且 **team 层全队一份，一人写进缓存、全队白读**——
  收益从「个人多轮」跃升「组织级共享」。**这是「因更能干而变大、非灌水」的正解**：大的是团队真需要的规矩。
- **守密人裁定的边界**：分层的「发现/信任/顺序」脑子**不进 SDK**——组织拓扑焊进引擎会毁掉通用性。
  SDK 只承接一个**通用 seam**：接受「已分段、带缓存标记」的 system 提示词，在段界放缓存断点。

小学生比喻：全班共用的班规只需老师在黑板写一次，全班都看这一份（团队缓存全队白读）；
每个同学的个人便签贴自己桌上（个人层）；从校外捡来的小纸条（项目层）默认不可信，单独放讲台角落、不许它推翻班规。

## 5. SDK seam（唯一进 SDK 的部分，【已测】）

为什么唯独这一小块必须在 SDK：**缓存断点只能在组装 Messages API 请求那层放，host 交出字符串后就够不着 wire 了。**

已落地 `options.systemPrompt` 新增 `segments` 形态（`src/types.ts` / `src/query.ts` / `src/engine/cache-control.ts`）：

```ts
query({ options: { systemPrompt: { type: 'segments', segments: [
  { text: coreForSelectedAgent, cache: true },   // L0
  { text: teamRootMd,           cache: true },    // L1 ← 全队共享缓存
  { text: userRootMd,           cache: true },    // L2
  { text: wrappedProjectMd,     cache: false },   // L3 低信任、不缓存
] } } })
```

- 引擎**逐字转发**这些块、尊重调用方 `cache_control`、自己不加不移（`cacheSystemBoundary: 'preserve'`）。
- 断点预算：预留 1 给 tool schemas → system 最多 3 个缓存段，超出**从最不共享端丢弃并 debug 告警**（无静默截断）。
- segments 路径下关消息级缓存，防总断点超 API 上限 4。
- host 传什么/什么顺序/哪段可缓存——**全 host 决定，SDK 一无所知组织拓扑**。SDK 仍是通用 drop-in 引擎。

小学生比喻：装订机（SDK）只负责在你交来的、已分好组的讲义之间夹分隔卡；哪几组讲义该夹、按什么顺序——你（host）说了算，装订机不问内容。

## 6. 对官方的黑盒对照（【已测】，只比可客观比的两轴）

- **正确性**：同模型同任务，check 通过率**打平 9/9**。
- **速度**：BPT 墙钟约 **2.8× 更快**——**结构性优势**（直驱 API、无 CLI 子进程冷启动），不是调参调出来的。
- **成本**：约**便宜 24%**。
- **质量轴不在此判**：agent 循环的「转数/决策」层受官方**专有系统提示词**支配，净室纪律禁止复制，
  故 BEHAVIORAL 保真度**结构性封顶**、永不逐比特对齐。追平只谈 SURFACE（表面契约约 90%+），
  BEHAVIORAL 天花板不可逾越——这是诚实的定位，不是遗憾。

小学生比喻：两辆车比油耗和速度可以客观计时；但「开起来的手感」取决于对方不公开的调校参数，
我们不偷看它的参数表，就不假装能复刻手感——只老实报能测的油耗和圈速。
