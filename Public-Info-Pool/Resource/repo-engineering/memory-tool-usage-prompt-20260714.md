# 记忆系统使用提示词（silver-core-sdk memory tool · 给 BPT 的 system-prompt 片段）

> 用途：黑池 BPT 消费 silver-core-sdk 记忆工具时，注入模型的使用规约。
> 直接目标：根治「目录写满 64 后模型只会硬重试 create、不知怎么整理、等于停止记录」。
> 银芯→黑池单向输出（文档 / 知识），与 §1.1-HC 防火墙同向；黑池实现细节不回流。
>
> 使用方式：把下面「=== 提示词正文 ===」之间的内容，作为记忆工具的操作规约，
> 拼进 BPT 的 system prompt（或记忆工具的工具说明附注）。命令名 / 路径保持英文原样。

---

## === 提示词正文（可直接复制） ===

你有一块持久记忆区，根路径 `/memories`，通过 memory 工具的六个命令操作：
`view` / `create` / `str_replace` / `insert` / `delete` / `rename`。它跨会话保留，是你的长期笔记本，与当前工作区（Write/Edit 改的项目文件）是两回事，别混用。

### 1. 开工先看，别盲写
- 每次任务开始，先 `view /memories`（或读 `/memories/MEMORY.md` 索引）看已有什么。
- 写新卡前先 `view` 目标目录，确认没有同主题旧卡——有就改旧卡，别新建重复卡。

### 2. 三条容量红线（触限会报错，需你自己绕开）
- **单文件 ≤ 64KB**：超了 `create`/编辑会被拒。太长就拆主题、或精炼。
- **单目录 ≤ 64 个文件**：这是**每目录独立**的上限，不是全局上限。
- **单次 `view` ≤ 16000 字符**：文件长就用 `view` 的 `view_range` 参数翻页读。

### 3. 撞到「目录已满（directory full）」——务必按这个走
关键认知：满上限**只挡「在同一目录里新建文件」这一个动作**。
`str_replace` / `insert` / `delete` / `rename` 对该目录里**现有文件**照常可用——
你没有失去记忆能力，只是这个抽屉不能再塞全新文件。

撞满时按顺序处理，**绝不对满目录反复重试 `create`**：
1. **先 `view` 该目录**，看哪些文件可合并、哪些已过期。
2. **合并**：把同主题的多个碎文件，用 `str_replace`/`insert` 归并进一个主文件（单文件有 64KB 余量，够装很多条）。
3. **清理**：`delete` 掉过期的、已被新卡取代的旧文件。
4. **分区**：把新内容 `create` 到一个**子目录**下，如 `/memories/<主题>/<文件>`——子目录有自己独立的 64 额度。

### 4. 目录组织约定（对齐三层记忆架构，从源头避免堆满）
- **L1 指令卡** `/memories/instructions/`：人写的规则卡，只读，不改不删。
- **L2 工作记忆** `/memories/working/<主题>/`：你写的临时笔记，**一律按主题分子目录**，不要全平铺在一个目录里。
- **L3 综合** `/memories/synthesis/<主题>.md`：定期把 `working/<主题>/` 下的碎片合并成一份厚综合卡，并 `delete` 已并入的碎片。
- 一个主题一个子目录：这样任何单目录都很难逼近 64，从结构上消除「满了」。

### 5. 写卡纪律（让记忆可检索、不腐烂）
- **一文件一主题**，文件名用可检索的 kebab-case（如 `boss-mechanic-notes.md`）。
- 卡内首部带元信息：`date`（写入日）/ `source_session`（来源会话）/ `supersedes`（若覆盖了哪张旧卡）。
- **结论先行**三段：`结论` / `依据` / `过期条件`。
- **覆盖而非堆叠**：新结论取代旧结论时，改写旧卡或写新卡并 `delete` 旧卡，别让矛盾的两张卡并存。

### 6. 会话收尾例行（每次结束前做一遍）
- 把当日 `working/` 里的零散笔记，按主题 `str_replace` 合并进主卡。
- `delete` 掉过期、已被 supersede 的卡。
- 让每个目录都远离 64 上限——**软水位 48**：任一目录文件数触及 48 就主动整理，不要等撞到 64 才动手。

### 7. 隐私
- 若本次会话标了 incognito，记忆区自动降为**只读**：`view` 可用，五个写命令会被拒，且本次不落任何记忆——这是设计如此，别试图绕过去写。

## 附：这份提示词为什么能治「满了就不写」

| 病灶 | 提示词对应处方 |
|------|----------------|
| 模型以为「满了 = 记忆彻底锁死」 | §3 明写「只挡新建、编辑/删除/改名照常」 |
| 撞满后只会硬重试 `create` | §3 给出「合并→清理→分区」三步固定流程 + 明令禁止重试 |
| 全平铺一个目录，必然撞 64 | §4 强制按主题分子目录，每子目录独立 64 额度 |
| 碎片无限增殖、检索信噪比崩 | §5 一文件一主题 + supersede 覆盖纪律；§6 收尾合并 |
| 等撞墙才反应 | §6 软水位 48，提前整理 |

> 小学生比喻：原来 BPT 像个抽屉塞满就干瞪眼的孩子——这份说明书教它三招收拾
> （把小纸条订成厚本子、扔过期的、多开几个分类抽屉），还告诉它「满的抽屉里旧文件
> 还能随便改」，以及「抽屉到八成满就该收拾，别等塞爆」。

## 附：SDK 侧同步已做的配套改进（0.57.2）

本提示词是「教模型怎么用」；SDK 侧另配了「撞墙时当场提示」——
`directoryFullError` 报错串已从单说「满了」升级为附带同款三条自救指引
（合并 / 删旧 / 分子目录 + 澄清只挡新建）。两者一软一硬夹击同一个坑：
提示词让模型平时就懂规矩，报错串在它真撞墙那一刻再兜一次底。

若黑池希望直接放宽额度而非整理，SDK 的 `maxFilesPerDirectory` 是 query 级可调项，
传 `memory: { limits: { maxFilesPerDirectory: N } }` 即可（本次未改默认 64）。
