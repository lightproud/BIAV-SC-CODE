## 二、洼地清单（按风险排序）

### P0-1 `projects/news/scripts/collect_arca_daily.py` — 0%（53 语句全空窗）

2026-07-10 方案 2 过渡桥：每日 CC 例程无人值守跑「采集 → 归档 → commit → push（带重试）」。
其核心卖点正是**响亮失败**（零条采集 / push 失败须非零退出），但这套失败语义目前零测试——
最需要可靠的报警器本身没验过。风险最高：它每天在无人环境自动执行。

建议：新增 `tests/test_collect_arca_daily_unit.py`，monkeypatch `fetch_arca_live` 与
`subprocess`，覆盖四条路径：正常采集归档、零条非零退出、push 失败重试与最终失败退出、
`--no-push` 旁路。预计 10-15 个用例即可从 0% 到 90%+。

比喻：家里新装了会自动打 119 的烟雾报警器，但从来没按过测试键——真着火那天才知道响不响。

### P0-2 `scripts/mcp_server.py` — 73%（31 行未覆盖）

未覆盖行全部是各 `kb_*` 工具的 `KBIndexMissing` 异常分支与遥测 `_log` 降级路径
（183-184 / 209-210 / 232-233 / 254-255 / 275-282 / 300-301 / 323-330 / 357-364）。
MCP 是艾瑞卡的**唯一动态平面**（§1.4 第 5 条），索引缺失时应返回结构化 error JSON
而非崩溃，这个契约现在没被钉住。

建议：在 `tests/test_mcp_server_unit.py` 补一组「索引缺失」参数化用例：临时目录无
`kb_index.json` 时逐一直调 7 个工具函数，断言返回 `{"error": ...}` JSON 且不抛异常。
成本极低（工具函数可直接调用，无需起服务端）。

比喻：电梯正常运行测过了，但「停电时应急灯亮不亮」从没拉过闸试一次。

### P1-1 `scripts/extract_aliases.py` — 15%（113 行未覆盖）

别名侧表的**生成期工作面**。读取侧 `silver_aliases.py`（97%）与三墙纪律有测试守护，
但生成器本身近乎裸奔——若生成逻辑坏掉产出畸形侧表，读取侧测试用的是固定 fixture，
拦不住。别名侧表直接喂 `kb_anchor` 合流腿，上游坏则三腿之一失灵。

建议：对 `extract_aliases.py` 的抽取 / 归并 / 写盘函数补单测（tmp fixture 输入 →
断言产出 schema 与三墙字段齐全），重点钉「未确认压权重」「出身牌必填」两条墙在生成端成立。

比喻：图书馆借书规则查得很严，但编书目卡片的那台打字机没人校对——卡片印错，借书规则再严也白搭。

### P1-2 知识库评测器矩阵 — 评判体系自身无回归

| 模块 | 覆盖率 | 主要空窗 |
|------|--------|----------|
| `kb_telemetry.py` | 58% | `--harvest` 零命中回流闭环（141-179）整段未测 |
| `build_kb_vectors.py` | 66% | CLI 主流程 / 采样落盘（123-171） |
| `kb_eval.py` | 67% | `main()` 记分卡打印（92-120） |
| `kb_semantic_ab.py` | 74% | 报告生成段（190-213） |
| `kb_golden_gen.py` | 77% | CLI 与写盘（136-149） |
| `kb_qual.py` | 77% | 报告段（158-188） |
| `kb_ab.py` | 82% | 报告段（140-163） |

核心评测逻辑各有回归测试（`test_kb_golden.py` / `test_kb_ab.py` 等），未覆盖的集中在
**CLI 入口与报告渲染**。风险中等：评测器报告路径坏掉时，守密人读到的记分卡可能是
半截或错版，而「评判知识库的尺子」自己坏了最难被发现（评判体系 #1↔#2 闭环依赖
`--harvest`，该段整段无测试）。

建议：每个评测器补一个端到端 smoke（tmp 索引 fixture → 调 `main()` / subprocess →
断言退出码 0 且报告含关键段落）；`kb_telemetry --harvest` 单独补闭环用例
（伪造零命中日志 → 断言难题候选正确回流）。约 7-10 个用例。

比喻：体检仪器都在给别人量血压，但仪器自己该一年一检——尺子不准，量谁都白量。

### P1-3 `scripts/refresh_claude_code_prompts.py` — 0%（73 语句全空窗）

每周定时 CI 无人值守跑（浅克隆上游 → 同步进 `Public-Info-Pool/Reference/`）。
同步与删除语义没测试：若上游布局变化触发误删或半同步，参照库会静默劣化。

建议：用 tmp 目录 + PATH 桩 git 补单测，钉住「同步幂等」「上游缺目录时不误删本地」两条。

### P2-1 wiki 前端层 — 零 JS 测试设施

14 个 Vue 组件 + `characters.ts` 数据桥，`package.json` 无任何测试脚本，唯一守护是
「VitePress 构建通过」。Python 侧已有 `test_generate_wiki_pages_unit.py` 钉桥产物
`characters.runtime.json` 的生成，但**消费端**（`characters.ts` 读哪些字段）与产物之间
无契约测试——生成端改字段名，构建照样绿，图鉴页运行时才空。

建议（二选一，推荐前者）：
- 轻方案：Python 契约测试——解析 `characters.ts` 中访问的字段名集合，断言均存在于
  runtime JSON 产物 schema（零新工具链，一个测试文件搞定）；
- 重方案：引入 vitest 对 `CharacterGrid` / `characters.ts` 做组件级测试（新增 JS
  测试工具链，维护态节拍下投入偏重，暂不推荐）。

比喻：厨房出菜（生成端）有质检，端菜的服务员（前端消费）没对过菜单——菜名改了，
端上桌才发现对不上号。

## 三、结构性建议

1. **覆盖率门槛 85 → 89**：当前实测 91%+，85 的门槛留了 6pp 空转余量——回归掉 5 个点
   CI 都不响。建议 P0/P1 补完后把 `--cov-fail-under` 提到 89（留 2pp 呼吸空间），
   门槛才起「拦回归」而非「摆设」作用。
   比喻：体重 60kg 的人把警戒线画在 66kg，胖 5 斤秤都不叫——线要画在贴身处才有用。
2. **变异测试扩员（可选，维护态可缓）**：`setup.cfg [mutmut]` 现钉 4 个核心模块。
   `archive_layout.py`（归档布局单一真相源，97%）与 `news_common.py`（95%）逻辑密度高、
   常量噪声低，符合现有入选标准，可作下一批候选。
3. **优先序**：P0-1（arca 日采）> P0-2（MCP 错误分支）> P1-2（评测器 smoke）>
   P1-1（别名生成器）> P1-3（周同步）> P2 择机。P0 两项合计预计 25 个用例内解决，
   即可消掉全仓最危险的两块无人值守空窗。
