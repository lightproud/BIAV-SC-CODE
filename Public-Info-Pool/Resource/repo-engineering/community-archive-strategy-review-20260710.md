# 社区信息归档：规划 / 实现 / 维护策略整体评估

> 类型：repo-engineering（评估报告，非决策记录）
> 作者：艾瑞卡会话 2026-07-10
> 数据基准：本日实测（沉默源审计 + 逐平台目录核对 + git 提交时间线）+ 权威档案对账
>（`projects/news/CONTEXT.md` / `memory/project-status.md` / `archive_layout.py` /
> `archive_engine.py` / `silent_sources_audit.py` / 维护态节拍表提案 20260702）
> 时点敏感：Phase 2 M3 稳定化期本日（07-10）到期，M4 验收期 07-11 起、07-19 收官。

## 一、总评

社区信息归档主线整体处于**「架构已成型、运行基本健康、收尾债集中在文档与验收」**状态。

- **规划层**：三层采集架构（T1 新闻流 / T2 数据层归档 / T3 维护回填）+ 声明式注册表 +
  归档布局单一真相源（SSOT）的设计是成熟的，且每一次架构升级都对应一次真实事故的复盘
  （lesson #30 数据层混淆 → §4.1 纪律；06-22 读写漂移 → 07-02 `archive_layout.py`）。
- **实现层**：21 源 16 活跃 / 1 降级 / 0 休眠 / 4 从未产出；全量档案 763 万条、约 2.0 G；
  自动化提交无断档。实现质量高于文档新鲜度——多处「待实施」标注实际已实施。
- **维护层**：**最大的结构性缺口**。维护态节拍表（20260702 提案）已把 07-19 后的值班方案、
  数据增长触发线、CI 门禁重启全部写好，但**尚未裁定生效**；验收清单四项中「贡献流程跑顺」
  仍未演练，是唯一硬缺口。

一句话：车已经造好、也在路上跑，但「谁定期检查、坏了谁修、修到什么标准」的值班表
还压在守密人桌上没签字。

## 二、规划层分析

### 强项

1. **三层职能切分清晰且被坚持**（2026-06-21 定性）：T1 每时新闻流 / T2 声明式归档
   （`archive_sources.json`，加源 = 加配置零代码）/ T3 回填维护。T2 绝不并入 T1 的
   错峰纪律在 workflow 层面得到执行（update-news :00 / discord 18:00 UTC / volunteer :15 / jp :45）。
2. **每条硬规则都有事故出身**：§4.1 全量 vs 输出层（lesson #30）、`archive_layout.py`
   SSOT（06-22 分层实施后 6 源被误判沉默 10+ 天）、`FOLDED_SOURCE_LAYOUT` 折叠映射
   （「叫什么」与「放哪里」分治）。规则不是凭空立的，复发风险有契约测试锁定
   （`tests/test_archive_layout.py` 读写往返）。
3. **增长有预案**：节拍表提案给出实测基线（档案层 2.01 G、增速约 205 MB/月）与三条触发线
   （工作树 ≥4 G / 单月 ≥400 MB×2 / clone ≥5 min），「何时重议」与「重议什么」解耦。

### 缺口

1. **2026-06-21 归档结构规范的实施尾巴未收**（CONTEXT.md「实施待办」六项）：
   - discord 三服布局仍不统一：Global 在 `discord/channels/` 根、志愿者与日服在
     `discord/guilds/{id}/`。且 `archive_layout.py` 明文把 discord 排除在 SSOT 之外
     （「独立归档器与目录语义，调用方自理」）——最大的源（800 万条）恰好游离在
     布局单一真相源之外。
   - `official` 并入 `steam/global/news`、`youtube_comments` 并入 `youtube/*/comments`：
     写入侧折叠已生效，但源名与健康报表仍按旧名独立呈现，语义双轨。
2. **文档与实况漂移**：CONTEXT.md（最后更新 06-28）仍标注「区服/类型分层待实施」，
   实际 06-22 已实施、07-02 已完成 1,395 个历史文件归位；M1/M2 任务清单多项状态过期
   （如「日服 Discord 待填 JP_GUILD_ID」——实测日服今日已正常落档）。
3. **审计粒度盲区（本日实测证实）**：`silent_sources_audit.py` 只到平台粒度，区服子层
   断档不可见——`appstore/jp` 连续日更 44 天后自 06-10 骤停 30 天、`steam/jp/news`
   停在 06-25、`youtube/jp/video` 停在 06-26，平台级报表却显示 appstore/steam/youtube
   全部「活跃」。这正是 06-22 事故的同构复发（当时是「平台级看不见分层」，现在是
   「平台级看不见区服」），病根同一个：健康监控粒度落后于布局粒度。

## 三、实现层分析（2026-07-10 实测）

### 健康面

- 21 源：16 活跃 / 1 降级（official，沉默 12 天）/ 0 休眠 / 4 从未产出
  （bahamut / arca_live / note_com / twitter）。
- Discord 三服全部在档：主服 570 频道 14,899 JSONL（每日 18:00 UTC 节拍，昨日正常）；
  日服今日 10:05 UTC 落 07-10 档（+155 条）；志愿服零新增仅刷状态。
- 今日（07-10）已落档：bilibili / pixiv / reddit / ruliweb / stopgame / taptap /
  weibo / steam（review+discussion）/ youtube（global）。
- 全量分析索引 `community_index.json`：7,630,770 条，生成于 07-06（周更节拍内）。

### 异常与隐患

1. **`official` 源（=Steam 官方公告）沉默 12 天**：last 06-28。需分辨「官方真没发」
   vs「抓取断了」。忘却前夜若处于版本间歇期属正常，但降级态不应无人认领。
2. **JP 区服疑似断裂**（见规划层缺口 3）：`appstore/jp` 的形态（44 天连续日更后骤停）
   更像采集端断裂而非评论枯竭，建议列为 M4 前排查项。
3. **「零新增不写档」的语义模糊**：采集成功但无新数据、与采集根本没跑，落盘结果相同
   （都没有当日文件）。平台级有 tracker 的 `last_check_date` 兜底，区服级没有——
   这是盲区 3 的机制根源。
4. **零产出 4 源挂账**：注册在案、窗口内 0 条。节拍表提案已建议 07-19 前逐一
   「修复或明文退役」，尚未执行（arca_live 已知 Playwright 选择器超时、疑站点改版）。
5. **数据卫生**：weixin 桶按文章发布日期落档，存在 2016–2020 年老文（最早 2016-02-03），
   把审计窗口拉长到 3,811 天并稀释覆盖率统计；ruliweb「内容日期偏旧致归档桶不新」
   为同病根边界情况（07-02 已定性观察即可）。建议给「内容日期 vs 采集日期」二义性
   做一次明文定性，避免未来分析误读。
6. **youtube_comments 存在单日缺档**（2026-06-02），属 07-02「写旧读新」修复的历史遗留，
   可用 T3 回填工具补齐或明文放弃。

## 四、维护策略层分析

### 现状

- **全自动档**运转正常：T1 每时 / T2 每日错峰 / 索引每周（`build-analysis-index.yml`）/
  机器提交带 `[skip ci]` 防循环。
- **人工档无正式节拍**：health 检查靠会话即兴触发（如本次核查）；节拍表提案设计了
  周检（15 分钟：source-health + 机器提交节奏）与月检（增长对账 + pytest + 漂移抽查），
  **待裁定**。
- **CI 硬门禁仍撤防**：2026-06-21 撤 required `test` 的两条理由（checkout 慢 / 卡
  "Expected"）已随 07-02 sparse checkout 改造消失，重启提案待裁定。
- **验收缺口唯一项**：贡献流程从未真实走完一轮「提交 → 校验 → 合并」。

### 结构性评估

维护策略的问题不是「没想过」，而是「想好了没落地」：节拍表、触发线、门禁重启、
僵尸源处置四件事全部已成文于 20260702 提案，全部停在「待裁定」。Phase 2 还剩 9 天，
这批裁定若不在 M4 内完成，07-19 后系统将以「无值班表的值班状态」运行——
自动化会继续跑，但降级源无人认领的时长会从「天」漂移到「月」（appstore/jp 已示范：
30 天无人发现）。

## 五、建议（按优先级）

### P0 —— M4 验收期内（07-11 → 07-19）必办

1. **裁定维护态节拍表提案**（含周检/月检节拍、增长触发线、CI required `test` +
   up-to-date 门禁重启）。这是「维护策略」问题的现成答案，只差签字。
   采纳后按提案自述：节拍摘要进 `memory/methodology.md` 或 CLAUDE.md §7，
   触发线进 `memory/decisions.md`。
2. **贡献流程验收演练**：守密人（或指定协作者）以 C3 身份提交一条真实角色数据修正，
   走完模板全程。四项验收标准里唯一没排练过的环节。
3. **排查两处活异常并认领**：official 12 天沉默（分辨真沉默 vs 抓取断）+
   appstore/jp 30 天骤停（44 天日更后急停，采集端断裂嫌疑大）。

### P1 —— Phase 2 收官前后（07 月内）

4. **健康审计下钻到区服粒度**：`silent_sources_audit.py` 已 import `archive_layout`，
   对分层平台按 `<平台>/<区服>[/<类型>]` 逐叶报 last-seen 即可堵住盲区；同时给
   稀疏区服（jp 系）配独立沉默阈值，避免「评论本来就少」刷屏假警报。
5. **零产出 4 源逐一裁定**：修复（arca_live 选择器）或明文退役（移出 KNOWN_SOURCES +
   注册表除名 + CONTEXT 记录理由），不留僵尸。
6. **CONTEXT.md 对账刷新**：M1/M2 清单勾销已完成项、删除「待实施」的已实施标注、
   日服接入两步指引改为「已启用」实况。文档漂移是本仓库反复付费的老病，
   收口成本远低于误导成本。
7. **discord 布局收编或明文豁免**：要么统一三服到 `discord/{global,jp,volunteer}/`
   （规范原文），要么在 `archive_layout.py` 里把现状（Global 在根、其余在 guilds/）
   登记为规范形态并给出遍历函数——二选一，但不能继续「最大源游离于 SSOT 之外 +
   规范说要统一却无人排期」的悬置状态。

### P2 —— 维护态常态化（07-19 后）

8. **周检自动化减负**：把周检档的机械部分（source-health 摘要 + 降级/新增异常 +
   机器提交节奏）做成每周一次的 digest（scheduled workflow 产 md 或会话 Routine），
   人工只看摘要、异常才下钻——把 15 分钟压到 3 分钟，提高实际执行率。
9. **「零新增 vs 没跑」语义补强**：采集器每轮对每个启用区服记 attempt 时间戳
   （扩展现有 tracker 的 last_check_date 到区服级），沉默判定改为
   「有尝试无数据 = 真沉默；无尝试 = 采集断」。
10. **内容日期 vs 采集日期定性**：对 weixin 类按内容日期落桶的源，在归档规范里
    明文其审计口径（覆盖率统计应从游戏公开期起算，不含老文长尾），或在审计器里
    加窗口下限。

## 六、结论

规划的方向感与实现的自愈记录都是健康的——每次事故都变成了规则、测试或 SSOT 模块，
这是这条线最值得保持的品质。当前的风险不在架构，在**收尾纪律与值班空窗**：
六项规范待办、四项待裁定提案、一项验收演练，全部有文可依、无一需要新设计。
M4 的 9 天恰好是把它们清零的窗口。
