# 三、战斗与数值体系

> 数据来源：AwakerPotency.lua（1,035 条）、Stage.lua（5,709 条）、Item.lua（3,774 条）、GameConfig_References.lua（34,333 条）、numeric_config.txt（66,517 行）、design-decisions.json

## 3.1 战斗定位

Roguelite 卡牌构筑回合制，2D 横版。制作人明确提到的游戏影响源：

- **Slay the Spire**：核心肉鸽卡牌玩法起点
- **Darkest Dungeon**：压力系统张力（"多数崩溃，少数逆转"的感受比例参考）
- **Chaos Zero Nightmare（卡厄斯梦境）**：探索中卡牌变化（战斗设计师米酒深受启发）
- **Curse of the Dead Gods**：克苏鲁题材疯狂量表参考

## 3.2 AwakerPotency 数值结构

AwakerPotency.lua 的 1,035 条记录是角色技能潜力的完整配置。条目分两类：

**"人格深化"条目**：大量条目名称为"人格深化"且无描述——这些是角色天赋树中的纯数值节点（属性加成），不含文本描述。

**技能描述条目**：包含 `PotencyDesc` 字段的条目，提供具体的技能效果文本。采样：

| 技能名 | 效果描述 | 分析 |
|--------|---------|------|
| 模糊的梦想 | 「整装待发」造成的固定伤害提高 50%，释放后自身免疫脆弱、虚弱、易伤 1 回合 | 百分比增伤 + 状态免疫，攻防兼备型天赋 |
| 笑呵呵玩乐 | 「基础防御」抽 1 张牌。每 3 回合，将 1 张附加虚无和消耗的「腺体分裂」加入手中 | 防御卡附带抽牌，定时生成特殊卡 |
| 逃离死亡 | 「等价交换」基础回复和弃牌回复效果提高 20%，每弃 1 张牌额外回复生命并获得等量护盾 | 弃牌转化为生存资源 |
| 无限通道 | 「基础打击」享受 2 倍力量加成+跃迁获 1 AP。「基础防御」获得攻击力 [Arg1]% 的力量+跃迁获 1 AP | 基础卡强化 + AP 回收 |
| 倔强意志 | 使 1 条触腕攻击 5 次，获得 25% 造成伤害的反击 | 深海触腕多段攻击 + 反击联动 |

**关键发现**：`[Arg1]` 占位符表明数值通过运行时参数注入，Lua 表中存储的是公式模板而非固定值。这意味着 numeric_config.txt（66,517 行）中的数值配置是对应的参数源。

## 3.3 关卡数据规模

Stage.lua 的 5,709 条关卡配置是游戏内容量的直接度量。配合 StageGroup.lua 的分组数据和 Task.lua 的 6,317 条任务数据，可以推断：

- 游戏实际关卡深度远超表面章节数（第一部 8 章 + 第二部进行中 4 章）
- 每个主线章节内含大量子关卡、分支路径、随机事件节点
- 活动关卡、融灾禁区、幻梦深潜等附加模式各自贡献大量关卡条目

Item.lua 的 3,774 条道具数据揭示了游戏经济系统的复杂度。从采样数据看，道具类型包括：

| 类型 | 示例 | 说明 |
|------|------|------|
| 密契（圣遗物） | 环形诗篇V、扭曲双子·白VI、无垢启示录I、有机形态II | 按套装分组，主属性随机 |
| 密契属性池 | 狂气回充、死亡抵抗、银钥充能、伤害强效、界域精通、黑印掉落、暴击率、暴击伤害 | 8 种主属性 |
| 命轮（武器） | 星天之兽（萝坦） | 角色专属或通用 |
| 钥令 | 注射守护 | 守密人技能，编队时选择 |
| 升级材料 | 陨星碎块 | 角色培养素材 |

## 3.5 被砍机制：疯狂量表

design-decisions.json 记录了一个被砍的核心机制——**疯狂量表（Madness Scale）**：

**原始设计**：探索选择会影响疯狂量表，到达最终阶段时获得危险但强力的能力。

**被砍原因**（4 条）：
1. 与现有系统叠加后认知负担过高
2. 关卡时长变化太大，量表粒度难以把控
3. 负面反馈已经够多了
4. 角色身份不匹配——守密人是"选择面对不可名状之物"的人，不应频繁被击溃

**简化结果**：症状卡（Symptom Card）——负面卡牌占据手牌空间，比量表更轻量，且可被特定角色技能弃掉（如艾尔瓦的狂气爆发）。

## 3.6 GameConfig 引用体系

GameConfig_References.lua（34,333 条，1.26 MB）是全游戏配置的索引骨架，使用 `Category@Detail` 格式引用。这意味着游戏的数值系统是高度参数化的——技能效果、关卡难度、平衡参数均通过配置表驱动而非硬编码。

这种架构使得：
- 数值调整无需改代码，热更新即可生效
- 同一技能模板可通过不同参数实例化为不同角色的技能
- 与 AwakerPotency 中的 `[Arg1]` 占位符对应
