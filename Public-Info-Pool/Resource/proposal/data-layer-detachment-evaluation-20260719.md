# 数据层剥离评估方案（立项）

> **立项依据**：守密人 2026-07-19 裁定「丙 · 立项重估形态」——回应 KIMI3 静态审查头号 P0
> 「把社区数据档案剥出 git 主历史/工作树，主仓回到『代码 + 指针 + 小样本』形态」。
> **性质**：评估战线立项 charter，非迁移执行令。产出目标形态方案 + 分阶段决策门 + 风险矩阵，
> 每个不可逆动作前留守密人裁定门。
> **前置约束**：本方案不触任何不可逆动作；历史重写属 T29 禁区，须在本战线 P0 盘点通过 + 守密人裁定后方可议。

---

## 1. 背景与触发

KIMI3 静态审查判「主仓同时承担源码 + 事实档案 + 社区数据 + AI 指令 + 发布流水线」为最大战略风险，
建议把数据档案迁出 git 历史、主仓只留 manifest/checksum/schema/样例。该诊断方向正确，但直接撞本仓
两条既有约束：

- **2026-06-21 de-tier 决策**（§5.2）：discord 全量档案**刻意永驻 git**（退役月度 git_rm 瘦身），
  为读方直读、单向送黑池、零还原成本；
- **T29 历史重写禁区站岗**：2026-06-21 瘦身删的 ~3.0G 旧档，原判「30 月仅实证 3 月在 Releases 有副本」，
  历史 blob 被视为唯一二次抢救网。

守密人裁定丙：正式开评估战线厘清，而非照单执行或简单驳回。

## 2. 现状事实基线（2026-07-19 实测）

| 指标 | 实测值 | 说明 |
|------|--------|------|
| 工作树（含 .git）| 1.1 GB | — |
| git pack（.git）| 417 MB | 普通 clone 成本主体 |
| Public-Info-Pool 总 | 663 MB | 其中 Record 652 MB |
| ├ discord 档案 | 626 MB | Record 主体 |
| └ Resource（产物）| 6.5 MB | 报告/分析，本就该留 git |
| `.gz` 冷层文件数 | 18,392 | 2026-07-12 甲案冷热分层产物 |

**既有缓解（非「未设防」）**：
1. **冷热分层**（2026-07-12 甲案）：上上月起压 `.gz`，实测压至 ~18%；
2. **sparse checkout**（2026-07-02）：CI 从 2.6G 全 checkout 降至 ~100MB；
3. **三条数据增长触发线**：工作树 ≥4G / 单月增速 ≥400MB 连 2 月 / 全新 clone ≥5min——任一触发才收缩。

> 比喻：仓库不是「堆满杂物没人管的房间」，而是「装了压缩袋 + 只翻当月抽屉 + 设了体重秤报警」的储物间。
> KIMI3 说「东西太多」没错，但没看见已装的这三样。当前工作树 1.1G，距 4G 触发线仍有余量。

**射程外硬约束（不可动）**：§1.1-HC 黑池防火墙——数据层无论落 git 还是落对象存储，
**方向性单向输出**不变；本评估只动「银芯自有公开档案的存储形态」，绝不为黑池回流开口。

## 3. 关键发现 · blob 抢救盘点（T29 前置，本战线 P0 首件）

查 `community-data` Release（2026-07-01 发布，**晚于** 2026-06-21 瘦身）资产清单：

- **33 个月 discord 归档副本**：`discord-archive-YYYY-MM.tar.gz`，覆盖 **2023-07 → 2026-05** 连续
  （仅缺 **2023-08 / 2023-09** 两月）；
- 压缩总量 285.5 MB。

**这实质推翻 T29 的悲观假设**：抢救网不是「30 月仅 3 月」，而是「~33 月近乎全覆盖」。历史重写的
头号不可逆风险被大幅压低。

> 比喻：原担心搬家会把只存了 3 个月的老照片弄丢，翻云相册发现 33 个月几乎都在——搬家风险从「灾难级」
> 降到「需核对清单级」。

**但须诚实标注三项未决核验**（§4.2 R3：盘点 ≠ 内容完整）：
1. **区服/平台覆盖面**：资产名为 `discord-archive-*`，尚未确认是否含 jp / volunteer 区服 +
   16+ 其他平台（可能仅 discord 主服）；
2. **内容完整度**：每月 tar.gz 是否含全频道全量，还是抽样——未下载校验；
3. **2023-08/09 缺月**：确属无副本，还是彼时无数据——待查。

这三项 = 本战线 **P0 盘点子任务**，通过前不得进任何历史动作。

## 4. 目标形态（KIMI3 「代码 + 指针 + 小样本」）

主仓收敛为三类留存：
- **代码 / CI / 脚本 / 配置**：原样留 git；
- **指针层**：每月档案留 `manifest`（文件清单 + checksum + schema + 行数/条数统计）+ Release/对象存储 URL；
- **小样本**：每来源留 N 条样例供测试/schema 校验（现 `tests/` 已有此模式）。

全量本体迁 Release 或对象存储，读方经指针按需取（呼应 RELEASES.md「藏宝图」+「只指针不复刻」哲学）。

## 5. 选型维度（P1 待评估，不在本 charter 定论）

| 方案 | 优点 | 代价 | 与本仓契合度 |
|------|------|------|--------------|
| **维持 Release 桶** | 零新运维、已在用、免费 | 单文件 2GB 上限、无版本化、云容器只读不可写 release | 高（现成） |
| **对象存储**（S3/R2 等）| 无大小限、可版本化、可设保留期 | 新增凭据面 + 账单 + 运维 runbook | 中（违「零运行时依赖」倾向） |
| **git-LFS / annex** | 仍在 git 语义内、指针原生 | LFS 配额/带宽成本、clone 仍需 smudge、迁移复杂 | 低（换汤不换药） |

> 比喻：老照片是继续塞进「免费但每本限厚的相册」（Release）、租「按量付费的大仓库」（对象存储）、
> 还是「贴张取件条但取件仍要排队」（LFS）——各有账要算，P1 专门比。

## 6. 不可逆风险矩阵

| 动作 | 可逆性 | 风险 | 缓释 |
|------|--------|------|------|
| 冻结新增大二进制入 git（乙案增量掐口） | **可逆** | 低 | 采集流水线写档路径改为落 Release 指针；随时可回退 |
| 建 manifest 指针层 | **可逆** | 极低 | 纯新增，不删原档 |
| **git 历史重写**（剥历史 blob） | **不可逆** | **高** | 须 P0 盘点全绿 + 全新 clone 各方确认 + 守密人裁定；force-push 前全量 mirror 备份 |
| 迁新仓 | 半可逆 | 中 | 旧仓保留只读归档 |

**红线**：历史重写与迁新仓为独立决策门，绝不随本 charter 自动执行；§1.1-HC 防火墙方向性在任何形态下不变。

## 7. 分阶段路线（每阶段末为守密人决策门）

- **P0 · 盘点核验**（本战线首件，可立即做）：核 §3 三项未决（区服/平台覆盖、内容完整度、缺月）；
  产「抢救网完整度报告」。**门①**：抢救网是否足够完整到支持后续动作？
- **P1 · 目标形态定选型**：§5 三方案比选 + 采集流水线改造面评估（archive_layout 写档路径）；产选型决策稿。
  **门②**：选哪套存储 + 是否先行乙案（冻结增量）？
- **P2 · 增量掐口**（乙案，若门②批）：新增大二进制强制走 Release 指针化，主仓新增只留 manifest；
  **不碰历史**。这是「不撞不可逆、只掐未来」的安全折中，可长期停在此态。
- **P3 · 历史处置**（可选，最高风险）：仅当门①②全绿 + 守密人显式裁定，才议历史重写/迁新仓。
  **门③**：是否值得为 clone 成本承担不可逆历史动作？（当前 pack 417M，触发线未破，倾向「暂不」）
