## 一、这是什么,为何重要

本文档定义黑池的下一个产品形态:**Cowork——面向非技术职能的「对话 + 交付物」工作界面**。它回答三个问题:用户用起来和 Code 类工具有什么区别;它解决黑池当前的什么问题;技术上如何在 silver-core-sdk 之上实现。

为何重要:黑池已有 20+ 名日常用户,主力是**策划、市场、运营、文案**——他们的工作产物是文档、表格、分析和结论,不是代码。当前界面继承自面向开发场景的交互假设(同步对话、原始报错、结果自行验证),这批用户实际是在「用别人的工具凑合」。要把用户规模推向 60+,增量几乎全部来自同类非技术职能;**决定黑池能不能长大的,是 Cowork 形态做得好不好,而不是再多一个技术功能。**

> 一句话定位:Code 的中心是代码库,Cowork 的中心是交付物。整个设计由这一个差别推导而来。

### 作者盲区与本文效力

本文写于银芯侧。**受 §1.1-HC 防火墙约束,作者看不到黑池任何代码与内部架构**——因此凡涉及「黑池要写什么、怎么写」的内容(第四、六节),本质是基于 silver-core-sdk 公开能力的**建议性推测**,以黑池实际代码为准,预期与现实有偏差。**本文的规范性部分是第五节「界面需求规格」**:用户用起来应该是什么样,这部分不依赖对黑池内部的了解,是可以直接对照开工与验收的内容。

### 本文结构

二·用户视角(两种走法与四条差异)/ 三·问题清单(五个痛点映射)/ 四·架构假设(三层责任与数据流,推测)/ **五·界面需求规格(规范性部分)** / 六·实现映射(四能力 × SDK,建议)/ 七·实施路线 / 八·边界与风险

## 二、用户视角:同一件事的两种走法

用一个真实场景对比。任务:**「整理沙耶之歌联动的社区反馈,输出一份复盘报告。」**

### Code 形态里,这件事怎么走

用户打开一个挂在某个仓库目录上的会话。工具默认你有一个 cwd,上下文从仓库文件自然获得——但这个任务没有仓库,于是用户先要想办法把社区数据「弄进来」:找路径、贴文本、或者求助技术同事。AI 跑起来之后满屏滚动工具调用日志,中途断网一次,回来发现会话状态不明,只好重来。最后 AI 说「报告已生成到 output/report.docx」,用户要自己找到这个文件、下载、打开,才发现分类口径不对——于是整轮重来。全程用户守在屏幕前,像盯一个不能离手的锅。

### Cowork 形态里,同一件事怎么走

用户新建一个任务,系统识别这是运营类工作,**自动接好连接**:运营知识库、社区数据源、相关钉钉文档目录,文件抽屉里明明白白列着「本任务连着什么」。用户一句话说完需求,AI 先返回一张**计划卡**:三步,汇总四语反馈、按好评差评建议分类、生成 docx。用户扫一眼,发现想按语言分而不是按情感分,点「修改计划」改一句话,再点确认。然后**关掉手机去开会**。四十分钟后钉钉弹出通知:任务完成。点进收件箱,报告卡片上有缩略预览,正文里的每个关键结论都挂着可点的来源角标。预览没问题,下载,转发给团队。

> 差别不在界面好不好看,在于整个交互契约变了:从「人守着 AI 跑」变成「人给 AI 派活,回头取件」。

### 四条本质差异

| 维度 | Code 形态 | Cowork 形态 |
|---|---|---|
| 上下文锚点 | checkout 的仓库,cwd 天然存在,上下文从文件系统自然获得 | 没有 cwd。锚点是「任务 + 它的连接」:SVN 工作副本、外部文档、知识库,由系统装配后供给会话 |
| 验收方式 | 用户看 diff、跑测试,自己有能力判断对错;工具可以把信任交给用户 | 用户没有验证手段,验收必须由产品代劳:预览、来源引用、执行前计划确认 |
| 交互节奏 | 高频短循环(改-跑-看),人机同步,断线是事故 | 低频长任务(提交-后台跑-取件),人机异步,断线是常态且必须无感 |
| 错误面 | stack trace 是有效信息,权限询问是合理交互 | 报错必须翻译成人话并自动降级;权限不询问,用安全默认消灭问题 |

注意这四条没有一条是「界面长什么样」。界面本身仍是对话框加文件区,黑池已经有了;**差异全部发生在界面背后的产品层。**界面需求详见第五节;实现映射(第六节)几乎不谈前端,原因在此。

## 三、解决什么问题:黑池现状五个痛点

以下痛点来自黑池当前用户构成(策划/市场/运营/文案为主力)与现行引擎版本(silver-core-sdk pin 在 0.3x)的组合。每个痛点都对应一个 Cowork 能力,不存在「顺手做的功能」。

| 现状痛点 | 后果 | 对应能力 |
|---|---|---|
| **每个任务都要重新交代背景**——AI 不知道该连哪个 SVN 目录、哪个文档、哪个知识库 | 非技术用户不会「喂上下文」,要么答非所问,要么放弃使用 | 能力一 · 连接装配(职能预设) |
| **产物是个下载链接**——打开前不知道对不对 | 返工率高;用户对 AI 产物默认不信任,重要工作不敢交 | 能力二 · 产物预览 + 来源引用 |
| **AI 一上来就干**——方向错了要整轮重来 | 长任务试错成本极高,用户学会「只交小事」 | 能力二 · 计划确认 |
| **断线丢会话、长任务必须盯着**——0.3x 引擎没有韧性兜底 | 手机场景基本不可用;任务时长被人的耐心封顶 | 能力三 · 异步化(依赖引擎升 0.51) |
| **报错直接甩 stack trace,权限弹窗看不懂** | 非技术用户遇错即弃,并扩散「这东西不行」的印象 | 能力四 · 人话错误面 + 安全默认权限 |

值得强调第四条:断线韧性四层、记忆系统、会话续接这些修复**已经存在于 silver-core-sdk 0.43–0.51**,黑池的痛只是因为 pin 在旧版。这意味着路线第一步(升 pin)是零新代码、高回报的动作。

## 四、架构假设:三层责任与数据流(推测性内容)

总原则:**SDK 提供会话级原语,黑池实现产品层,前端只做呈现。**silver-core-sdk 不需要为 Cowork 改一行——它已经提供 mounts 与双模装配、记忆 M1/M2、kb_* 十一个知识工具(OKF 厚锚)、断线韧性四层、会话续接配方、TaskStop/TaskOutput。黑池要新建的全部是围绕这些原语的产品设施。

| 层 | 职责 | 本方案新增 |
|---|---|---|
| 前端呈现层 | 对话框 + 文件抽屉;渲染消息流中的结构化卡片(计划卡/交付卡/进度卡);任务收件箱页 | 五种卡片组件、收件箱页、文件抽屉 |
| 黑池产品层 | workspace 生命周期、装配预设、任务队列与状态机、产物渲染管线、引用透传、事件→进度翻译、钉钉通知、错误文案映射、权限策略 | 本文第六节全部内容 |
| SDK 原语层 | 会话执行、上下文装配、记忆、知识检索、韧性、续接、工具协议 | 无新增;仅要求 pin ≥ 0.51 |

### 一次任务的完整数据流

① 用户创建任务 → 产品层按职能预设生成**连接清单**:SVN 路径在服务端 checkout 任务工作副本,钉钉文档/表格登记连接器引用,kb 照旧;② 产品层以清单调 SDK 建会话(副本与连接器封装为 mounts 与工具,memory 注入用户偏好);③ 会话产出 plan → 落库、TaskStop 暂停 → 前端渲染计划卡;④ 用户确认 → 续接配方恢复执行;⑤ 执行事件翻译成进度步骤;⑥ 产出物进入**变更集**:新文件走渲染管线出预览,SVN 修改生成 diff 卡,外部文档编辑生成变更摘要,kb 结论挂厚锚引用;⑦ 写回类变更(SVN 提交 / 钉钉文档更新)默认停在待确认,确认后落地;⑧ 状态机置 delivered → 钉钉通知 → 收件箱取件。全程客户端可离线,会话状态与工作副本都活在服务端。

## 五、界面需求规格(本文的规范性部分)

本节以「可开工」为粒度定义用户侧的页面、卡片与交互规则。页面共三个:**任务收件箱**(主入口)、**任务详情页**(对话流 + 卡片 + 文件抽屉)、**创建任务流程**;卡片共五种:**计划卡、进度卡、变更 diff 卡、交付卡、错误卡**。

### 5.1 创建任务

入口在收件箱右上。必填只有两项:**职能**(默认记住上次选择)和**一句话需求**;展开高级区可见按职能预填的连接清单,允许增删。提交即离开——创建动作不承诺任何等待,用户下一次需要出现是计划卡就绪时(钉钉通知)。需求输入支持语音(与 bpt-v2t 方向汇合)。

### 5.2 计划卡:步骤怎么分

步骤划分是产品约定(注入会话的规则),也是用户的心智模型,四条规则:

| 规则 | 内容 |
|---|---|
| 按产物分步 | 每一步必须产出一个**名词**——一份清单、一张表、一节文稿。禁止按内部工具调用分步(「调用检索」不是步骤,「反馈汇总清单」才是) |
| 数量与时长 | 3–7 步;超过 7 步须归并;单步预估超过 20 分钟须再拆。预估总时长低于 5 分钟的短任务**不出计划卡直接执行**(阈值可配) |
| 每步四要素 | 标题(动宾)、输入(用到哪些连接)、产出(名词)、预估时长。四要素在卡片上逐条可展开查看 |
| 写回收尾 | 写回类动作(SVN 提交、文档更新)单独成步且必须放在末段——所有对外部系统的改动集中在变更确认之后,执行中段永不写回 |

卡片交互:底部三键**确认执行 / 修改计划 / 放弃**。修改 = 自然语言输入意见,返回修订版计划卡并**高亮被改动的步骤**;放弃则任务关闭、连接清单保留可复用。

### 5.3 执行中的检查:步骤产出抽查

这是「怎么检查」的核心机制——验收不是只发生在最后:

| 机制 | 规则 |
|---|---|
| 步骤产出快照 | 每步完成即固化一份**中间产物快照**,进度卡上该步骤变为可点,展开即见(清单、表格片段、文稿节选)。快照只读 |
| 步骤边界暂停 | 用户任何时刻可点「在下一步前暂停」——会话在**步骤边界**停下(不打断步骤中段),用户抽查后选择继续、或改计划从第 N 步重跑 |
| 修正路径唯一 | 发现中间产物有问题,修正只有一条路:改计划、指定从第 N 步重跑。**不提供就地编辑中间产物**——避免人机写冲突,也保证产物与步骤的对应关系永远可信 |
| 进度即步骤 | 进度条就是步骤列表:完成步打勾并显示实际耗时,当前步高亮。无法归类的执行活动一律归入当前步,显示「整理中」,**宁可粗,不编造细节** |

### 5.4 变更 diff 卡:写回怎么审

头部显示目标系统(SVN 路径 / 钉钉文档名)与变更统计(几个文件、几行、几个单元格)。主体按目标类型渲染:

| 类型 | 呈现规则 |
|---|---|
| 配置表(SVN) | 按表分组;只显示变更行及上下各一行上下文;单元格级高亮,旧值灰色删除线、新值金色;超过 50 变更行折叠分页,头部保留统计 |
| 文本(SVN) | 行级 diff,惯例呈现 |
| 钉钉文档 | 修改前后段落左右对照 |

操作三选一:**确认落地**(= commit / API 写入,message 自动带任务号,卡片固化显示 commit 号或文档版本,附回滚入口)、**整体放弃**、**退回修改**(附意见,重新执行写回前的相关步骤)。**v1 刻意不做逐行挑选:全收或全拒。**逐行 cherry-pick 会让「确认」从判断退化为编辑,成本收益倒挂;需要部分采纳时,正确姿势是退回修改并说明。

### 5.5 交付卡与收件箱

**交付卡**:缩略预览 + 规模信息(页数/表数)+ 生成时间;点开全屏预览;正文引用角标点开右侧抽屉显示原文片段与出处;操作为下载 / 转发钉钉 / **按同一连接清单重跑**(复用清单开新任务,这是高频动作,必须一键)。

**收件箱**三栏:待确认(置顶,角标提醒)、进行中(当前步骤 + 已用时)、已完成(近 7 天,更早进归档搜索);排序恒为 待确认 > 进行中 > 完成时间倒序。收件箱是手机端首页,对话详情是二级页。

### 5.6 文件抽屉与错误卡

**文件抽屉**挂在任务详情右侧:按连接类型分组(SVN / 文档 / 表格 / kb / 上传),每条显示 mode 与状态(已连 / 失效 / 待授权);增删连接即修改清单,执行中的增删自下一步骤生效。

**错误卡**:一句人话 + 降级说明 + 一个主操作(重试 / 排队 / 重建);原始日志折叠于「技术详情」;同类错误在同一任务内不重复出卡。

## 六、实现映射:四能力 × SDK(建议,以黑池实际为准)

### 能力一 · 任务工作区:连接装配,而非文件容器

**关键修正(守密人 2026-07-13 裁定)**:黑池用户的资料活在外部系统里——SVN 工程目录、钉钉文档、各类表格。策划的工作对象本身就是 SVN 中的配置表,改完还要写回去。因此 workspace 不是「把文件搬进来的沙箱」,而是**一份连接清单**:声明这个任务连着哪些活系统、什么权限;资料留在原地,AI 通过连接器操作。

```jsonc
// workspace 连接清单(任务记录的一部分)
{
  "task_id": "t_20260713_0042",
  "profile": "planning",
  "connections": [
    { "type": "svn",     "ref": "svn://build/config/battle/", "mode": "rw",
      "checkout": "srv-ws://t_0042/battle/" },   // 服务端工作副本,写回=commit
    { "type": "dingdoc", "ref": "ding://doc/xxxx",  "mode": "rw" },
    { "type": "table",   "ref": "ding://sheet/yyyy", "mode": "read" },
    { "type": "kb",      "ref": "kb://design-bible", "mode": "read" }
  ],
  "outbox": "srv-ws://t_0042/outputs/",
  "memory_scope": "user"
}
```

**SVN 连接器**是四类连接器中最重的,也最接近 Code 的 cwd:黑池跑在内网,服务端对声明路径做浅 checkout(任务级工作副本),会话在副本内读写,写回统一走「变更集确认后 commit」(见能力二)。这既复用团队现有的 SVN 权限体系,也让「策划改配置表」这种介于文档与代码之间的工作,获得与代码同级的可追溯性。**钉钉文档/表格连接器**经内网 DWS(钉钉内部套件)调用官方 API 读写——链路已验证可达,读取结果进入会话上下文,写入进入变更集待确认。**职能装配预设**是静态映射表:策划默认连设定 kb 与所属工程 SVN 路径,运营默认连社区数据与运营文档目录,用户可在文件抽屉增删连接。

**生命周期**:创建(生成清单 + 建工作副本)→ 活跃 → 交付(变更全部落地或放弃,副本冻结为快照)→ 归档(清理工作副本,保留清单与产物,可按清单一键重建)。工作副本是易耗品,清单才是资产。

- SDK 已有:mounts / 双模装配、记忆 M1/M2、kb_* 工具
- 黑池要写:四类连接器(svn / 钉钉文档 / 表格 / 文件)、连接清单 schema 与职能预设表、任务级工作副本的建立/冻结/清理

### 能力二 · 内置验证:预览 · 引用 · 变更 diff · 计划确认

四件事共同回答一个问题:**用户凭什么信这个结果。**前三件面向「生成新东西」,第四件面向「修改已有系统」——后者正是策划/运营工作介于文档与代码之间的部分。

**产物预览**:渲染管线监听出件区,docx 经 LibreOffice headless 转 html(保留样式,首页出缩略图),xlsx 经 SheetJS 转分页表格视图,md/html 直出。渲染失败不阻塞交付,卡片退化为「下载 + 文件信息」。

**来源引用**:OKF 厚锚在检索结果里已携带出处。产品层只做**透传**:会话在引用处输出锚 ID,产品层替换为前端引用节点,点击展开侧栏显示原文。

**变更 diff 卡**:写回类变更(SVN 提交、钉钉文档更新)一律先冻结为**变更集**并渲染 diff 卡,确认后落地。确认动作即落地动作:点确认 = commit / API 写入,message 自动带任务号,可追溯可回滚。呈现规则见 5.4。

**计划确认**:约定 plan 消息——会话动手前输出结构化计划并调用 TaskStop 自行暂停;确认走续接配方恢复,修改则把意见作为新输入续接出修订版。

```jsonc
// plan 消息(会话 → 产品层)
{ "type": "plan", "steps": [
    { "id": 1, "title": "汇总四语社区反馈", "sources": ["data://community/daily"] },
    { "id": 2, "title": "按好评/差评/建议分类" },
    { "id": 3, "title": "生成复盘 docx", "output": "ws://outputs/复盘报告.docx" }
  ], "est_minutes": 35 }
```

- SDK 已有:OKF 厚锚(出处原生携带)、TaskStop + 会话续接配方
- 黑池要写:渲染管线与缩略图、变更集模型 + svn/表格/文档三种 diff 渲染、锚 ID 透传、plan 协议与四张卡片组件

### 能力三 · 长任务异步化:连接与会话解耦

地基是引擎 0.43 的**断线韧性四层**(请求期重试 → 有界回合重放 → 断流打捞 → 会话续接配方):会话状态活在服务端,客户端连接只是观察窗口。产品层补三件:

**任务状态机**:

```
queued → planning → awaiting_confirm → running → delivered
                          ↘ revising ↗        ↘ failed(带人话原因,可一键重试)
```

状态机由产品层持有,是收件箱、通知、重试的唯一事实源;会话不感知状态机,只通过 plan / TaskOutput / 完成信号驱动迁移。

**事件到进度的翻译**:以 plan 步骤为骨架,将 SDK 执行事件归拢为「当前在第几步」。宁可粗,不可假:翻译不出的事件归入当前步骤,不编造细节。

**通知与收件箱**:状态迁移到 awaiting_confirm / delivered / failed 时推钉钉(webhook 机器人,不自建推送),通知带任务链接。收件箱规格见 5.5。

- SDK 已有(需 pin ≥ 0.51):断线韧性四层、会话续接配方/回合重放、记忆 M1/M2
- 黑池要写:任务队列与状态机、事件→步骤翻译表、钉钉 webhook + 收件箱页

### 能力四 · 错误面与权限:消灭问题,而非翻译问题

**错误:分类 → 文案 → 降级动作。**SDK 重试兜底已消化大部分瞬时故障;剩余错误按类映射为一句人话加一个降级动作,原始错误只进日志,永不进对话流。

| 错误类 | 用户看到的话 | 降级动作 |
|---|---|---|
| 数据源不可达 | 「资料库暂时连不上,先用已有资料继续,结果可能不全,完成后标注缺口」 | 剔除该连接继续;交付物标注缺口 |
| 产物渲染失败 | (不打扰)交付卡显示「预览暂不可用」 | 退化为下载;渲染异步重试 |
| 会话不可恢复 | 「这个任务中断了,已保留计划和资料,点一下可从头重跑」 | 同清单重建会话,免重新交代 |
| 预算/额度触顶 | 「今天的用量满了,任务排到明天自动继续」 | 回队列,状态 queued,到点自动跑 |

**权限:安全默认,不弹窗。**Cowork 会话的权限就是连接清单:清单内按声明的 mode 读写,**清单外一切不可达**;所有写回动作又统一被变更 diff 卡拦一道——读的边界靠清单,写的落地靠确认。两道结构性闸门之下,运行期不需要任何权限弹窗。SVN 侧还天然叠加团队既有的账号权限,AI 拿到的永远是任务账号的子集。

- SDK 已有:请求期重试/断流打捞、错误事件的类型化输出
- 黑池要写:错误类→文案→降级动作映射表、连接清单边界的权限策略实现

## 七、实施路线:四步与验收标准

| 步 | 内容 | 验收标准 | 说明 |
|---|---|---|---|
| 一 | 引擎 pin 0.3x → 0.51,回归现有功能 | 现有全部场景无回退;断线后会话可续接 | 零新功能,纯地基;韧性与记忆是能力一、三的前提 |
| 二 | 产物预览 + 来源引用 | docx/xlsx 交付卡内可预览;kb 结论可点出处 | 见效最快的一步,直接改变信任;连接清单先用最简版随此步落地 |
| 三 | 计划确认 | 长任务默认先出计划;修改计划走通一轮 | 依赖步一的续接能力 |

顺序的逻辑:每一步都独立可交付、独立有收益,且是下一步的依赖。不并行,不跳步——**想清再动手,最简优先,外科手术式改动。**

## 八、边界与风险

### 明确不做

**不做 IDE 集成、不伺候 Code 场景**:技术同事的 IDE 内需求由成熟品满足,黑池对他们的价值是 IDE 不擅长的跨库问询与 bug 归因,顺着现有用法生长即可。**不做通用全能界面**:那是头部厂商几百人尚未收敛的方向,独立开发资源下赌它违反最简优先。**不做多套按职能分化的 UI**:职能差异全部下沉到装配预设与 skill 库,界面只有一种。

### 主要风险

**升 pin 的回归风险**:0.3x → 0.51 跨度大,双传输线与记忆系统引入行为变化。对策:步一独立成段,回归验收通过前不叠加任何新功能。**计划确认的体验风险**:小任务也弹计划会烦人。对策:按预估时长设阈值,短任务直跑;阈值做成配置。**渲染管线的格式坑**:docx 转 html 对复杂样式有损。对策:预览定位为「看对不对」,不承诺像素级还原,原件永远是交付物本体。**翻译表的维护成本**:事件归步骤靠约定,SDK 升级可能漂移。对策:翻译失败自动归入当前步骤的兜底已在设计内,漂移只降体验不出错。

> 收束回一句话:界面维持对话框 + 文件区不变,全部力气花在四层能力上——锚点、验证、异步、错误面。这四件事做完,黑池对策划、运营、文案的可用性将与今天有代差。
