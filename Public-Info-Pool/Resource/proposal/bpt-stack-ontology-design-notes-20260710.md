## §1 终点结论：二物一泵，分层缓立

Agent 应用的本质定为**二物一泵**：

| 件 | 定义 | 备注 |
|----|------|------|
| **上下文**（物质） | 广义：**模型所见 + 所许**——知识 / 记忆 / 工具配给 / 权限政策的装配与积累 | 银芯知识层为其运行中参照原型 |
| **界面**（物质） | 人对运行中 agent 的一切接触面：看 / 批 / 收 / 定时 / 托管 | 含宿主服务件（投递 / 触发 / 远程 / 监督）|
| **引擎**（泵） | 二物之间的商品泵：agent 环 + 工具执行 | bpt-agent-sdk，drop-in 表面已发货（v0.6）|

**委派 / 治理 / 记忆均为二物之上的「模式」，不是第三种物质**（归约验算见 §6）。

**分层缓立**：行业仅引擎层已标准化（Messages API / MCP），应用层抽象未熟（知识侧
OKF v0.1 于 2026-06 方出、委派侧零标准、界面侧各产品各长各的），此时给层立法为时过早
——本会话四层草案七轮拷问改到 r4、层名全长成「A 与 B」即是病征（证据见 §5）。
**层应由施工长出**：待委派 MVP / 治理 schema 提取等真实代码成形、关节自现，再命名。

> 比喻：城还没长出来就先画四环路，路名才会改四遍。先让路被踩出来，再铺柏油、再立路牌。

## §2 探索历程：七轮边界拷问 + 一次海拔归约

起点：守密人确认官方 Cowork 与 Agent SDK 无接口关系（并列产品）后，提出「BPT 体验朝
Cowork 前进」，经委派层（Cowork SDK）议案、四层栈定名，进入逐边界拷问：

| 轮 | 守密人出题 | 裁决 | 产出（沉淀去向） |
|----|-----------|------|-----------------|
| 0 | 四层定名：L0 引擎 / L1 委派 / L2 界面 / L3 上下文治理 | 立草案 r1 | 机制 / 策略 / 呈现三刀雏形 |
| 1 | L2 / L3 应否合并 | **不合并** | 变化速率错配 / 多客户端 / 防火墙变脏三证；「部署可合、概念不可合」 |
| 2 | 反过来：治理沉于界面之下 | **采纳**（r2） | **缺席测试**、注入缝单向下流（→ §3-2、§4） |
| 3 | L2 拆成工具与记忆两层？ | **分域不分层** | 跨域原子证据（技能 / 角色包 / 子代理）、两域一底座 |
| 4 | 「L2 每个团队都不一样」 | 直觉正确 → **机制 / 实例切分** | 形式定死内容开放（types.json 先例）、冷启动 → 种子包（→ §3-1） |
| 5 | Silver Core SDK 一包含三层、全部对外暴露 | **采纳有条件** | 一包一版、drop-in 保序、层向棘轮、治理是服务非锁（大路 / 检修口）（→ §3-7） |
| 6 | L1 / L2 应否合并 | **不合并但换序**（治理沉底，r3） | 缺席梯度单调、服务在下范式在上 |
| 7 | 改名：L3 服务与界面 / L2 委派与记忆 | **有条件收**（r4） | **情节 / 语义记忆分户**、宿主层完型、工作飞轮（→ §3-4/5、§6） |
| 归约 | 「本质都是上下文、界面」 | **采纳**，分层降格为脚手架 | 本档 §1 本体 + §3 台账 |

每一刀局部都有理、都让模型变硬——但七轮仍收敛不住，正是「territory 未凝固」的元证据。
**拆脚手架，留钢筋**：分层作废，拷问中活下来的不变量与刀具全部保留。

## §3 不变量台账（钢筋七条——先行生效，不依赖任何分层）

1. **机制 / 实例切分**：治理 / 知识类资产 = 一套机制 × 每团队一份实例。**形式定死、内容开放**
   （`Public-Info-Pool/types.json`「形式定死、清单可增」与 OKF「一概念一文件」同款纪律）；
   机制可发货，实例发不出去、只能各自长；冷启动靠**种子包**（官方 Cowork 角色插件包的本质）。
2. **政策经注入缝单向下流**：上层政策一律经引擎既有注入缝（`canUseTool` / hooks /
   `settingSources` / 提示装配）进入执行；任何模块不得向上 import。守卫：模块向 import
   棘轮测试（本仓红线测试手艺复用）。
3. **缺席测试作刀不作法**：需要给模块排序时，以「运行时可否缺席」定内外（可缺席者在外）；
   它是可复用判准，不再预先固化为层编号。
4. **策展晋升、禁自动蒸馏**：情节 → 语义只走显式策展膜（promote / `record_decision` /
   `record_lesson`），绝不自动灌注。银芯 2026-06-14 自动记忆环退役（06-20 连码带数据删净）
   为一手教训；团队规模下诱惑更大、坑更深。
5. **情节 / 语义记忆分户**：情节记忆（任务日志 / 运行史 / 交付清单）随工作走；语义记忆
   （决策 / 教训 / 知识库 / 偏好）随治理走。「记忆」一词入典必标户口，防两个「记忆」打架。
   > 比喻：日记本 vs 百科全书——干完活先记日记，值得留的经验由管理员誊进百科；
   > 来聊天的人不写日记，但人人都要查百科。
6. **防火墙切分**：机制（代码 / schema / 工具）可发货；实例（团队真实内容）归黑池、
   永不回流——与 §1.1-HC 黑池防火墙同向，银芯出空书架与图书管理员，黑池装自己的书。
7. **发货物一包一版**：银芯侧 SDK 发货物合一包一版（根导出 = 官方形状 drop-in 表面、
   COMPAT.md 管辖不变；新能力走子路径导出，**随建随长、不预刻层结构**）；版本纪律
   （改 src 必 bump + CHANGELOG + `check-version-bump.mjs` 守卫）沿用。根因：三拨同名
   0.6.0 tarball 对账之痛，一个版本号根治版本矩阵。

## §4 刀具箱（可复用判准，供未来切分时取用）

- **缺席测试**：运行时可否缺席定内外；
- **依赖方向**：服务在下、范式在上；看谁消费谁，不看谁「听起来更高级」；
- **变化速率错配**：因不同原因、按不同节奏变化的东西不同置（最耐久资产不与最易变件同居）；
- **原子完整性**：跨域原子（技能 = 教你用家伙的书；角色包 = 书 + 家伙一箱；子代理 =
  提示词 + 工具集）不可劈——切分方案若劈开原子即错；
- **机制 / 策略 / 呈现三刀**：机制与策略分置、呈现另置；
- **命名病征**：模块名长成「A 与 B」= 刀口不利信号；
- **YAGNI**：为已出现的第二个消费者建通用性，不为想象中的建（真多租户等第二个团队出现再说）。

## §5 证据：抖动地图 = 标准化地图

四层草案七轮里「谁在抖」，恰好对齐行业标准化程度——给「应用层抽象未熟」以可指认的事实：

| 部位 | 行业标准现状 | 本探索中的抖动 |
|------|-------------|---------------|
| 引擎 | **已标准化**（Messages API / MCP / agent 环共识） | **零次**——从头到尾无人动它 |
| 知识 / 上下文 | 刚起步（OKF v0.1，2026-06 发布） | 抖过（记忆拆户、域划分） |
| 委派 | **零标准**（官方 Cowork 2026-01 方上线、插件体系进行时） | **抖得最凶**——每轮换位或换名 |
| 界面 | 零标准（各产品各长各的） | 抖过（并入宿主服务、缺席半格修正） |

官方自身亦在动：Agent SDK 周更（0.3.x）、Cowork 插件市场演进中、桌面三标签为产品分组
而非架构分层。给没凝固的东西立法，法条必然一轮一改。

## §6 归约验算与工作飞轮

**委派归约**：一个委派任务 = **一段有生命周期的长上下文 + 若干人的检查点**——
计划审批 = 界面检查点；任务日志 = 上下文持久化；通知 = 界面事件；定时 = 界面侧触发。
归约无余项，故委派是**模式**非物质。

**治理归约**：上下文的**装配权与积累权**（什么进场、什么留存、谁有权改）。同为模式。

> 比喻：一家餐馆的本质是「菜」和「堂」；后厨机器是买来的标准件。「宴席预订」（委派）
> 不是第三种东西——还是菜和堂，只是换了下单方式。

**工作飞轮**（全栈闭环，产品本质所在）：

```text
干活（委派执行）→ 记情节（任务日志 / 交付清单）
              → 策展晋升（显式膜，禁自动蒸馏）→ 语义记忆 / 知识库
              → 装配挂载 → 下一次上下文
              → 下一个任务干得更聪明
```

「团队越用越聪明」即此环转动；§3-4 把银芯教训焊死在环上。

## §8 四层模型终版存档（r4——归档非约束，供关节重现时回查）

```text
L3  服务与界面        宿主层：多端 UI + 宿主服务件（投递/触发/远程/监督/账号）；界面半边可缺席
L2  委派与记忆        跑任务、记履历（情节记忆居此）；纯聊天部署整层可缺席
L1  上下文与工具治理   知识域/能力域两柱 + 规则引擎与共享底座；语义记忆居此；永不缺席
L0  引擎              bpt-agent-sdk；永在
```

版本演化：r1（L1 委派 / L2 界面 / L3 上下文治理）→ r2（治理与界面互换：L2 上下文与工具治理 /
L3 运行界面）→ r3（治理沉底：L1 治理 / L2 委派）→ r4（情节记忆入 L2、宿主服务入 L3）。
配套曾定：分域不分层（知识域 / 能力域）、机制 / 实例切分、Silver Core SDK 一包三层导出
（`.` / `/governance` / `/delegate`）。**上述编号与命名全部随「分层缓立」冻结**；不变量与
刀具（§3 / §4）继续有效。
