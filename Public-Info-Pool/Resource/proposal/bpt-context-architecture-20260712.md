> 性质：需求/架构建议档。黑池侧落地细节（存储选型、具体表结构）由黑池自定，本档只定分层、纪律与顺序。

## ⓪ 这是怎样一件工作、有什么意义

BPT 的核心命题是为全团队提供统一上下文（团队级），并与个人级视角区分开。团队信息源极多且异质：钉钉群聊天记录、Teambition 任务单进展、知识库全部输入输出、日报/周会汇总、SVN 全部提交（游戏工程）、BPT 自身使用记录、银芯社区归档、会议转录。本档回答一个问题：**这么多源，如何提炼成对团队有效的上下文。**

核心裁定（源自银芯自身的失败教训）：**不在入口处提炼，在查询时组装。** 银芯的「会话蒸馏 + 语义召回」自动环死于入口提炼——摘要一旦生成即与原文脱钩、开始腐烂，最终整套退役；活下来的是「全量归档 + 人工策展的薄状态层」。BPT 源更多，此教训只会更成立。

## §1 三层模型

### L1 全量档案层（不提炼，只归一）

- 全部信息源以 Record（append-only）落档 —— **黑池现状已满足此半边**。
- L1 唯一增量工作：**打实体锚**。每条 Record 标注其涉及的实体（人员 / 任务 ID / 系统模块 / 版本号 / 游戏实体如角色、命轮）。
- 跨源关联的钥匙不是摘要，是实体：「策划 A + 任务 #1234 + 某系统」可把 Teambition 单、SVN 提交、群内争论、周会结论串成一条线。
- 锚是**后挂索引**，不改 Record 本体，append-only 性质不破。
- 出身牌纪律（照搬银芯 OKF 厚锚三墙）：每个 AI 生成的锚带原文引用、可撤回、未确认压权重。

### L2 策展状态层（极小，人机共管）

- 全仓唯一做「提炼」的地方，但提炼对象不是信息流，而是**当前有效状态**：
  - 一本**裁定簿**（黑池版 decisions.md）：谁裁定了什么、为何、覆盖了什么。
  - 各线**状态档**（黑池版 project-status.md）：三条产线在做什么、卡在哪。
- 纪律照搬银芯：单一权威、指针不复刻、每条带出处（指回 L1 原始 Record）。
- 团队级默认上下文 = 只装这一层，几千字封顶。
- **裁定簿不按线分本**：全局一本，每条打线别标签（内容 / 工程 / 运营 / 全局），按标签过滤即得单线视图。拆分时机 = 单本吵到看不动时再拆，现在拆是提前付治理成本。

### L3 查询时组装（提炼发生在这里）

- 「知识是从特定角度观察数据的副产品」——角度由查询给出，不预设。
- 团队日常问题收敛为四类，检索按类路由：

  | 问题类 | 主源 | 取法 |
  |---|---|---|
  | 状态（在做什么） | Teambition / SVN / BPT 使用记录 | 确定性计算 |
  | 决策（为什么这么定） | 裁定簿 → 转录 / 钉钉原文 | L2 直答 + L1 锚回溯 |
  | 方法（怎么做） | 知识库 | 检索 |
  | 事件（出了什么事） | 日报 / 社区归档 | 检索 + 时效衰减 |

- 组装结果必须带锚（可跳回原文），转述不落库。

## §2 信息源分诊：三条路进状态层，只有一条设门

| 路 | 源 | 理由 | 处置 |
|---|---|---|---|
| **计算路（无门）** | Teambition、SVN、BPT 使用记录、知识库 I/O | 状态是显式字段，提炼 = 确定性聚合，非 AI 转述，不会失真 | 机器直算，直接进 L2 状态档 |
| **引用路（无门）** | 日报、周会汇总 | 本身已是人提炼过的产物，再提炼 = 摘要的摘要，只失真 | 作为 L2 输入直接引用 |
| **候选路（设门）** | **会议转录、钉钉聊天** | 唯二「关键信息以自然语言隐式存在」的源，决策/承诺/分歧藏在对话流里，只能 AI 转述取出，而转述有损 | AI 产出「候选决策 / 候选待办 / 候选风险 + 原文指针」→ 待确认队列 → 人确认后升 L2；不确认的仅作检索线索 |

- 候选路优先级：**转录第一，钉钉第二**。会议是决策与分歧的现场，转录是唯一逐字记下「为什么」的源，信号纯度高于群聊。
- 确认仪式建议挂周会：五分钟过一遍本周候选，点头即升。
- 「候选 + 确认」这道门不是提炼的成本，是**转述失真的保险**——只有隐式源需要买这份保险。

## §3 团队级 vs 个人级

同一语料的两个视角，不是两套语料：

- **团队级** = L2 策展状态层 + L3 全域检索。
- **个人级** = 在此之上叠「我的工作集」过滤器（锚到「我」这个实体的全部 Record 切面：我的任务、我的提交、我参与的群、我的会话历史）+ 权限裁剪。
- 实体锚层建成后，个人视角是**免费的**，无需单独建设。

## §4 落地顺序（依赖序）

1. **实体注册表**（人员 / 任务 / 模块 / 版本 / 游戏实体）——锚层、候选管线、个人视角三者的共同前置。
2. **Record schema 强制留锚位**（哪怕先空着）。银芯教训：4R 归位是后补的，付了一次 1382 文件大迁移的代价；黑池源仍在增，早定省大钱。
3. **锚层回填**：结构化源机器映射（零成本），转录 / 钉钉 AI 扫描（带出身牌）。
4. **候选提炼管线**（先转录后钉钉）+ 周会确认仪式。
5. **L2 状态层建档**：裁定簿冷启动 = 管线首跑回填最近 1–2 个月会议转录，扫出候选裁定，守密人过一遍确认——管线验收与簿子播种一石二鸟。
6. **L3 查询路由**接四类问题。
