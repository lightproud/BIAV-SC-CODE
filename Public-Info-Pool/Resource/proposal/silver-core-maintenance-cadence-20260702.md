## 二、07-19 后的维护态节拍（提案）

Phase 2 结束后，银芯从「建设期项目」降档为「值班中的系统」。节拍分三档：

### 持续档（全自动，无人值守）
- T1 新闻流：`update-news.yml` 每时 :00（现状不变）
- T2 归档：discord 每日 18:00 UTC + 各源错峰（现状不变）
- T3 回填：`repair_gaps` 随管线（现状不变）
- 静态索引重建：`build-analysis-index.yml` 每周一 04:17 UTC（现状不变）

### 周检档（会话内人工，建议每周一次，约 15 分钟）
- 读 `output/source-health.json`：degraded / never 源有无新增；validation 丢弃计数有无异常
  （丢弃指标 2026-07-02 起入 health，见改进计划 P0-3）
- 扫 `gh` 侧自动开的 health alert issue（若有）
- `git log --oneline -20` 确认机器提交节奏无断档

### 月检档（会话内人工，建议每月一次）
- 数据增长对账（见下节触发线）
- `pytest tests/` 全量跑一次留档
- 全仓 md 漂移抽查（CLAUDE.md 三卫 + 随机抽 2 份 CONTEXT 与实况对照）

## 三、数据增长触发线（实测定标）

2026-07-02 实测（`Public-Info-Pool/Record/Community/` 全量档案层）：

- 当前体量：**2.01 G**（工作树档案部分）；`.git` 517 M
- 近 3 满月增速：**约 205 MB/月**（2026-04 178.0 / 05 252.0 / 06 185.5 MB）
- 线性外推：档案层 12 个月后约 4.5 G，`.git` 同步增长

**触发线（提案，命中任一即重开瘦身/分层议题，不等到疼了才动手）**：

| 触发条件 | 预计命中时间（线性外推） | 届时候选动作 |
|----------|------------------------|------------|
| 档案层工作树 ≥ 4 G | 约 2027-04 | 重议冷热分层（如 >18 月旧档移 Release，保留 git 指针） |
| 单月增速 ≥ 400 MB 连续 2 月 | 事件驱动（如新联动爆量） | 提前触发上一行议题 |
| 全新 clone 用时 ≥ 5 分钟（GitHub Actions 实测） | 与 .git 增长相关 | 守密人执行 §9 历史压缩（既有待办） |

原则不变：text → git / 二进制 → Releases（2026-06-21 总纲）；触发线只决定「何时重议」，
不预设结论。
