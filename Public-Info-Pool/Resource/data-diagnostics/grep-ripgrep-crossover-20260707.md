# Grep 性能拐点研究：bpt-agent-sdk 纯 JS grep vs ripgrep

- **日期**：2026-07-07
- **提出方**：守密人（「Grep 接 ripgrep 与否，我需要数据再决定」）
- **对象**：`projects/bpt-agent-sdk/src/tools/grep.ts`（HONEST-SUBSET：纯 JS 正则，无 ripgrep 二进制）
- **测量环境**：本会话云容器，Node（单线程事件循环）+ ripgrep 14.1.0；合成语料，暖缓存后取 5 次中位数

---

## 0. 一句话结论

**拐点不由「仓库大小」单独决定，而由「这次查询是否必须全量扫描整个语料」决定。** 命中 250 条早停的查询几乎与仓库大小无关（恒 ~140ms）；必须全扫的查询（稀有词 / count / `head_limit:0` / 命中不足 250）成本**线性于总字节，且 97% 花在串行 `fs.readFile` 上，不是正则**。后者在 4K–12K 文件仓上比 ripgrep 慢 **87–127 倍**。

## 2. 机制拆解（插桩坐实，非推测）

### 2.1 主导成本是串行 I/O，不是正则

同一 4000 文件语料，把一次全扫拆成三段计时：

| 阶段 | 耗时 | 占比 |
|---|---|---|
| fast-glob 遍历 | ~15ms | 1% |
| **`fs.stat` + `fs.readFile`（串行，逐文件）** | **~1490ms** | **97%** |
| 逐行 `RegExp.test` | ~40ms | 3% |

> 结论：纯 JS grep 慢在**一次一个文件的串行系统调用**（小文件密集时有效吞吐仅 ~16–20 MB/s），**不是** V8 正则慢。微基准里 V8 对 200 万行跑字面量正则仅 85–140ms——正则从来不是瓶颈。

### 2.2 P1 为何「作弊般」地平：250 早停

`grep.ts` 有 `DEFAULT_HEAD_LIMIT = 250` → `collectCap = 250` → 输出满 250 条即 `break`：

- **P1 常见词**：前 ~250 个文件就凑满 250 条命中 → 扫 250 个文件就停 → 恒 ~140ms，**仓库再大也不扫完**（曲线平是早停假象，不是真快）。
- **P2 稀有/零命中**：永远凑不满 250 条 → **被迫扫完每一个文件** → 线性于总字节，串行 I/O 全额付出。

所以真实世界的成本 = 「凑满 250 条命中前，要串行读多少字节」。

### 2.3 ripgrep 为何恒快

并行多线程遍历+搜索 + Rust 有限自动机（无回溯）+ SIMD 字面量预筛 + mmap。85MB/41ms ≈ 2 GB/s，比纯 JS 串行快约两个数量级——这是**多核 + 架构**差距，纯 JS 单核追不平。

## 3. 拐点定位（全扫描regime）

「felt-lag」（人能感到卡）以 **>1s** 为线：

| 全扫描文本量 | SDK 纯 JS | 体感 |
|---|---|---|
| < ~7MB / ~1000 文件 | < 400ms | 流畅 |
| ~18MB / ~2500 文件 | ~1s | **拐点：开始卡** |
| 28MB / 4000 文件 | ~1.6s | 明显卡 |
| 85MB / 12000 文件 | ~5.2s | 难以接受 |

ripgrep 全程 19–147ms，无拐点。

**命中密集regime（250 早停）无拐点**：任意仓库大小恒 ~140ms、仅 1–2x rg，加 ripgrep 几乎零收益。

---

## 4. 第三选项：不引 ripgrep，只并行化纯 JS 读

既然 97% 花在串行 `fs.readFile`，把读并行化（并发池）实测（4000 文件全扫）：

| 方案 | 最坏档耗时 | 相对串行 |
|---|---|---|
| 串行（现状） | 1293ms | 1.0x |
| 并行(8) | 561ms | 2.3x |
| 并行(16) | 557ms | 2.3x |
| 并行(32) | 553ms | 2.3x |

并行读约 **2.3x**（I/O 等待重叠），~20 行代码、零依赖、不动可移植卖点——但仍单核解码+正则，**到不了 ripgrep 量级**（10–30x rg）。

---

## 5. 一个绕不开的正确性副作用

250 早停让 `files_with_matches` / `count` 结果**静默截断且依赖文件排序**（只给排序后前 250）。要「全仓精确计数 / 找全部匹配文件」必须传 `head_limit:0`——而这**正好触发最慢的全扫描路径**。即：**要完整就得慢，要快就可能不完整**，这对张力是内建的。审计 / 全量统计类用法尤其踩这条。

## 6. 选型建议（三选一，按黑池实际用法定）

| 选项 | 最坏档 | 代价 | 何时选 |
|---|---|---|---|
| **A 维持纯 JS 现状** | 全扫 1.6–5.2s | 零 | grep 几乎都是常见词交互查（吃 250 早停）、仓库中小 |
| **B 并行化纯 JS 读** | 全扫 ~0.55–2s | ~20 行、零依赖、保可移植 | 想去掉最坏悬崖又坚决不起子进程 |
| **C 机会性 ripgrep + 纯 JS 兜底（推荐）** | 全扫 ~20–150ms | 检测 `rg` 在 PATH 则 shell 之、否则回落纯 JS | 想要 rg 速度**又不牺牲**「无硬二进制依赖」卖点 |

**推荐 C**：它不是「用 ripgrep 替换纯 JS」，而是「有 rg 就用、没 rg 照跑纯 JS」——纯 JS 仍是保底地板，可移植卖点分毫不失；而绝大多数开发机 / CI 都有 rg，机会性提速能吃满。次选 B（纯零子进程洁癖时）。**除非黑池的 grep 用法证实以 250-早停命中为主（那 A 就够，加什么都是白加）**。

> 小学生比喻：慢不是因为「算得慢」（正则只占 3%），是因为**一趟只搬一个箱子的搬运工**（串行读文件）。常见词查询像「找到 250 个就收工」，搬几趟就够、看着很快；稀有词查询是「必须把整仓箱子全搬一遍才能确认没有」，仓库越大搬得越久。ripgrep 是「叫了一支并行搬运队 + 会跳着扫的透视眼」。选项 B 是「给现在这个搬运工多派几只手同时搬」（快 2.3 倍，但还是一个人指挥）；选项 C 是「谁家有搬运队就借用、没有就自己搬」——最省心，还不用把搬运队焊死进合同。
