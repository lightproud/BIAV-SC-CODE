## 1. v1-vs-v4 提示词 A/B（run 28723716555，硬任务 10/11）— ⚠ 本节数据无效，见 §4

> **本节整节结论作废**：下表两臂**都跑了极简默认提示词**（测量 bug——A/B harness 选了 variant 却没设 `systemPrompt:{type:'preset',preset:'claude_code'}`，而 variant 只在 preset 路径生效，被静默忽略）。故「v4 与 v1 无差异」是「v4 根本没进去」的假象，不是真无收益。修复 + 真对照见 §4。缓存根因分析（§1 后半 + §1.x）仍有效，因其用受控探针独立验证，不依赖 A/B 臂。

- **判定：无可测收益**。v4（从公开还原忠实再现的官方主循环提示词）与 v1（精简自研）在这两道能真失败的硬任务上**都过、轮数相同、v4 略贵略慢**。
- **决策：v4 不提为默认**（同 v2 结论：看起来更周全 ≠ 数字更好）。硬任务 10/11 对 v1 也不够难、区分度不足——需更狠任务定位 v4 的价值边界。
- 缓存：task 11 = 0/0（前缀够不着 Haiku 2048 门槛、没写）；task 10 = 4140/0（**写了、但没读回**）。

**缓存读-miss 调查结论（2026-07-05，三条硬证据，纠正早前「前缀漂移」的误判）**：
1. **前缀不漂**：新增确定性回归测试 `tests/query.test.ts`「stable prefix does not drift across turns」证明——2 轮工具往返里，被缓存的 system 稳定块 + tools **逐字节一致**、断点落在稳定块上。API **能**在 turn 2 读中。
2. **计量无误**：`loop.ts` 的 `result.usage` 经 `addUsage` **跨轮累加**，cache_read 求和正确，read=0 是真值非漏算。
3. **故 read=0 非代码 bug**：真因是**短任务的缓存写入传播延迟 + 任务太短**——turn 1 写的缓存 ~1.5s 后 turn 2 尚未传播可读、且 3 轮任务没有足够后续轮去累积读取（长任务 8/9 有时间传播 + 多轮 → 45%）。
**结论（2026-07-05 受控探针实证，取代上面所有推测）——是「边缘尺寸死区」**：
用 `tests/integration/cache-probe.mjs` 同任务背靠背 3 次、**常规前缀 (~3.5k tok) vs 大前缀 (~8k tok)** 并排实测（`--big`）：
- **常规 ~3.5k**：3 次全跑、几乎每轮 **write≈0 / read=0**——缓存**根本没被真正归档**。
- **大前缀 ~8k**：run1 turn0 **write=8668**，后续轮 **read=8668/8807**；**run2/run3 turn0 直接 read=8266（跨请求复用成立）**；每次读 **17k–25k**——**完美缓存**。

**定论**：Haiku 的**有效**缓存门槛**远高于名义 2048**；我们 ~3.5k 精简前缀落在「过名义门槛、却小到不被真正归档」的**死区**（写≈0、读=0），官方 5k–15k 前缀舒服落在**可靠缓存区**（99%）。
- **非代码 bug**：三重证明——(1) 确定性 wire 捕获证 cache_control 落位正确（tools+system+message，均 >2048）；(2) `loop.ts` result.usage 跨轮累加无误；(3) 大前缀同代码同路径**缓存完美**。是**尺寸落死区**，不是断点/计量/漂移。
- **解法（实证背书，非推测）**：稳定前缀做到 ~6k+ 即可靠缓存 + 跨轮 + **跨请求**复用——正是 **org 分层**（团队/个人提示词叠成大而共享前缀：一人写、全队每次读）。此前「组织级缓存红利」由理论升格为**实测**。
- **纠正早前判断**：先说「门槛不够（<2048）」、后说「写了读不中/传播」，实测下**都不准**——真因是「过名义门槛但落有效死区」。灌水不是解（会付冷启动膨胀价），**真实分层的大而共享前缀**才是。

**三轴结论**：

- **成本**：本 SDK **~8% 便宜**（$0.150 vs $0.162）。靠精简提示词的净省，尽管官方缓存命中远高。

**诚实保留（勿过读）**：
- 官方 #3/#11 的 0% **可能是无头 CI 里官方 spawn Claude Code CLI 的水土不服，未必质量真差**。**不据此宣称「我们更准」**——需多轮复现才可采信。正确性这轴带 CLI-headless 噪声。
- 速度 2.7× 与成本 8% 是结构性差异，噪声小、可信。

**⚠ 缓存命中的方法学陷阱（守密人 2026-07-05 观察 → 修正解读）**：官方 94–99% 命中**是背靠背 benchmark 的假象、高估了真实优势**，机制两条叠加（**非**跨客户共享——Anthropic 缓存按 org 隔离）：
1. **median of 3**：每任务跑 3 次，第 1 次写官方那**巨大且逐字相同**的前缀、第 2/3 次读 → 每任务中位命中率天然偏高（任务 1 也 94% 即此故）。
2. **跨任务复用**：9+2 任务共用**同一份**官方前缀，5 分钟 TTL 窗口内背靠背跑完 → 任务 1 写、其余全读。
本 SDK 0% 因**前缀够不着 Haiku 2048 门槛、根本没写缓存**（长任务涨大才 45%）。
**推论**：真实 BPT 单用户零星干活（任务间隔 >5min TTL）→ 缓存全冷 → 官方每次**重付大前缀写入费**；故冷启动现实里本 SDK 的成本优势**很可能比 8% 更大**。「官方缓存更强」一半是假象。
**待坐实**：读官方每任务 `cache_creation` vs `cache_read` 拆分（artifact 内）确认「repeat1/任务1 大量写、其余大量读」。**待补测法**：vs-official 加 `repeat=1` 冷启动模式（列入 G-cmp），消除同 org 暖前缀偏置、量真实冷成本差。

## 4. 测量 bug 更正 + v1-vs-v5 真对照 + 默认提升（run 28725983766，2026-07-05 晚）

### 4.1 测量 bug（先拿真相）

- **病灶**：`tests/integration/ab-benchmark.mjs` 与 `cache-probe.mjs` 选了 `--variant` 却没同时把 `systemPrompt` 设成 `{type:'preset',preset:'claude_code'}`。而 `harnessPromptVariant` **只在 claude_code preset 路径**上被 `buildSystemPromptParts` 读取——不设 preset 时走极简默认，variant 被**静默忽略**。
- **后果**：早前所有 v1-vs-v2/v3/v4 A/B **两臂都是极简默认提示词**，故测出「无差异」纯属假象（候选臂根本没换提示词）。§1 结论据此作废。
- **修复**：两个 harness 现在只要给了 `VARIANT` 就**一并**设 preset。修复 commit 与本对照同 PR（#421 分支）。
- 小学生比喻：以为在比「甜面包 vs 咸面包」，结果两个盘子里放的都是白馒头——尝不出差别不是因为面包一样，是因为压根没端上来。

- **判定翻案：v5 ~3× 便宜、同正确、略快**。反直觉但自洽——v5 提示词大 16×（~3774 vs ~238 tok）却更便宜，因为：v1 前缀太小落**有效缓存死区**（0% 命中、每轮全价重发），v5 前缀够大→**95% 命中**（首写后每轮读价约一成）。**缓存红利 > 前缀增量成本**。
- 小学生比喻：v1 像每次买单张公交票（便宜但张张全价），v5 像买月票（一次贵、之后每程近乎白坐）——多程下来月票远省。

### 4.3 v5 缓存探针（受控背靠背，run 同上）

v5 常规前缀（~3774 tok，非 --big）3 次背靠背：
- run1/turn0 write=6461 read=0（首写前缀+工具）→ turn1 read=**6461** → turn2 read=**6596**（同 run 跨轮命中）
- **run2/turn0 read=6059、run3/turn0 read=6059**（**跨 request 命中**，TTL 内，新 cwd 不影响稳定前缀复用）

即 v5 每轮、每 run 稳定命中 6000+ token 前缀，与官方 99% 同构。**证实 §1「边缘尺寸死区」定论**：把稳定前缀做大（v5 ~3.7k 已够、--big ~8k 更满）即从死区进入可靠缓存区。官方 99% 不是玄学，是「提示词够大」的必然。

### 4.4 决策

- **v5 提为 claude_code preset 默认**（守密人 2026-07-05「2 模拟行为」+「目标是跟官方提示词一致」裁定）。数据背书：更便宜、同正确、更忠实官方主循环。v1–v4 保留为显式变体（想要极简可 `harnessPromptVariant:'v1'`）。
- **v5 保真收尾**：对着归档官方还原逐段自审，补回两条漏收的官方主循环子句（`act-when-ready` + 安全策略 `censoring-assistance`）；两条产品专属子句（反馈 URL、CLI harness 头）确认应省不补。v5 现 ~3774 tok。
- **测量纪律加固**：任何 variant A/B 必确认 preset 已设（否则 variant 无效）；本 bug 列入踩坑，后续对照先自检「候选臂真的换了提示词吗」。
