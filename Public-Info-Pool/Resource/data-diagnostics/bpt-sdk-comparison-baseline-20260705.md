## 0. 一句话

## 1. v1-vs-v4 提示词 A/B（run 28723716555，硬任务 10/11）

- **判定：无可测收益**。v4（从公开还原忠实再现的官方主循环提示词）与 v1（精简自研）在这两道能真失败的硬任务上**都过、轮数相同、v4 略贵略慢**。
- **决策：v4 不提为默认**（同 v2 结论：看起来更周全 ≠ 数字更好）。硬任务 10/11 对 v1 也不够难、区分度不足——需更狠任务定位 v4 的价值边界。
- 缓存：task 11 = 0/0（前缀够不着 Haiku 2048 门槛、没写）；task 10 = 4140/0（**写了、但没读回**）。

**缓存读-miss 调查结论（2026-07-05，三条硬证据，纠正早前「前缀漂移」的误判）**：
1. **前缀不漂**：新增确定性回归测试 `tests/query.test.ts`「stable prefix does not drift across turns」证明——2 轮工具往返里，被缓存的 system 稳定块 + tools **逐字节一致**、断点落在稳定块上。API **能**在 turn 2 读中。
2. **计量无误**：`loop.ts` 的 `result.usage` 经 `addUsage` **跨轮累加**，cache_read 求和正确，read=0 是真值非漏算。
3. **故 read=0 非代码 bug**：真因是**短任务的缓存写入传播延迟 + 任务太短**——turn 1 写的缓存 ~1.5s 后 turn 2 尚未传播可读、且 3 轮任务没有足够后续轮去累积读取（长任务 8/9 有时间传播 + 多轮 → 45%）。
**结论**：**缓存代码无缺陷**，两个症状（task 11 门槛、task 10 短任务/传播）均**非可修 bug**；真正让缓存生效的杠杆是**真实部署的更大稳定前缀（org 分层）+ 更长会话**，非人为灌水。v4（更大提示词）实测缓存行为与 v1 相同（task 10 均 ~4140/0），证明提示词大小非杠杆、tools + 真实分层内容才是。

**三轴结论**：
- **速度**：本 SDK **~2.7× 快**（38.0s vs 104.2s 墙钟）。**结构性**——无子进程冷启动、直驱 API。可信。
- **成本**：本 SDK **~8% 便宜**（$0.150 vs $0.162）。靠精简提示词的净省，尽管官方缓存命中远高。
- **正确性**：本轮本 SDK **11/11 全过**，官方 **9/11**（#3 write-edit-cycle、#11 反硬编码 各 0%）。
- **缓存**：官方 94–99% 命中（大提示词缓存吃得满）vs 本 SDK 0%（短）/45-46%（长）。官方缓存显著更强，但本 SDK 净成本仍更低。

**诚实保留（勿过读）**：
- 官方 #3/#11 的 0% **可能是无头 CI 里官方 spawn Claude Code CLI 的水土不服，未必质量真差**。**不据此宣称「我们更准」**——需多轮复现才可采信。正确性这轴带 CLI-headless 噪声。
- 速度 2.7× 与成本 8% 是结构性差异，噪声小、可信。

**⚠ 缓存命中的方法学陷阱（守密人 2026-07-05 观察 → 修正解读）**：官方 94–99% 命中**是背靠背 benchmark 的假象、高估了真实优势**，机制两条叠加（**非**跨客户共享——Anthropic 缓存按 org 隔离）：
1. **median of 3**：每任务跑 3 次，第 1 次写官方那**巨大且逐字相同**的前缀、第 2/3 次读 → 每任务中位命中率天然偏高（任务 1 也 94% 即此故）。
2. **跨任务复用**：9+2 任务共用**同一份**官方前缀，5 分钟 TTL 窗口内背靠背跑完 → 任务 1 写、其余全读。
本 SDK 0% 因**前缀够不着 Haiku 2048 门槛、根本没写缓存**（长任务涨大才 45%）。
**推论**：真实 BPT 单用户零星干活（任务间隔 >5min TTL）→ 缓存全冷 → 官方每次**重付大前缀写入费**；故冷启动现实里本 SDK 的成本优势**很可能比 8% 更大**。「官方缓存更强」一半是假象。
**待坐实**：读官方每任务 `cache_creation` vs `cache_read` 拆分（artifact 内）确认「repeat1/任务1 大量写、其余大量读」。**待补测法**：vs-official 加 `repeat=1` 冷启动模式（列入 G-cmp），消除同 org 暖前缀偏置、量真实冷成本差。

## 3. 决策与后续

- **v4 不提为默认**（无可测收益）。保留为变体，待更狠任务或引擎机制到位后重测。
- **vs-official 常规化**：本档为首个数据点。再现工作每推进一批（G1–G4 引擎机制）即重跑 vs-official，看 2.7× / 8% / 正确性是否保持或改善——数据说话。
- **每项 G 目标附 before/after**：G1 压缩前置层测 input-token 削减、G3 双断点测缓存命中、G2 测 Haiku 摘要成本。
