# Discord 归档进一步节省 — 10 法择优研究

> 数据诊断 · 2026-06-22 · 守密人「继续找 10 种方法择优」派发
> 基线：今日紧凑 schema 已把 discord 工作树 3.4G→**2.0G**（决策见 `memory/decisions.md`）。本研究找在此之上的下一步。
> 方法：抽样 1,500 文件 / 833,504 条（全量 ×9.1 外推），逐法实测/估算。

## 一、精简后 2.0G 还剩什么水分（字节解剖）

| 成分 | 占比 | 本质 |

## 三、一个决定次序的硬事实

`du .git` = **2.1G**，今日精简只缩了**工作树**（2.0G）——**.git 没动**，因历史里新紧凑 blob + 旧胖 blob 并存。
推论：**#1–#8 任何工作树优化都只缩工作树、不缩 .git，直到 §9 历史压缩跑过**。故最优次序：

> 先把工作树压到目标形态（选 #2 或 #1）→ 再跑**一次** §9 历史压缩 + `git gc`。历史只改写一次，落盘即最小。

## 四、择优结论（守密人 2026-06-22 裁定：止步 41%，保留 timestamp）

驳回伤 grep 的（#3/#4/#5/#7）与违总纲的（#9/#10）后，grep-safe 且无损只剩 **#1 TSV** 与 **#2 丢 timestamp**（#1 已含 #2）。初评以「省幅 vs 自描述性」取舍，倾向 #2。但守密人一问点破盲区——**信息无损 ≠ 使用无损**：

> **本质分界线**：今日已做的 41% 精简删的是 `pinned`/`flags`/`embeds` 等 99%~100% 恒默认、**查询零价值**的纯水分（删了零便利损失，是免费午餐）；再往下的 #1/#2 省的都是**有查询价值或自描述价值**的字节（timestamp 不是水分，是肉眼可读、grep/jq 可直接过滤的有用信息）。**41% 那一刀正好停在「纯水分」与「有用信息」的分界上。**

丢 timestamp 的便利性代价（守密人指出）：

- **按日期查**：日期在文件名，更方便/持平；**管道消费**：加 `ts_from_id()` 一次性改即可；
- **临时精确时间查（grep/jq 捞秒级/毫秒排序）**：timestamp 没了得现解 snowflake，grep/jq 一行写不出——**而「text 进 git」整个决策的初衷正是保护「clone 即查、不用外挂」的便利**。

叠加硬事实：**`.git` 在 §9 历史压缩前不会变小**，丢 timestamp 现省的 355MB 连磁盘都落不了地。

**最终裁定（守密人）**：

- **到此为止，不丢 timestamp、不转 TSV，保留当前 grep-友好的紧凑 JSONL**。41% 是「省体积」与「保便利」的最优停点。
- 真要再瘦 .git，走 **§9 历史压缩 + `git gc --aggressive`**（守密人本地择期）——那才动 .git 的 2.1G，且**不碰任何字段、零便利代价**，是唯一「省体积不损便利」的下一步。
- #1 TSV / #2 丢 timestamp 归档为「已评估，因损便利/损自描述而不取」，除非未来体积压力远超便利诉求再翻案。

小学生比喻：第一刀剪掉的是信纸上「未加急/无附件」这种家家一样的空格子，剪了没人心疼；timestamp 是信上的日期时间——信封（文件名）写了大概日子，但要精确到几点几分还得靠它，为省这点纸把日期擦了、以后看信还要按编号自己算钟点，不划算。该停手了。真要腾地皮，是把仓库角落的旧底稿一次性烧掉（§9），那不动现役的信一个字。
