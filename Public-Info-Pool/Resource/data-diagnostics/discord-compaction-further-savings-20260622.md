# Discord 归档进一步节省 — 10 法择优研究

> 数据诊断 · 2026-06-22 · 守密人「继续找 10 种方法择优」派发
> 基线：今日紧凑 schema 已把 discord 工作树 3.4G→**2.0G**（决策见 `memory/decisions.md`）。本研究找在此之上的下一步。
> 方法：抽样 1,500 文件 / 833,504 条（全量 ×9.1 外推），逐法实测/估算。

## 一、精简后 2.0G 还剩什么水分（字节解剖）

| 成分 | 占比 | 本质 |
|---|---|---|
| 重复键名（`"id":` 等每行重复）| **29.4%** | 纯结构冗余（JSON 的代价）|
| content 正文 | 14.6% | 真信号，不可动 |
| timestamp | 12.1% | **可从 id 反推（snowflake，33,847 条 0 偏差验证）** |
| id | 7.5% | 可进制压缩（伤 grep）|
| channel_id | 7.5% | 同文件恒定（中位每文件 1 个，但有多频道文件最多 27）|
| author_id | 7.2% | 可内联成整数索引（伤 grep）|
| author_name | 4.3% | 可归一化到作者表（伤 grep）|

## 二、10 种方法实测/估算（全量外推，标注 grep 影响）

## 三、一个决定次序的硬事实

`du .git` = **2.1G**，今日精简只缩了**工作树**（2.0G）——**.git 没动**，因历史里新紧凑 blob + 旧胖 blob 并存。
推论：**#1–#8 任何工作树优化都只缩工作树、不缩 .git，直到 §9 历史压缩跑过**。故最优次序：

> 先把工作树压到目标形态（选 #2 或 #1）→ 再跑**一次** §9 历史压缩 + `git gc`。历史只改写一次，落盘即最小。

## 四、择优结论

驳回伤 grep 的（#3/#4/#5/#7）与违总纲的（#9/#10）后，grep-safe 且无损只剩 **#1 TSV** 与 **#2 丢 timestamp**（#1 已含 #2）。二者取舍 = 「省幅」对「自描述性」：

- **稳健最优（推荐）：#2 丢 timestamp**。无损、保留自描述 JSONL、低成本、-355MB。snowflake 反推 0 偏差，日期已在文件名、精确序在 id，零信息损失。
- **激进最大：#1 混合 TSV**。再多省 ~540MB（共 -896MB → 1.1G），但放弃「JSONL 自描述」这一 BPT/OKF 核心气质，需写格式契约文档 + 改写全部消费方。仅当守密人愿用自描述换体积时取。
- **无论选哪个**：都应在 **§9 历史压缩前**完成，并配 `git gc --aggressive`，让 .git 真正落到工作树大小。

小学生比喻：信里的「邮戳日期」其实已经盖在信封（文件名）和编号（id）上了，信纸内部再写一遍纯属重复——擦掉它（#2）最干净。要再狠就把所有信改成「不写栏目名、只按固定格子填」的表格（#1 TSV），省一大截但以后读信得另带一张「第几格是什么」的说明书。而真正腾仓库地皮，得先精简完再一次性烧掉旧底稿（§9）。
